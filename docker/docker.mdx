## Q1: Docker 的历史？为什么要有它？

```text
┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  物理机时代  │ → │  虚拟机时代  │ → │  容器化时代  │ → │  云原生时代  │
│ (2000 年前)  │   │ (2001-2012)  │   │ (2013-至今)  │   │ (2015-至今)  │
└──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
```

<Steps>

  <Step title="2013 年: Docker 发布">

    2013 年 3 月，Solomon Hykes 在 PyCon 大会上首次展示了 Docker。Docker 的核心理念是：**Build once, run anywhere**（一次构建，到处运行）。它利用 Linux 内核的 Namespace 和 Cgroups 技术，实现了轻量级的操作系统级虚拟化。

  </Step>
  <Step title="虚拟机 vs 容器: 两种隔离方案的对比">

    在 Docker 之前，业界主要靠虚拟机（VM）实现环境隔离。虚拟机通过 Hypervisor 模拟完整的硬件环境，每个 VM 都需要运行一个完整的操作系统（Guest OS），启动时间在分钟级别，占用资源巨大。

    容器则不同——它共享宿主机的内核，只隔离进程、文件系统和网络。启动时间在秒级甚至毫秒级，资源开销极小。

  </Step>
  <Step title="Docker 的核心价值">

    Docker 解决了三个核心痛点：**环境一致性**（开发、测试、生产环境完全相同）、**快速交付**（秒级启动、秒级销毁）、**资源高效**（比虚拟机节省 50%-90% 的资源）。这三点让 Docker 迅速成为 DevOps 的基础设施。

  </Step>

</Steps>

**Docker 解决的核心问题：**

```text
问题："在我电脑上能跑啊！"

开发环境：Node 16 + Ubuntu 20.04 + nginx 1.18
测试环境：Node 18 + CentOS 7  + nginx 1.20
生产环境：Node 14 + Debian 11 + nginx 1.22

→ 每次部署都是一场赌博

解决：把应用和它的运行环境打包成一个镜像

开发环境 = Docker 镜像（Node 16 + Ubuntu 20.04 + nginx 1.18）
测试环境 = 同一个 Docker 镜像
生产环境 = 同一个 Docker 镜像

→ 100% 一致性，零环境差异
```

<Note title="💡 一句话总结">

Docker 用容器技术解决了"环境不一致"的问题，让应用可以一次构建、到处运行。

</Note>

<CardGroup cols={2}>

<Card title="🚫 没有 Docker">

每个项目都要手动配环境，换台电脑就要重来一遍。"在我电脑上能跑"是开发和运维之间永恒的战争。

</Card>
<Card title="✅ 引入 Docker">

一个 `Dockerfile` 定义所有依赖，`docker build` 一键打包，`docker run` 秒级启动。新人入职第一天就能跑起整个项目，不用花一周配环境。

</Card>
</CardGroup>

### 虚拟机 vs 容器

| 对比维度 | 虚拟机 (VM) | 容器 (Container) |
|---------|------------|-----------------|
| **隔离级别** | 硬件级隔离（完整 OS） | 进程级隔离（共享内核） |
| **启动时间** | 分钟级 | 秒级 / 毫秒级 |
| **资源占用** | GB 级（每个 VM 一个 OS） | MB 级（只打包应用和依赖） |
| **性能损耗** | 5%-15%（Hypervisor 开销） | 接近原生（&lt; 1%） |
| **密度** | 单机 10-20 个 VM | 单机数百个容器 |
| **代表技术** | VMware、VirtualBox、KVM | Docker、Podman、containerd |

## Q2: Docker 核心概念

Docker 的核心概念只有 5 个，理解它们之间的关系就掌握了 Docker 的全貌。

```text
┌───────────────────────────────────────────────────────┐
│                    Docker 核心概念                      │
│                                                       │
│  Dockerfile ──build──→ 镜像(Image) ──run──→ 容器      │
│                                    │                  │
│                              Docker Compose           │
│                             (多容器编排)               │
│                                                       │
│  数据卷(Volume) ←──挂载──→ 容器(Container)            │
└───────────────────────────────────────────────────────┘
```

### 镜像（Image）

镜像是一个**只读的模板**，包含了运行应用所需的一切：代码、运行时、库、环境变量、配置文件。你可以把镜像理解为一个"快照"——它不会变化，每次修改都是创建一个新镜像。

```bash
# 拉取镜像
docker pull node:20-alpine

# 查看本地镜像
docker images

# 构建镜像（从 Dockerfile）
docker build -t my-app:1.0 .
```

镜像采用**分层存储**（Layer）结构。每一层都是只读的，多个镜像可以共享相同的层，极大地节省存储空间。

```text
┌─────────────────────────────┐
│  Layer 5: COPY . .          │  ← 你的应用代码（最常变化）
├─────────────────────────────┤
│  Layer 4: RUN npm install   │  ← 依赖安装
├─────────────────────────────┤
│  Layer 3: WORKDIR /app      │  ← 工作目录
├─────────────────────────────┤
│  Layer 2: RUN apk add ...   │  ← 系统依赖
├─────────────────────────────┤
│  Layer 1: node:20-alpine    │  ← 基础镜像（很少变化）
└─────────────────────────────┘
```

### 容器（Container）

容器是镜像的**运行实例**。一个镜像可以创建多个容器，每个容器之间相互隔离。容器的本质是一个进程，但它拥有自己独立的文件系统、网络和进程空间。

```bash
# 创建并启动容器
docker run -d -p 3000:3000 --name my-app my-app:1.0

# 查看运行中的容器
docker ps

# 进入容器内部
docker exec -it my-app /bin/sh

# 停止并删除容器
docker stop my-app && docker rm my-app
```

<Warning title="⚠️ 容器的生命周期">

容器是"用完即弃"的。不要在容器里手动修改文件或安装依赖——这些修改在容器销毁后会丢失。如果需要持久化数据，应该使用数据卷（Volume）。

- `docker stop`：优雅停止（发送 SIGTERM，等待 10 秒后 SIGKILL）
- `docker kill`：强制停止（直接发送 SIGKILL）
- `docker rm`：删除已停止的容器

</Warning>

### Dockerfile

Dockerfile 是一个**文本文件**，定义了如何构建镜像。它由一系列指令组成，每条指令都会创建一个新的镜像层。

```dockerfile
# 基础镜像
FROM node:20-alpine

# 设置工作目录
WORKDIR /app

# 复制依赖文件（利用缓存）
COPY package.json package-lock.json ./

# 安装依赖
RUN npm ci --production

# 复制源代码
COPY . .

# 构建前端项目
RUN npm run build

# 暴露端口
EXPOSE 3000

# 启动命令
CMD ["node", "server.js"]
```

### Docker Compose

Docker Compose 用于**定义和运行多容器应用**。通过一个 `docker-compose.yml` 文件，你可以同时启动前端、后端、数据库、Redis 等多个服务。

```yaml
version: '3.8'
services:
  frontend:
    build: ./frontend
    ports:
      - "3000:3000"
  backend:
    build: ./backend
    ports:
      - "8080:8080"
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
```

### 数据卷（Volume）

数据卷用于**持久化容器中的数据**。容器销毁后，数据卷中的数据不会丢失。

```bash
# 创建数据卷
docker volume create my-data

# 挂载数据卷到容器
docker run -v my-data:/app/data my-app:1.0

# 挂载宿主机目录到容器（开发环境常用）
docker run -v $(pwd)/src:/app/src my-app:1.0
```

<Tip title="🎢 通俗比喻: 租房子">

镜像就像一栋**毛坯房的图纸**——它定义了房子的结构，但图纸本身不能住人。容器就是按图纸**盖出来的实体房**——你可以同时盖 10 栋一模一样的房子。Dockerfile 是**施工说明书**，告诉工人（Docker Engine）怎么一步步把图纸变成房子。数据卷就是**保险柜**——即使房子拆了，保险柜里的东西还在。

</Tip>

## Q3: Docker 底层原理

Docker 容器并不是一个轻量级虚拟机——它本质上就是宿主机上的一个**普通进程**。Docker 通过三项 Linux 内核技术实现了"看起来像独立系统"的效果。

```text
┌───────────────────────────────────────────────────┐
│                  Docker 容器                        │
│  ┌─────────────────────────────────────────────┐  │
│  │  Namespace（隔离：进程、网络、文件系统）      │  │
│  ├─────────────────────────────────────────────┤  │
│  │  Cgroups（限制：CPU、内存、IO）              │  │
│  ├─────────────────────────────────────────────┤  │
│  │  UnionFS（分层：只读层 + 可写层）            │  │
│  └─────────────────────────────────────────────┘  │
│                     │                              │
│              共享宿主机内核                         │
│                     │                              │
│  ┌─────────────────────────────────────────────┐  │
│  │           Linux Kernel                       │  │
│  └─────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────┘
```

### 命名空间（Namespace）

Namespace 是 Linux 内核提供的一种**资源隔离**机制。Docker 使用 6 种 Namespace 来隔离容器的运行环境：

| Namespace | 隔离内容 | 效果 |
|-----------|---------|------|
| **PID** | 进程 ID | 容器内只能看到自己的进程，PID 从 1 开始 |
| **NET** | 网络 | 每个容器有独立的网卡、IP、端口 |
| **MNT** | 文件系统挂载 | 每个容器有独立的文件系统视图 |
| **UTS** | 主机名 | 每个容器可以有自己的 hostname |
| **IPC** | 进程间通信 | 容器间的信号量、消息队列隔离 |
| **USER** | 用户 ID | 容器内的 root 不等于宿主机的 root |

```bash
# 查看容器的 PID Namespace
docker exec my-app ps aux
# PID 1 是容器的主进程，而不是宿主机的 init

# 查看容器的 NET Namespace
docker exec my-app ifconfig
# 容器有独立的网卡和 IP 地址
```

### 控制组（Cgroups）

Cgroups（Control Groups）用于**限制和监控**容器可以使用的资源。没有 Cgroups，一个容器可以吃掉宿主机的全部资源，导致其他容器和宿主机本身无法运行。

```bash
# 限制容器最多使用 1 个 CPU 核心和 512MB 内存
docker run -d --cpus="1" --memory="512m" my-app:1.0

# 查看容器的资源使用情况
docker stats my-app
```

| 资源类型 | 限制参数 | 说明 |
|---------|---------|------|
| **CPU** | `--cpus` | 限制 CPU 核心数 |
| **内存** | `--memory` | 限制最大内存 |
| **IO** | `--device-read-bps` | 限制磁盘读取速度 |
| **网络** | `--network` | 限制网络带宽 |

### 联合文件系统（UnionFS）

UnionFS 是 Docker 镜像分层存储的基础。它将多个目录（层）叠加在一起，对外表现为一个统一的文件系统。每一层都是只读的，只有最上面的"容器层"是可写的。

```text
┌─────────────────────────────┐
│  可写层（Container Layer）   │  ← 容器运行时的修改都在这里
├═════════════════════════════╡  ← "写时复制" (Copy-on-Write)
│  镜像层 5: COPY . .         │  ← 只读
├─────────────────────────────┤
│  镜像层 4: RUN npm ci       │  ← 只读
├─────────────────────────────┤
│  镜像层 3: WORKDIR /app     │  ← 只读
├─────────────────────────────┤
│  镜像层 2: RUN apk add      │  ← 只读
├─────────────────────────────┤
│  镜像层 1: node:20-alpine   │  ← 只读（基础镜像）
└─────────────────────────────┘
```

**写时复制（Copy-on-Write）**：当容器需要修改一个只读层的文件时，UnionFS 会先把文件从只读层复制到可写层，然后在可写层中修改。这保证了只读层不会被改变，多个容器可以安全地共享同一个镜像。

## Q4: Dockerfile 最佳实践

### 多阶段构建（Multi-stage Build）

多阶段构建是前端镜像精简的核心技术。它允许在一个 Dockerfile 中使用多个 `FROM` 指令，每个阶段可以独立选择基础镜像，最后只把需要的产物复制到最终镜像中。

```dockerfile
# ============ 阶段 1: 构建 ============
FROM node:20-alpine AS builder

WORKDIR /app

# 先复制依赖文件（利用缓存）
COPY package.json package-lock.json ./
RUN npm ci

# 再复制源代码并构建
COPY . .
RUN npm run build

# ============ 阶段 2: 生产环境 ============
FROM nginx:alpine

# 从阶段 1 复制构建产物
COPY --from=builder /app/dist /usr/share/nginx/html

# 复制 nginx 配置
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

```text
镜像体积对比：

单阶段构建（node:20-alpine 全量）：  ~1.2 GB
多阶段构建（nginx:alpine + dist）：  ~25 MB

体积缩减 98%！
```

### 镜像体积优化

```text
优化策略：

1. 选择小基础镜像     → alpine > slim > full
2. 合并 RUN 指令      → 减少镜像层数
3. 多阶段构建         → 只保留构建产物
4. 清理缓存和临时文件 → 减少每层的体积
5. 使用 .dockerignore  → 不复制无关文件
```

**❌ 反面教材：**

```dockerfile
FROM node:20          # 1GB+ 的完整镜像
WORKDIR /app
COPY . .              # 把 node_modules、.git 都复制进去了
RUN npm install       # 又装了一遍依赖
RUN npm run build
```

**✅ 最佳实践：**

```dockerfile
FROM node:20-alpine AS builder   # 选用 alpine（~180MB）
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci                        # 单独安装依赖（利用缓存）
COPY . .
RUN npm run build

FROM nginx:alpine                 # 最终镜像只有 nginx（~25MB）
COPY --from=builder /app/dist /usr/share/nginx/html
```

### 缓存优化

Docker 按层缓存。如果某一层没有变化，Docker 会直接使用缓存，不会重新执行。关键原则：**把变化频率低的指令放在前面，变化频率高的放在后面。**

```dockerfile
# ✅ 正确顺序：依赖文件变化少，源代码变化多
COPY package.json package-lock.json ./   # 变化频率：低
RUN npm ci                                # 依赖安装（可缓存）
COPY . .                                  # 源代码（变化频率：高）
RUN npm run build                         # 构建（每次都要执行）

# ❌ 错误顺序：每次改一行代码都要重新安装依赖
COPY . .                                  # 每次都变 → 缓存失效
RUN npm ci                                # 每次都要重新安装！
RUN npm run build
```

```text
缓存失效的连锁反应：

改了一行 JS 代码
  → COPY . . 缓存失效
    → 如果 COPY package.json 在后面，npm ci 也要重新执行
      → 多等 2 分钟下载依赖
```

### 安全最佳实践

| 实践 | 说明 |
|------|------|
| **不以 root 运行** | 使用 `USER` 指令切换到非 root 用户 |
| **不安装不必要的包** | 减小攻击面 |
| **使用固定版本标签** | 避免 `node:latest`，用 `node:20.11.0-alpine` |
| **扫描镜像漏洞** | 使用 `docker scout` 或 `trivy` |
| **不存储密钥** | 使用环境变量或 Docker Secrets |

```dockerfile
# 安全示例
FROM node:20-alpine

# 创建非 root 用户
RUN addgroup -g 1001 appgroup && \
    adduser -u 1001 -G appgroup -s /bin/sh -D appuser

WORKDIR /app
COPY --chown=appuser:appgroup package.json package-lock.json ./
RUN npm ci --production
COPY --chown=appuser:appgroup . .

# 切换到非 root 用户
USER appuser

EXPOSE 3000
CMD ["node", "server.js"]
```

<Tip title="💡 面试技巧">

回答 Dockerfile 最佳实践时，重点讲多阶段构建和缓存优化——这是前端面试的高频考点。安全实践可以作为加分项。

</Tip>

## Q5: Docker 在前端的应用

### 开发环境容器化

把开发环境装进 Docker，保证团队每个人用的环境完全一致。

```yaml
# docker-compose.dev.yml
version: '3.8'
services:
  frontend:
    build:
      context: .
      dockerfile: Dockerfile.dev
    ports:
      - "3000:3000"
    volumes:
      - ./src:/app/src          # 热更新：本地代码变化，容器内实时同步
      - /app/node_modules       # 排除 node_modules（使用容器内的）
    environment:
      - NODE_ENV=development
```

```dockerfile
# Dockerfile.dev
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
EXPOSE 3000
CMD ["npm", "run", "dev"]
```

### CI/CD 流水线

Docker 让 CI/CD 流水线变得可预测和可复现。

```text
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  代码提交 │ → │ CI 构建  │ → │ 镜像推送 │ → │ K8s 部署 │
│  (Git)   │   │ (构建+测试│   │ (镜像仓库)│   │ (生产环境)│
└──────────┘   └──────────┘   └──────────┘   └──────────┘
```

```yaml
# .github/workflows/deploy.yml（GitHub Actions）
name: Build and Deploy

  push:
    branches: [main]

  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build Docker image
```

## Q6: Docker Compose 详解

### docker-compose.yml 语法

```yaml
version: '3.8'

services:
  # 服务名称
  web:
    # 构建配置
    build:
      context: .              # 构建上下文（Dockerfile 所在目录）
      dockerfile: Dockerfile   # 指定 Dockerfile 文件名
      args:                    # 构建参数
        NODE_ENV: production
      target: builder           # 多阶段构建的目标阶段

    # 镜像（如果指定了 build，则 build 后的镜像名）
    image: my-app:1.0

    # 端口映射（宿主机:容器）
    ports:
      - "3000:3000"
      - "8080:8080"

    # 环境变量
    environment:
      - NODE_ENV=production
      - DB_HOST=postgres
    # 或使用 .env 文件
    env_file:
      - .env

    # 数据卷
    volumes:
      - ./src:/app/src          # 绑定挂载
      - node_modules:/app/node_modules  # 命名卷

    # 依赖关系
    depends_on:
      - postgres
      - redis

    # 重启策略
    restart: unless-stopped

    # 网络
    networks:
      - app-network

    # 资源限制
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M

# 定义数据卷
volumes:
  node_modules:
  pg-data:

# 定义网络
networks:
  app-network:
    driver: bridge
```

### 服务编排

```bash
# 启动所有服务（后台运行）
docker compose up -d

# 查看服务状态
docker compose ps

# 查看服务日志
docker compose logs -f web

# 只重建某一个服务
docker compose up -d --build web

# 停止并删除所有容器
docker compose down

# 停止并删除所有容器 + 数据卷
docker compose down -v
```

### 网络配置

Docker Compose 会自动创建一个默认网络，所有服务都在同一网络中，可以通过**服务名**互相访问。

```text
┌─────────────────────────────────────────────┐
│          默认网络 (app-network)               │
│                                             │
│   web:3000 ←──→ backend:8080 ←──→ db:5432  │
│                                             │
│   服务名就是主机名：                          │
│   web 可以用 http://backend:8080 访问后端    │
│   backend 可以用 postgres://db:5432 连数据库  │
└─────────────────────────────────────────────┘
```

```yaml
# 自定义网络（隔离不同环境）
networks:
  frontend-net:
    driver: bridge
  backend-net:
    driver: bridge

services:
  web:
    networks:
      - frontend-net            # 只能访问前端网络
  backend:
    networks:
      - frontend-net
      - backend-net             # 同时在两个网络中
  db:
    networks:
      - backend-net             # 只在后端网络（web 无法直接访问）
```

### 数据持久化

```yaml
services:
  postgres:
    image: postgres:16-alpine
    volumes:
      - pg-data:/var/lib/postgresql/data    # 命名卷（Docker 管理）
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # 绑定挂载（初始化脚本）

  web:
    build: .
    volumes:
      - ./src:/app/src          # 绑定挂载（开发环境热更新）

volumes:
  pg-data:                      # 命名卷声明
    driver: local
```

<Warning title="⚠️ 命名卷 vs 绑定挂载">

- **命名卷（Named Volume）**：由 Docker 管理，存储在 Docker 的数据目录中。适合数据库等需要持久化的数据。
- **绑定挂载（Bind Mount）**：映射宿主机的目录到容器。适合开发环境的热更新。

不要在生产环境使用绑定挂载——它依赖宿主机的目录结构，不具备可移植性。

</Warning>

<AccordionGroup>

  <Accordion title="Docker 和虚拟机的区别？" icon="layer-group">

    **答题要点：**
    不能只背表格，要从底层原理解释为什么容器更轻量。

    **高分回答：**
    "Docker 容器和虚拟机的核心区别在于**隔离级别**。虚拟机通过 Hypervisor 模拟完整的硬件环境，每个 VM 都需要运行一个完整的 Guest OS，启动时间在分钟级别，占用 GB 级资源。而 Docker 容器共享宿主机的 Linux 内核，通过 Namespace 隔离进程视图（PID、NET、MNT 等），通过 Cgroups 限制资源使用（CPU、内存），启动时间在秒级，占用 MB 级资源。

    但这也意味着容器的隔离性不如虚拟机——所有容器共享同一个内核，如果内核有漏洞，所有容器都受影响。所以在多租户场景下，虚拟机仍然更安全。Docker 适合同一信任域内的环境隔离，虚拟机适合跨信任域的硬隔离。"

  </Accordion>

  <Accordion title="Dockerfile 的 COPY 和 ADD 有什么区别？" icon="layer-group">

    **答题要点：**
    ADD 有隐式行为，COPY 更透明。面试官想听的是你理解"隐式行为的风险"。

    **高分回答：**
    "`COPY` 只做一件事：把本地文件复制到镜像中。`ADD` 除了复制文件，还有两个额外功能：**自动解压 tar 包**和**支持远程 URL**。

    ```dockerfile
    # COPY：简单、透明、可预测
    COPY package.json /app/

    # ADD：自动解压（有时候不是你想要的）
    ADD archive.tar.gz /app/    # 自动解压到 /app/

    # ADD：支持远程 URL（但不推荐）
    ADD https://example.com/file.txt /app/
    ```

    最佳实践是**优先使用 COPY**，因为它的行为是确定的。只有在需要自动解压 tar 包时才用 ADD。使用 ADD 的远程 URL 功能更是不推荐——因为它不会自动更新缓存，而且可以在镜像中留下不需要的中间层。"

  </Accordion>

  <Accordion title="Docker 的多阶段构建是什么？怎么用？" icon="stopwatch">

    **答题要点：**
    多阶段构建是前端面试的重点，要讲清楚"为什么要分阶段"和"体积对比"。

    **高分回答：**
    "多阶段构建允许在一个 Dockerfile 中使用多个 `FROM` 指令，每个阶段独立选择基础镜像。关键在于：**最终镜像只包含最后一个阶段的内容**。

    对前端项目来说，构建需要 Node.js 环境和 `node_modules`（几百 MB），但最终运行只需要 Nginx 和静态文件（几十 MB）。多阶段构建把'构建阶段'和'运行阶段'分开：

    ```dockerfile
    # 阶段 1：构建（需要 Node.js）
    FROM node:20-alpine AS builder
    WORKDIR /app
    COPY package*.json ./
    RUN npm ci
    COPY . .
    RUN npm run build

    # 阶段 2：运行（只需要 Nginx）
    FROM nginx:alpine
    COPY --from=builder /app/dist /usr/share/nginx/html
    ```

    镜像体积从 ~1.2GB 缩减到 ~25MB，缩减了 98%。这不仅节省存储和传输时间，更重要的是**减小了攻击面**——最终镜像里没有 Node.js、没有 npm、没有源代码，只有 Nginx 和静态文件。"

  </Accordion>

  <Accordion title="怎么优化 Docker 镜像体积？" icon="stopwatch">

    **答题要点：**
    要给出具体可执行的优化手段，而不是泛泛而谈。

    **高分回答：**
    "优化 Docker 镜像体积有 5 个核心策略：

    **1. 选择小基础镜像**
    ```dockerfile
    FROM node:20          # ~1GB（Debian 完整版）
    FROM node:20-slim     # ~200MB（Debian 精简版）
    FROM node:20-alpine   # ~50MB（Alpine Linux）
    ```

    **2. 多阶段构建**——只保留构建产物，丢弃构建工具。

    **3. 合并 RUN 指令**——每个 RUN 创建一层，合并可以减少层数和缓存体积：
    ```dockerfile
    # ❌ 三层
    RUN apk add --no-cache python3
    RUN apk add --no-cache make
    RUN apk add --no-cache g++

    # ✅ 一层
    RUN apk add --no-cache python3 make g++
    ```

    **4. 清理缓存**——在同一个 RUN 中安装和清理：
    ```dockerfile
    RUN apk add --no-cache python3 && \
        rm -rf /var/cache/apk/*
    ```

    **5. 使用 .dockerignore**——排除 node_modules、.git、测试文件等不必要文件。

    实际项目中，组合使用这 5 种策略，通常可以把镜像从 1GB+ 压缩到 50MB 以内。"

  </Accordion>

  <Accordion title="Docker 的网络模式有哪些？" icon="node">

    **答题要点：**
    讲清楚 bridge 和 host 的区别，以及什么场景用什么模式。

    **高分回答：**
    "Docker 有 4 种主要的网络模式：

    **1. bridge（默认）**：每个容器有独立的 Network Namespace，通过 docker0 网桥互相通信。适合大多数场景。
    ```bash
    docker run --network bridge my-app
    ```

    **2. host**：容器直接使用宿主机的网络栈，没有网络隔离。性能最好，但端口可能冲突。
    ```bash
    docker run --network host my-app
    ```

    **3. none**：没有网络，完全隔离。适合安全敏感的场景。
    ```bash
    docker run --network none my-app
    ```

    **4. overlay**：跨主机的容器通信，用于 Docker Swarm 或 Kubernetes 集群。

    前端开发中，**bridge 模式**最常用。它提供了足够的隔离性，同时通过端口映射（`-p 3000:3000`）让宿主机可以访问容器内的服务。"

  </Accordion>

  <Accordion title="Docker Compose 和 Kubernetes 的区别？" icon="node">

    **答题要点：**
    讲清楚适用场景——Compose 适合开发和小规模部署，K8s 适合大规模生产。

    **高分回答：**
    "Docker Compose 和 Kubernetes 解决的是不同规模的问题：

    **Docker Compose** 适合单机环境，通过一个 YAML 文件定义和运行多容器应用。它很简单，一个 `docker compose up` 就能启动整个开发环境。但它**不支持跨主机部署、自动扩缩容、滚动更新**等生产级功能。

    **Kubernetes** 是一个容器编排平台，适合大规模生产环境。它支持跨主机部署、自动扩缩容、服务发现、负载均衡、滚动更新、自愈能力等。

    简单来说：
    - 开发环境、小项目、单机部署 → Docker Compose
    - 生产环境、多服务、需要高可用 → Kubernetes

    很多团队的流程是：开发时用 Docker Compose，部署到 K8s 时把 `docker-compose.yml` 转换成 K8s 的 Deployment + Service YAML。"

  </Accordion>

  <Accordion title="Dockerfile 中 CMD 和 ENTRYPOINT 的区别？" icon="skull">

    **答题要点：**
    CMD 可以被覆盖，ENTRYPOINT 不容易被覆盖。两者的组合使用是关键。

    **高分回答：**
    "`CMD` 和 `ENTRYPOINT` 都用于指定容器启动时运行的命令，但行为不同：

    **CMD**：设置默认命令，但可以被 `docker run` 的参数覆盖。
    ```dockerfile
    CMD ["node", "server.js"]
    # docker run my-app              → 运行 node server.js
    # docker run my-app npm test     → 运行 npm test（CMD 被覆盖）
    ```

    **ENTRYPOINT**：设置固定入口命令，`docker run` 的参数会作为 ENTRYPOINT 的参数。
    ```dockerfile
    ENTRYPOINT ["node"]
    CMD ["server.js"]
    # docker run my-app              → 运行 node server.js
    # docker run my-app app.js       → 运行 node app.js（CMD 被覆盖）
    ```

    **最佳实践**：用 ENTRYPOINT 定义主命令，用 CMD 定义默认参数。这样用户可以通过传参自定义行为，而不会意外覆盖主命令。"

  </Accordion>

  <Accordion title="Docker 数据卷和绑定挂载的区别？什么时候用哪个？" icon="skull">

    **答题要点：**
    讲清楚两者的数据管理方式和适用场景。

    **高分回答：**
    "数据卷（Named Volume）和绑定挂载（Bind Mount）都是 Docker 的持久化方案，但管理方式不同：

    **命名卷**由 Docker 管理，存储在 `/var/lib/docker/volumes/` 目录下。适合数据库等需要持久化的数据：
    ```bash
    docker volume create pg-data
    docker run -v pg-data:/var/lib/postgresql/data postgres
    ```

    **绑定挂载**直接映射宿主机的目录到容器。适合开发环境的热更新：
    ```bash
    docker run -v $(pwd)/src:/app/src node-app
    ```

    **选择原则**：
    - 生产环境持久化数据 → 命名卷（可移植、Docker 管理生命周期）
    - 开发环境代码热更新 → 绑定挂载（实时同步、方便调试）
    - CI/CD 临时数据 → 不需要持久化（容器销毁即清理）

    一个常见错误是在开发环境用绑定挂载覆盖了容器内的 `node_modules`，导致依赖冲突。解决方法是先用匿名卷占位：`-v /app/node_modules`。"

  </Accordion>

  <Accordion title="Docker 的分层缓存机制是怎么工作的?" icon="stopwatch">

  **答题要点：** 每条指令创建一层，未变化的层直接用缓存。关键原则：变化频率低的指令放前面。

  **高分回答：**
  "Docker 的镜像由多个只读层叠加而成，每条 Dockerfile 指令（RUN、COPY、ADD 等）创建一层。构建时，Docker 会检查每一层是否变化——如果某一层的输入（指令 + 上下文）没变，就直接用缓存，不重新执行。关键原则是把变化频率低的指令放在前面：`COPY package.json` → `RUN npm ci` → `COPY . .` → `RUN npm build`。这样改一行 JS 代码，只有最后两层失效，前面的依赖安装层直接命中缓存。如果把 `COPY . .` 放在 `npm ci` 前面，每次改代码都要重新安装依赖。"

  </Accordion>

  **高分回答：**
  "虚拟机通过 Hypervisor 模拟完整的硬件环境，每个 VM 都需要运行一个完整的 Guest OS，启动时间在分钟级别，占用 GB 级资源。Docker 容器共享宿主机的 Linux 内核，通过 Namespace 隔离进程视图（PID、NET、MNT 等），通过 Cgroups 限制资源使用（CPU、内存），启动时间在秒级，占用 MB 级资源。但容器的隔离性不如虚拟机——所有容器共享同一个内核，如果内核有漏洞，所有容器都受影响。所以多租户场景下虚拟机更安全，同一信任域内容器更高效。"

  </Accordion>

  <Accordion title="Docker Compose 中服务间怎么通信?" icon="node">

  **答题要点：** Docker Compose 自动创建默认网络，服务间通过服务名互相访问。

  **高分回答：**
  "Docker Compose 会自动创建一个默认 bridge 网络，所有服务都在同一网络中。服务间通过**服务名**作为主机名互相访问——比如前端容器可以用 `http://backend:8080` 访问后端，后端可以用 `postgres://db:5432` 连接数据库。不需要知道容器的 IP 地址。如果需要隔离不同环境（如前端只能访问 API 网关，不能直接访问数据库），可以用自定义网络把服务分配到不同的网络中。"

  </Accordion>

  <Accordion title="命名卷和绑定挂载的区别?什么时候用哪个?" icon="node">

  **答题要点：** 命名卷由 Docker 管理（适合持久化），绑定挂载映射宿主机目录（适合开发热更新）。

  **高分回答：**
  "命名卷（Named Volume）由 Docker 管理，存储在 `/var/lib/docker/volumes/` 目录下，适合数据库等需要持久化的数据。绑定挂载（Bind Mount）直接映射宿主机的目录到容器，适合开发环境的热更新——本地代码变化，容器内实时同步。选择原则：生产环境持久化数据用命名卷（可移植、Docker 管理生命周期），开发环境代码热更新用绑定挂载。一个常见错误是绑定挂载覆盖了容器内的 node_modules，解决方法是先用匿名卷占位：`-v /app/node_modules`。"

  </Accordion>

  <Accordion title="🔥 Dockerfile 中 CMD 和 ENTRYPOINT 的区别?" icon="skull">

  **答题要点：** CMD 可被 docker run 参数覆盖，ENTRYPOINT 不容易被覆盖。两者组合使用是关键。

  **高分回答：**
  "`CMD` 设置默认命令，可以被 `docker run` 的参数覆盖。`ENTRYPOINT` 设置固定入口命令，docker run 的参数会作为 ENTRYPOINT 的参数。最佳实践是用 ENTRYPOINT 定义主命令，用 CMD 定义默认参数：
  ```dockerfile
  ENTRYPOINT ["node"]
  CMD ["server.js"]
  ```
  这样 `docker run my-app` 运行 `node server.js`，`docker run my-app app.js` 运行 `node app.js`。用户可以通过传参自定义行为，而不会意外覆盖主命令。"

  </Accordion>

  <Accordion title="🔥 怎么扫描 Docker 镜像的安全漏洞?" icon="skull">

  **答题要点：** 用 docker scout 或 trivy 扫描，CI 中集成漏洞检查。

  **高分回答：**
  "扫描 Docker 镜像漏洞有两种主流工具：
  1. **Docker Scout**：Docker 官方内置，`docker scout cves my-app:latest` 直接扫描
  2. **Trivy**：开源工具，`trivy image my-app:latest` 扫描镜像中的 CVE 漏洞

  最佳实践是在 CI/CD 流水线中集成镜像扫描，构建后自动检查漏洞。如果发现高危漏洞（CRITICAL/HIGH），阻止部署。同时用固定版本标签（如 `node:20.11.0-alpine` 而不是 `node:latest`），避免基础镜像突然引入漏洞。"

  </Accordion>
