## Q1: Vue 和 React 的设计理念有什么本质区别？

<Steps>

  <Step title="React 的理念: JavaScript 驱动一切">

  React 的核心哲学是"UI = f(state)'--UI 是状态的函数。它把 HTML、CSS、JavaScript 全部交给 JavaScript 管理（JSX），模板就是 JavaScript 表达式。React 不提供"官方推荐"的方案，而是把选择权交给开发者（路由、状态管理、CSS 方案都是社区提供）。

  </Step>
  <Step title="Vue 的理念: 渐进式增强">

  Vue 的核心哲学是"渐进式框架'--你可以只用核心库，也可以逐步引入路由、状态管理、构建工具。Vue 提供"官方推荐"的完整方案（Vue Router、Pinia、Vite），模板是增强的 HTML（指令、插槽）。Vue 追求"开箱即用"，降低开发者的心智负担。

  </Step>
  <Step title="设计哲学的根本分歧">

  React 说："我给你自由，你自己选。' Vue 说："我给你推荐，你跟着走。" React 的灵活性是优势也是负担--你需要做更多技术决策。Vue 的完整性是优势也是限制--你被绑在 Vue 的生态里。

  </Step>
  <Step title="响应式机制的本质差异">

  Vue 采用**基于 Proxy 的细粒度响应式**--框架精确知道哪些组件依赖了哪些数据，只更新受影响的组件。React 采用**基于 setState 的全量 re-render**--状态变化后整个组件子树重新执行，通过 diff 找出差异。这两种机制各有优劣：Vue 的默认性能更好，React 的心智模型更简单。

  </Step>
  <Step title='关键启示：选型取决于团队和项目，而非技术优劣'>

  面试中回答"Vue vs React"时，千万不要说"哪个更好"--这是陷阱题。正确的思路是：React 适合需要高度灵活、技术决策能力强的团队；Vue 适合需要快速上手、统一技术栈的团队。真正的高级工程师不是选"最好的框架"，而是根据项目需求和团队能力做出最合适的选择。

  </Step>

</Steps>

**设计理念解决的核心问题：**

```text
问题：前端框架应该"提供完整方案"还是"只提供核心能力"？

React 的选择：只提供核心能力（React + ReactDOM），路由/状态管理/样式方案全部交给社区
→ 优势：灵活、生态丰富、可以自由组合
→ 劣势：技术决策多、新手容易迷失

Vue 的选择：提供完整方案（Vue + Vue Router + Pinia + Vite），官方推荐最佳实践
→ 优势：开箱即用、学习曲线低、团队统一
→ 劣势：灵活性低、被绑在 Vue 生态
```

<Note title="💡 一句话总结">

React 是"自由市场"，Vue 是"计划经济"--没有绝对的好坏，取决于团队规模和项目复杂度。

</Note>

<CardGroup cols={2}>

<Card title="🚫 React 的自由选择">

React 只提供核心能力，路由/状态管理/CSS 方案全部交给社区。就像去一个没有菜单的餐厅--你可以点任何菜，但你得自己决定吃什么，新手容易选择困难。

</Card>

Vue 提供完整方案，官方推荐 Vue Router、Pinia、Vite。就像去一个有套餐的餐厅--厨师已经帮你搭配好了，你只需要选套餐 A 还是套餐 B。

</Card>

## Q2: 响应式机制有什么本质区别？

### 两种响应式模型

Vue 和 React 的响应式机制是两者最核心的差异之一。

```text
┌───────────────────────────────────────┐
│        Vue 的响应式：自动追踪           │
│  Proxy 拦截属性访问 → track 收集依赖   │
│  属性变化 → trigger 精确更新           │
│  组件只更新依赖了该属性的部分           │
└───────────────────────────────────────┘

┌───────────────────────────────────────┐
│        React 的响应式：手动声明         │
│  useState/useReducer 声明状态         │
│  setState 触发整个组件重新渲染         │
│  需要 useMemo/useCallback 手动优化    │
└───────────────────────────────────────┘
```

| 对比维度 | Vue | React |
|---------|-----|-------|
| **响应式原理** | Proxy 自动追踪依赖 | setState 触发整个组件重渲染 |
| **更新粒度** | 精确到属性级别 | 粗粒度（整个组件） |
| **手动优化** | 基本不需要 | 需要 useMemo/useCallback |
| **心智模型** | "改了数据，视图自动更新" | "改了数据，重新渲染整个组件" |
| **性能开销** | Proxy 的 get/set trap | 重渲染 + diff |

<Warning title="⚠️ 争议性问题: Vue 的响应式比 React 更好吗?">

表面上：Vue 的自动追踪更"聪明'，不需要手动优化。
反面：Proxy 有运行时开销（每次属性访问都经过 trap），React 的直接函数调用更轻量。Vue 的"聪明"也有代价--依赖收集的三层 Map（WeakMap → Map → Set）增加了内存开销。
判断：对于中小型应用，Vue 的响应式更省心；对于大型应用，React 的手动优化给了开发者更多控制权。
升华：Vue 是"框架帮你优化"，React 是"你来优化"--两种哲学，没有绝对的好坏。

</Warning>

<Tip title="🎢 通俗比喻: 自动驾驶 vs 手动挡">

Vue 的响应式就像自动驾驶--你踩油门（改数据），车自动开（视图自动更新）。React 的响应式就像手动挡--你踩油门（setState），还得自己换挡（useMemo/useCallback 优化）。自动驾驶更省心，但老司机更喜欢手动挡的控制感。

</Tip>

## Q3: Diff 算法有什么区别？

### 基础梳理

| 对比维度 | Vue 2 | Vue 3 | React |
|---------|-------|-------|-------|
| **遍历方式** | 双端比较（4 个指针） | 左右 + LIS 优化 | 单向遍历（从左到右） |
| **编译优化** | 无 | PatchFlag + Block Tree + 静态提升 | 无（靠 React Compiler） |
| **时间复杂度** | O(n) | O(n) | O(n) |
| **key 的作用** | 标识节点复用 | 标识节点复用 | 标识节点复用 |
| **静态节点处理** | 每次 diff 都比较 | 编译时跳过 | 每次 diff 都比较 |

### 专家视角深度剖析

### 为什么 Vue 和 React 选择不同的 Diff 策略

很多候选人能背出"Vue 用双端比较，React 用单向遍历'，但说不清**为什么这样设计**。

React 选择单向遍历的原因：React 的设计理念是"UI = f(state)"，每次状态变化都重新渲染整个组件。单向遍历足够简单，配合 Fiber 架构的时间切片，可以把 diff 拆成多个小块执行，不阻塞主线程。React 把优化的重担交给了开发者（useMemo/useCallback）。

Vue 选择双端比较 + LIS 的原因：Vue 的响应式系统已经精确知道哪些节点变化了，diff 只需要处理变化的部分。编译器的 PatchFlag、Block Tree、静态提升进一步减少了 diff 的工作量。Vue 把优化的重担交给了框架。

```javascript
// Vue 3：编译器标记动态节点，diff 只比较标记的部分
createElementVNode("div", null, [
  _createElementVNode("span", null, "静态"),  // 静态提升，跳过
  _createElementVNode("span", null, _toDisplayString(_ctx.msg), 1 /* TEXT */)
])

// React：每次 diff 比较所有属性
h('div', [
  h('span', '静态'),  // 每次都比较
  h('span', msg)       // 每次都比较
])
// 核心调度：Vue 是"编译时优化"，React 是"运行时优化"
```

## Q4: 状态管理有什么区别？

### 基础梳理

| 对比维度 | Vue（Pinia） | React（Redux/Zustand） |
|---------|------------|----------------------|
| **官方推荐** | Pinia | 无官方推荐（Redux 最流行） |
| **设计理念** | 基于 Vue 的响应式系统 | 独立的状态容器 |
| **API 复杂度** | 简单（state + getters + actions） | Redux 较复杂（reducer + action + dispatch） |
| **TypeScript** | 完美支持 | Redux 较差，Zustand 好 |
| **DevTools** | Vue DevTools 内置支持 | Redux DevTools（需安装） |
| **体积** | ~1KB | Redux ~2KB，Zustand ~1KB |

### 专家视角深度剖析

### 为什么 React 需要外部状态管理

很多候选人知道"Vue 用 Pinia，React 用 Redux"，但说不清**为什么 React 需要外部状态管理而 Vue 不太需要**。

原因是 Vue 的响应式系统本身就是"状态管理"--你可以在任何组件外创建 `reactive` 对象，所有引用它的组件都会自动更新。Pinia 只是把这个能力包装了一下（加了 DevTools、插件、SSR 支持）。

React 的 `useState` 是组件级别的--状态绑定在组件实例上，组件卸载时状态消失。如果要在多个组件间共享状态，必须用 Context（性能差）或外部状态管理库（Redux/Zustand）。

```javascript
// Vue：响应式系统本身就是状态管理
// store.js
export const count = ref(0);  // 全局状态，任何组件都能用

// 组件 A
import { count } from './store';
// 直接用，自动响应

// React：必须用外部状态管理
// store.js
import { create } from 'zustand';
export const useStore = create(set => ({
  count: 0,
  increment: () => set(state => ({ count: state.count + 1 }))
}));

// 组件 A
import { useStore } from './store';
const count = useStore(state => state.count);  // 必须通过 hook 访问
```

<Warning title="⚠️ 争议性问题: Vue 不需要状态管理库吗?">

表面上：Vue 的响应式系统本身就是状态管理，不需要额外的库。
反面：在大型应用中，Pinia 提供了 DevTools 支持、插件系统、SSR 支持、模块化组织等能力，这些是裸 `ref`/`reactive` 做不到的。
判断：小型项目用 `ref`/`reactive` 足够；中大型项目用 Pinia 更好维护。
升华：Vue 的"不需要状态管理'是一种理想状态，实际项目中 Pinia 几乎是必选。

</Warning>

## Q5: 组件模型有什么区别？

| 对比维度 | Vue | React |
|---------|-----|-------|
| **定义方式** | SFC（单文件组件）+ Composition API | 函数组件 + Hooks |
| **模板语法** | 模板（增强的 HTML）或 JSX | 只有 JSX |
| **双向绑定** | `v-model` 原生支持 | 需手动实现受控组件 |
| **生命周期** | `onMounted`/`onUnmounted` 等独立钩子 | `useEffect` 统一处理 |
| **逻辑复用** | Composables | Hooks |
| **条件渲染** | `v-if`/`v-show` | `{condition && <Comp />}` |
| **列表渲染** | `v-for` | `.map()` |

<Warning title="⚠️ 争议性问题: 模板比 JSX 好吗?">

表面上：模板更直观，编译器可以做更多优化（PatchFlag、静态提升）。
反面：JSX 更灵活--它就是 JavaScript 表达式，可以使用所有 JS 能力（条件、循环、高阶函数）。模板的"优化'在大多数场景下收益不大（差异在毫秒级）。
判断：模板适合"描述 UI"的场景，JSX 适合"动态生成 UI"的场景。
升华：Vue 3 同时支持模板和 JSX，让开发者自己选择。React 只有 JSX，因为它认为"JavaScript 就够了"。

</Warning>

## Q6: 数据流有什么区别？

| 对比维度 | Vue | React |
|---------|-----|-------|
| **单向数据流** | ✅ 支持（props 向下，emit 向上） | ✅ 强制（只有 props 向下） |
| **双向绑定** | ✅ `v-model` 原生支持 | ❌ 需手动实现 |
| **状态提升** | 通过 Pinia 或 provide/inject | 通过 Context 或 Redux |
| **副作用管理** | `watch`/`watchEffect` | `useEffect` |

<Warning title="⚠️ 争议性问题: 双向绑定是好设计吗?">

表面上：双向绑定更方便--`v-model` 一行代码搞定表单双向同步。
反面：双向绑定让数据流变得不透明--你不知道数据是从哪里改的。React 强制单向数据流，数据只能从父组件流向子组件，修改只能通过 `setState`，数据流清晰可追踪。
判断：简单表单场景，双向绑定更方便；复杂应用中，单向数据流更可维护。
升华：Vue 3 的 `v-model` 本质上是"单向数据流 + 语法糖'--编译为 `modelValue` prop + `update:modelValue` 事件。它不是真正的"双向绑定"，而是"单向数据流的便捷写法"。

</Warning>

## Q8: Vue 和 React 的虚拟 DOM 实现有什么区别？

### Vue 的编译时优化

Vue 3 的虚拟 DOM 实现引入了大量编译时优化，减少了运行时 diff 的工作量：

```javascript
// Vue 3：编译器标记动态节点
// 模板：<div><span>静态</span><span>{{ msg }}</span></div>
// 编译后：
createElementVNode("div", null, [
  // 静态节点——静态提升，只创建一次
  _createElementVNode("span", null, "静态"),
  // 动态节点——PatchFlag 标记，diff 只比较文本
  _createElementVNode("span", null, _toDisplayString(_ctx.msg), 1 /* TEXT */)
])

// PatchFlag 告诉 diff 算法：这个节点只有文本会变，不需要比较其他属性
```

### React 的运行时优化

React 没有编译时优化（除了实验性的 React Compiler），每次 diff 都比较所有属性：

```javascript
// React：每次 diff 比较所有属性
h('div', [
  h('span', '静态'),  // 每次都比较
  h('span', msg)       // 每次都比较
])

// React 的优化靠开发者手动：useMemo、useCallback、React.memo
```

### Vue 3 的三层编译优化

| 优化技术 | 说明 | 效果 |
|---------|------|------|
| **PatchFlag** | 标记动态绑定的类型（文本、class、style 等） | diff 只比较标记的属性 |
| **Block Tree** | 将动态节点扁平化收集到数组中 | diff 只遍历动态节点，跳过静态子树 |
| **静态提升** | 静态节点只创建一次，后续复用 | 减少对象创建和 GC 压力 |

```text
Vue 3 vs React 的 diff 工作量对比：

假设：1000 个节点的组件，其中 5 个是动态的

Vue 3 diff：
  Block Tree 收集 → 只遍历 5 个动态节点
  PatchFlag 标记 → 每个节点只比较标记的属性
  实际比较：5 个节点 × 1-2 个属性 = ~10 次比较

React diff：
  遍历所有 1000 个节点
  每个节点比较所有属性（children、className、style 等）
  实际比较：1000 个节点 × N 个属性 = ~5000 次比较

结论：Vue 3 在编译时做了大量优化，运行时 diff 的工作量远小于 React
```

<Warning title="⚠️ 但 React 有时间切片">

虽然 Vue 3 的 diff 工作量更小，但 React 有 Fiber 时间切片——可以把 diff 拆成多个小块执行，不阻塞主线程。Vue 3 的 diff 是同步的，一旦开始就必须执行完。对于小型组件，Vue 3 的编译优化更有效；对于超大型组件树，React 的时间切片能保持更好的交互响应。

</Warning>

## 🔍 底层源码：React 和 Vue 的渲染机制有什么区别？

<div class="follow-up">
React 和 Vue 的渲染触发机制有什么区别？
</div>

**Vue 的渲染触发**：响应式数据变化时，`trigger` 函数通知所有依赖该数据的 `ReactiveEffect`。每个组件的渲染函数就是一个 `ReactiveEffect`，只有依赖了变化数据的组件才会重新渲染。详见 [Vue 3 源码 packages/reactivity/src/effect.ts](https://github.com/vuejs/core/blob/main/packages/reactivity/src/effect.ts)。

```javascript
// Vue：精确更新--只有依赖了 count 的组件才重新渲染
const count = ref(0);
// 组件 A 用了 count → 触发更新
// 组件 B 没用 count → 不触发更新
```

**React 的渲染触发**：`setState` 触发整个组件树从根节点开始重新渲染（Fiber 架构的 `scheduleUpdateOnFiber`）。React 通过 `shouldComponentUpdate`、`React.memo`、`useMemo` 等机制让开发者手动控制哪些组件可以跳过渲染。详见 [React 源码 packages/react-reconciler/src/ReactFiberHooks.js](https://github.com/facebook/react/blob/main/packages/react-reconciler/src/ReactFiberHooks.js)。

```javascript
// React：setState 触发整个组件重新渲染
const [count, setCount] = useState(0);
// 组件 A 用了 count → 触发更新
// 组件 B 没用 count → 也会重新渲染（除非用 React.memo 跳过）
```

<div class="follow-up">
React Compiler 和 Vue 的编译优化有什么相似之处？
</div>

React Compiler（原名 React Forget）在编译时自动添加 `React.memo`、`useMemo`、`useCallback`，减少不必要的 re-render。这和 Vue 3 的编译器优化（PatchFlag、Block Tree、静态提升）思路类似--**把运行时的工作提前到编译时**。区别是 Vue 3 的编译器从一开始就设计了这些优化，React Compiler 是后来补上的。

## 十、面试官视角：满分回答 vs 及格回答

### 问题："Vue 和 React 的核心区别是什么？"

**及格回答（60-70分）：**
> "Vue 用模板，React 用 JSX。Vue 有双向绑定，React 只有单向数据流。Vue 用 Proxy，React 用 setState。"

面试官评价：**背区别但不理解设计**。

**满分回答（90-100分）：**

> Vue 和 React 的核心区别不在 API 层面，而在**设计理念**。React 的哲学是"JavaScript 驱动一切"--它只提供核心能力（React + ReactDOM），把选择权交给开发者。Vue 的哲学是"渐进式增强"--它提供完整方案（Vue Router + Pinia + Vite），降低开发者的心智负担。<sup>[1]</sup>

> 在**响应式机制**上，Vue 用 Proxy 自动追踪依赖，数据变化时精确更新依赖了该数据的组件。React 用 setState 触发整个组件重新渲染，需要开发者用 useMemo/useCallback 手动优化。Vue 的方式更"聪明"，但 Proxy 有运行时开销；React 的方式更"直接"，但给了开发者更多控制权。<sup>[2]</sup>

> 在**Diff 算法**上，Vue 3 的编译器做了 PatchFlag、Block Tree、静态提升三层优化，让运行时 diff 只处理动态节点。React 没有这些编译器优化，但引入了 Fiber 架构和时间切片，把 diff 拆成多个小块执行，不阻塞主线程。React Compiler 后来补上了自动 memoization，思路和 Vue 类似。<sup>[3]</sup>

> 技术选型的建议：**中小团队、快速迭代、中后台项目选 Vue**（开箱即用、学习曲线低）。**大型团队、复杂应用、跨端需求选 React**（生态丰富、React Native、灵活度高）。没有绝对的好坏，只有适合不适合。

  <Accordion title="Vue 的响应式和 React 的 setState 有什么本质区别?" icon="layer-group">

  **答题要点：** Vue 是自动追踪依赖，React 是手动声明依赖。

  **高分回答：**
  "Vue 的响应式是**自动追踪**--组件渲染时访问了哪些响应式数据，Proxy 的 get trap 自动收集依赖，数据变化时精确更新。React 的 setState 是**手动声明**--开发者通过 useState 声明状态，setState 触发整个组件重新渲染，需要 useMemo/useCallback 手动控制哪些子组件可以跳过。
  ```javascript
    // Vue：自动追踪，只有用了 count 的组件才更新
    const count = ref(0);
    // 组件 A 用了 count → 自动更新
    // 组件 B 没用 count → 不更新

    // React：手动声明，整个组件重新渲染
    const [count, setCount] = useState(0);
    // 组件 A 用了 count → 重新渲染
    // 组件 B 没用 count → 也重新渲染（除非 React.memo）
```

"

  <Accordion title="为什么 React 需要外部状态管理 (Redux) , 而 Vue 不太需要?" icon="stopwatch">

  **答题要点：** Vue 的响应式系统本身就是状态管理，React 的 useState 是组件级别的。

  **高分回答：**
  "Vue 的响应式系统本身就是'状态管理'--你可以在组件外创建 `ref`/`reactive` 对象，所有引用它的组件都会自动更新。Pinia 只是把这个能力包装了一下（加了 DevTools、插件、SSR 支持）。
  React 的 useState 是组件级别的--状态绑定在组件实例上，组件卸载时状态消失。如果要在多个组件间共享状态，必须用 Context（性能差：值变化时所有 Consumer 都 re-render）或外部状态管理库（Redux/Zustand）。
  ```javascript
    // Vue：组件外创建 ref，任何组件都能用
    export const count = ref(0);

    // React：必须用外部状态管理
    import { create } from 'zustand';
    export const useStore = create(set => ({ count: 0 }));
```

"

  <Accordion title="Vue 和 React 的 Diff 算法有什么区别?" icon="node">

  **答题要点：** Vue 做编译时优化，React 做运行时优化。

  **高分回答：**
  "Vue 3 的编译器在编译时做了 PatchFlag（标记动态绑定）、Block Tree（扁平化动态节点）、静态提升（静态节点只创建一次），让运行时 diff 只处理动态节点。React 没有这些编译器优化，但引入了 Fiber 架构和时间切片--把 diff 拆成多个小块执行，每块执行后检查是否需要让出主线程。
  ```javascript
    // Vue 3：编译器标记，diff 只比较标记的部分
    createElementVNode('div', null, [
      _createElementVNode('span', null, '静态'),  // 静态提升，跳过
      _createElementVNode('span', null, msg, 1 /* TEXT */)  // 只比较文本
    ])

    // React：每次 diff 比较所有属性
    h('div', [
      h('span', '静态'),  // 每次都比较
      h('span', msg)       // 每次都比较
    ])
```

"

  **答题要点：** 这是陷阱题--不能说"Vue 更好"或"React 更好"，要给出具体场景。

  **高分回答：**
  "选型取决于团队规模、项目复杂度、技术栈要求：
1. **中小团队 + 快速迭代 + 中后台**：选 Vue。开箱即用，学习曲线低，官方生态完整（Vue Router + Pinia + Vite），团队不需要做太多技术决策。
2. **大型团队 + 复杂应用 + 跨端需求**：选 React。生态丰富（Redux/Zustand/MobX、React Native、Next.js），灵活度高，社区最大。
3. **已有技术栈**：如果团队已经在用某个框架，不要轻易换。迁移成本远大于框架差异带来的收益。
4. **个人学习**：建议两个都学。理解 Vue 的'自动追踪'和 React 的'手动声明'，对理解前端架构非常有帮助。
  **核心观点**：框架是工具，不是信仰。选哪个不重要，重要的是理解背后的设计理念。"

  <Accordion title="Vue 的 v-model 和 React 的受控组件有什么区别?" icon="layer-group">

  **答题要点：** v-model 是编译器语法糖，受控组件是手动实现的模式。

  **高分回答：**
  "Vue 的 `v-model` 是编译器语法糖——`<input v-model='name' />` 编译为 `value` prop + `onInput` 事件。开发者一行代码搞定双向同步。

  React 的受控组件需要手动实现：`<input value={name} onChange={e => setName(e.target.value)} />`。React 故意不提供语法糖，因为它认为'双向绑定让数据流不透明'。强制单向数据流 + 手动事件处理，让数据流向清晰可追踪。

  但 Vue 3 的 `v-model` 本质上也是单向数据流——它编译为 `modelValue` prop + `update:modelValue` 事件，只是语法上更简洁。所以两者的运行时行为是一样的，区别只是开发体验。"

  </Accordion>

  <Accordion title="Vue 的 provide/inject 和 React 的 Context 有什么区别?" icon="stopwatch">

  **答题要点：** 两者都是跨层级传递数据，但实现机制不同。

  **高分回答：**
  "Vue 的 `provide/inject` 基于响应式系统——provide 一个 `ref`，inject 的组件会自动响应变化。它不触发整个子树 re-render，只有依赖了该数据的组件才更新。

  React 的 Context 基于发布-订阅——Provider 的 value 变化时，所有 Consumer 组件都会 re-render。这是 React Context 的性能瓶颈——即使 Consumer 只用了 value 中的一个属性，整个 value 对象变化也会触发 re-render。

  解决方案：拆分多个 Context、用 `useMemo` 缓存 value、或用 Zustand/Redux 等外部状态管理库替代。Vue 的 provide/inject 没有这个问题，因为响应式系统会精确追踪依赖。"

  </Accordion>

  <Accordion title="Vue 的 watch 和 React 的 useEffect 有什么区别?" icon="node">

  **答题要点：** watch 是响应式数据变化时触发，useEffect 是组件渲染后触发。

  **高分回答：**
  "Vue 的 `watch` 是响应式数据变化时触发——它和响应式系统深度集成，只有被侦听的数据变化时才执行。`watchEffect` 自动追踪依赖，依赖变化时重新执行。

  React 的 `useEffect` 是组件渲染后触发——它不关心具体是哪个状态变化了，只要组件 re-render 了，effect 就会重新执行（除非依赖数组没有变化）。`useEffect` 的心智模型是'同步副作用'，不是'响应数据变化'。

  这导致了一个常见问题：React 的 `useEffect` 经常因为依赖数组不正确而出现 bug（过多或过少的执行）。Vue 的 `watch` 更直观——你明确指定侦听哪个数据，数据变了就执行。"

  </Accordion>

  <Accordion title="🔥 React 的 Concurrent Mode 和 Vue 的批量更新有什么区别?" icon="skull">

  **答题要点：** Concurrent Mode 是时间切片 + 优先级调度，Vue 的批量更新是同步合并。

  **高分回答：**
  "React 18 的 Concurrent Mode 实现了时间切片和优先级调度：把 diff 和渲染拆成多个小块，每块执行后检查是否需要让出主线程（通过 `shouldYield`）。高优先级更新（如用户输入）可以打断低优先级更新（如数据加载），确保交互响应流畅。

  Vue 3 的批量更新是同步合并：同一事件循环内的多次数据修改，只触发一次组件更新。通过 `queueJob` + 微任务实现——同步代码执行完后，在微任务中统一刷新队列。Vue 不能打断更新——一旦开始 diff，就必须执行完。

  核心区别：React 的 Concurrent Mode 是'可以被打断的异步渲染'，Vue 的批量更新是'不可打断的同步渲染'。在简单场景下，Vue 的同步渲染更快（没有调度开销）。在复杂场景下（大量组件同时更新），React 的时间切片能保持更好的交互响应。"

  </Accordion>
