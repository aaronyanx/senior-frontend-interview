## Q1: uni-app 的历史？为什么要有它？

```text
┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  HTML5+ 规范  │ → │  HBuilder    │ → │  多端爆发    │ → │  uni-app     │
│  DCloud 推动  │   │  IDE 生态    │   │  小程序+App  │   │  2018 年发布 │
│  2013 年      │   │              │   │  +H5 割裂    │   │  一套代码    │
└──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
```

<Steps>

  <Step title="2013-2017 年: DCloud 的技术积累">

    DCloud 从 2013 年开始推动 HTML5+ 规范，试图用 Web 技术开发 App。HBuilder IDE 逐渐积累了大量的开发者生态。2017 年微信小程序上线后，DCloud 发现了一个更紧迫的问题：**开发者不仅要写 App（iOS/Android），还要写小程序（微信/支付宝/百度），再加上 H5，一个业务要维护 5+ 个代码仓库。**

  </Step>
  <Step title="2018 年: DCloud 发布 uni-app">

    2018 年，DCloud 正式开源 uni-app。核心理念是：**用 Vue 的语法写一套代码，编译输出到 12+ 个平台——包括微信/支付宝/百度/字节/快手/QQ 小程序、App（iOS/Android）、H5、快应用。** uni-app 的名字取自"universal"（通用），强调全平台覆盖。

  </Step>
  <Step title="2020 年: uni-app 支持 Vue 3">

    随着 Vue 3 的发布，uni-app 快速跟进支持 Vue 3 + Composition API。Vue 3 的响应式系统（基于 Proxy）和更好的 TypeScript 支持，让 uni-app 的开发体验大幅提升。

  </Step>
  <Step title="2023 年: uni-app x 和 uts">

    DCloud 推出了 uni-app x 和 uts（Uni TypeScript）——一种新的跨端语言，目标是在 App 端实现接近原生的性能。uts 编译为 Kotlin（Android）和 Swift（iOS），直接调用原生 API，消除了 JS Bridge 的性能损耗。

  </Step>

</Steps>

**uni-app 解决的核心问题：**

```text
问题：多端开发的碎片化

App（iOS）  ─┐
App（Android）─┤
微信小程序   ─┼→ 同一套业务，维护 N 个代码仓库
支付宝小程序  ─┤
H5         ─┘

解决：Vue 语法 + 条件编译

  Vue 代码 ──→ uni-app 编译器 ──→ 微信小程序
            + 条件编译          ──→ App (iOS/Android)
            + 运行时适配        ──→ H5
                                ──→ 快应用
```

<Note title="一句话总结">

uni-app 让你用 Vue 语法写一套代码，通过条件编译和运行时适配输出到 12+ 个平台，核心是解决"全平台覆盖"的效率问题。

</Note>

## Q2: uni-app 核心架构

uni-app 的架构由三个核心机制组成：**条件编译、运行时框架、编译器。**

### 条件编译

条件编译（Conditional Compilation）是 uni-app 最核心的跨端机制。它的思想是：**在代码中用特定注释标记平台差异代码，编译器根据目标平台决定保留或删除。**

```text
┌─────────────────────────────────────────────────┐
│                   源代码                          │
│                                                   │
│  通用代码（所有平台都保留）                          │
│  #ifdef MP-WEIXIN                                │
│    微信专属代码                                    │
│  #endif                                          │
│  #ifdef APP-PLUS                                 │
│    App 专属代码                                   │
│  #endif                                          │
│  #ifdef H5                                       │
│    H5 专属代码                                    │
│  #endif                                          │
└─────────────────────────────────────────────────┘
         │
         ▼ 编译器根据目标平台裁剪
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ 微信小程序产物 │  │ App 产物      │  │ H5 产物      │
│ 通用代码      │  │ 通用代码      │  │ 通用代码      │
│ + 微信专属    │  │ + App 专属    │  │ + H5 专属    │
└──────────────┘  └──────────────┘  └──────────────┘
```

```javascript
// 条件编译的使用示例

// JS 中的条件编译
function login() {
  // #ifdef MP-WEIXIN
  uni.login({
    provider: 'weixin',
    success(res) {
      console.log('微信登录成功', res.code);
    }
  });
  // #endif

  // #ifdef MP-ALIPAY
  my.getAuthCode({
    scopes: 'auth_user',
    success(res) {
      console.log('支付宝登录成功', res.authCode);
    }
  });
  // #endif

  // #ifdef H5
  window.location.href = '/oauth/login';
  // #endif
}
```

```html
<!-- 模板中的条件编译 -->
<template>
  <view class="container">
    <!-- 所有平台都显示 -->
    <text>欢迎使用</text>

    <!-- #ifdef MP-WEIXIN -->
    <button open-type="getUserInfo">微信授权登录</button>
    <!-- #endif -->

    <!-- #ifdef APP-PLUS -->
    <button @click="appLogin">App 登录</button>
    <!-- #endif -->
  </view>
</template>
```

```css
/* 样式中的条件编译 */
.container {
  padding: 20rpx;
}

/* #ifdef MP-WEIXIN */
.container {
  padding-top: env(safe-area-inset-top); /* 微信状态栏适配 */
}
/* #endif */

/* #ifdef APP-PLUS */
.container {
  padding-top: var(--status-bar-height); /* App 状态栏适配 */
}
/* #endif */
```

<Warning title="核心机制: 条件编译的本质">

条件编译的本质是**编译时的代码裁剪**——编译器读取源代码中的 `#ifdef` / `#endif` 标记，根据目标平台保留或删除对应的代码块。

- 与运行时 `if (process.env.PLATFORM === 'weixin')` 不同，条件编译的未匹配代码**不会出现在编译产物中**——零运行时开销
- 条件编译可以用于 JS、CSS、HTML 模板、JSON 配置，甚至整个文件
- 条件编译的粒度可以精确到一行代码，也可以包裹整个组件

</Warning>

### 条件编译支持的平台

| 平台标识 | 平台 |
|---------|------|
| `MP-WEIXIN` | 微信小程序 |
| `MP-ALIPAY` | 支付宝小程序 |
| `MP-BAIDU` | 百度小程序 |
| `MP-TOUTIAO` | 字节小程序 |
| `MP-QQ` | QQ 小程序 |
| `MP-KUAISHOU` | 快手小程序 |
| `MP` | 所有小程序 |
| `APP-PLUS` | App（iOS/Android） |
| `H5` | H5（浏览器） |
| `APP-PLUS-NVUE` | App 的 nvue 页面（weex 渲染） |
| `WEB` | Web（H5 的别名） |

### 条件编译 vs 条件渲染

```text
条件编译（编译时）：
#ifdef MP-WEIXIN → 微信专属代码 → 编译产物中只保留微信版本
成本：零运行时开销

条件渲染（运行时）：
v-if="isWeixin" → 运行时判断平台 → 所有平台都包含代码
成本：运行时开销 + 包体积增大
```

### 小程序双线程模型

uni-app 在小程序端同样遵循双线程模型：

```text
┌─────────────────────────────────────────────────────────┐
│                    小程序逻辑层（JSCore）                  │
│  ┌────────────┐   ┌────────────┐   ┌────────────────┐  │
│  │  Vue 运行时 │ → │  虚拟 DOM  │ → │  setData()     │  │
│  │  响应式系统 │   │  diff 算法 │   │  更新视图层     │  │
│  └────────────┘   └────────────┘   └────────────────┘  │
└─────────────────────────────────────────────────────────┘
                          │ setData
                          ▼
┌─────────────────────────────────────────────────────────┐
│                    小程序视图层（WebView）                  │
│  ┌──────────────────────────────────────────────────┐  │
│  │  uni-app 内置组件（view, text, image...）        │  │
│  │  通过 template 渲染虚拟 DOM 树                    │  │
│  └──────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

uni-app 在小程序端的运行时基于 Vue 的响应式系统。当数据变化时，Vue 的响应式副作用（Reactive Effect）触发组件重新渲染，uni-app 的运行时把虚拟 DOM 的 diff 结果转换为 `setData()` 调用。

<Tip title="通俗比喻: 翻译公司的工作流程">

条件编译就像翻译公司接到一本书。翻译官（编译器）先通读全书，把中文通用段落（通用代码）翻译好，然后针对不同语种（平台），把夹杂的方言段落（`#ifdef` 标记的代码）替换为对应语种的版本。最终输出的是完全翻译好的成品书——读者不需要知道原文是什么语言。

</Tip>

### Vue 3 + Vite

uni-app 在 2023 年全面拥抱 Vue 3 + Vite：

```text
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  Vue 3 SFC   │ → │  Vite 构建   │ → │  各平台产物   │
│  <script     │   │  esbuild     │   │  小程序       │
│   setup>     │   │  Rollup      │   │  H5          │
│  Composition │   │  HMR         │   │  App         │
│  API         │   │              │   │              │
└──────────────┘   └──────────────┘   └──────────────┘
```

Vue 3 给 uni-app 带来的核心改进：
- **响应式系统升级**：基于 Proxy 的响应式比 Vue 2 的 `Object.defineProperty` 更高效，支持动态新增属性
- **Composition API**：逻辑复用更灵活（替代 mixins 的各种问题）
- **TypeScript 原生支持**：类型推导更准确
- **Tree Shaking**：打包体积更小

## Q3: uni-app vs Taro 对比

### 基础梳理

| 对比维度 | uni-app | Taro |
|---------|---------|------|
| **出品方** | DCloud | 京东凹凸实验室 |
| **核心框架** | Vue（主推）| React（主推）|
| **跨端机制** | 条件编译 + 运行时 | 运行时 |
| **跨端范围** | 12+ 平台（含 App 原生）| 微信/支付宝/百度/字节/H5/RN |
| **App 端方案** | weex 原生渲染 / uni-app x(uts) | React Native |
| **小程序性能** | 较好（条件编译产物更轻量）| 较差（React 运行时开销）|
| **开发者数量** | 600 万+ | 34k+ GitHub star |
| **插件生态** | 插件市场 8000+ | 社区组件库 |
| **IDE 支持** | HBuilderX（官方 IDE）| VS Code |
| **学习曲线** | 低（Vue 语法 + 条件编译）| 中（React + 理解运行时）|

### 框架选择：Vue vs React

```text
选 uni-app（Vue）的信号：
├── 团队技术栈是 Vue
├── 需要覆盖 App 端原生性能（uni-app x / uts）
├── 追求小程序端的极致渲染性能
├── 预算有限，需要快速出原型
└── 需要丰富的现成插件（uni-app 插件市场）

选 Taro（React）的信号：
├── 团队技术栈是 React
├── 需要输出 React Native App
├── 项目需要 React 生态组件（如 Ant Design）
├── 对 TypeScript 支持要求高
└── 需要 React 的高级特性（Hooks、Suspense、Concurrent）
```

### 性能对比深度剖析

很多候选人只知道"uni-app 小程序性能比 Taro 好"，但不理解**为什么好**。

**根本原因：跨端机制不同**

```text
uni-app 的条件编译：
源代码 → 编译器裁剪 → 各平台原生代码（轻量）
运行时只负责通用逻辑的适配
setData 粒度：更接近原生

Taro 的运行时模拟：
源代码 → 编译器语法转换 → 运行时（React）在 JSCore 中执行
运行时承载了 React 的全部逻辑
setData 粒度：虚拟 DOM diff 批量处理
```

| 性能指标 | uni-app | Taro 3.x |
|---------|---------|----------|
| 首屏渲染 | 较快（条件编译产物轻量）| 较慢（需初始化 React 运行时）|
| setData 效率 | 更精确的控制 | 批量 diff，可能冗余 |
| 包体积 | 较小 | 较大（包含 React 运行时）|
| 长列表滚动 | 良好 | 需要虚拟列表优化 |
| 内存占用 | 较低 | 较高 |

<Note title="一句话总结">

uni-app 的条件编译在编译阶段就解决了大部分平台差异，运行时只处理通用逻辑，所以小程序端性能更接近原生。Taro 的运行时方案把所有逻辑都放在运行时处理，开发体验好但性能有代价。

</Note>

## Q4: uni-app 的条件编译是什么？

条件编译是 uni-app 最核心、最高频的面试题。回答时必须说清楚：**是什么、为什么、怎么用、和运行时判断的区别。**

### 条件编译的本质

条件编译是一种**编译时的多态机制**。它让你在同一个源文件中编写多个平台的代码，编译器根据目标平台裁剪出对应的版本。

```javascript
// 条件编译的三种写法

// 1. 单行注释（最常用）
// #ifdef MP-WEIXIN
console.log('只在微信小程序中编译');
// #endif

// 2. 块注释（用于 HTML 模板）
<!-- #ifdef H5 -->
<div>只在 H5 中编译</div>
<!-- #endif -->

// 3. 文件级别的条件编译
// 创建文件：
// login.weixin.ts   → 只在微信小程序中编译
// login.h5.ts       → 只在 H5 中编译
// login.ts          → 通用实现（兜底）
```

### 条件编译 vs 运行时平台判断

| 维度 | 条件编译（#ifdef）| 运行时判断（if）|
|------|-----------------|----------------|
| 执行阶段 | 编译时 | 运行时 |
| 产物中是否包含 | 未匹配的代码被删除 | 所有平台代码都保留 |
| 包体积影响 | 不增加 | 增加 |
| 运行时性能 | 零开销 | 有判断开销 |
| 适用场景 | 平台 API 差异、组件差异 | 通用逻辑中的细微差异 |

```javascript
// ❌ 运行时判断：所有平台都包含代码，包体积增大
if (process.env.UNI_PLATFORM === 'mp-weixin') {
  uni.login({ provider: 'weixin' });
} else if (process.env.UNI_PLATFORM === 'mp-alipay') {
  my.getAuthCode({ scopes: 'auth_user' });
}
// 编译产物中两种代码都存在，不管目标平台是什么

// ✅ 条件编译：只保留目标平台的代码
// #ifdef MP-WEIXIN
uni.login({ provider: 'weixin' });
// #endif
// #ifdef MP-ALIPAY
my.getAuthCode({ scopes: 'auth_user' });
// #endif
// 编译到微信时，支付宝代码完全不存在
```

### 条件编译的使用场景

```text
适用条件编译的场景：
├── 调用平台专属 API（如 wx.login vs my.getAuthCode）
├── 使用平台专属组件（如微信的 open-type 按钮）
├── 样式适配（如状态栏高度、安全区域）
├── 页面配置差异（如微信的 onShareAppMessage）
└── 整个文件级别的平台差异

不适用条件编译的场景：
├── 通用业务逻辑（所有平台一样）
├── 组件封装（应该统一接口）
└── 数据请求（应该统一 service 层）
```

<Tip title="通俗比喻: 瑞士军刀">

条件编译就像瑞士军刀——刀、剪刀、开瓶器都装在一个壳子里，但你去登山时只带刀，去野餐时只带开瓶器。编译器就是帮你"拆卸"的工匠——根据你的目的地（目标平台），把不需要的工具卸下来，最终给你一把轻便的专用工具。

</Tip>

## Q5: uni-app 的页面生命周期

uni-app 的页面生命周期是面试中的高频考点，因为它融合了 Vue 生命周期和小程序生命周期。

### 生命周期执行顺序

```text
┌─────────────────────────────────────────────────────────┐
│                    页面首次加载                           │
├─────────────────────────────────────────────────────────┤
│  onLoad     → 页面加载（接收路由参数）                     │
│  onShow     → 页面显示（从后台切到前台也会触发）            │
│  onReady    → 页面初次渲染完成                            │
│  Vue mounted → Vue 组件挂载完成                           │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    页面后台切回                            │
├─────────────────────────────────────────────────────────┤
│  onShow     → 页面重新显示                               │
│  （不会触发 onLoad 和 onReady）                            │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    页面卸载                               │
├─────────────────────────────────────────────────────────┤
│  onHide     → 页面隐藏（跳转到其他页面时触发）              │
│  onUnload   → 页面卸载（redirectTo 或 navigateBack 触发）  │
│  Vue unmounted → Vue 组件卸载                            │
└─────────────────────────────────────────────────────────┘
```

```javascript
// Vue 3 + Composition API 写法
import { onLoad, onShow, onReady, onHide, onUnload } from '@dcloudio/uni-app';

export default {
  setup() {
    // 页面加载（只触发一次）
    onLoad((options) => {
      console.log('页面参数:', options.id);
    });

    // 页面显示（每次从后台切回都触发）
    onShow(() => {
      refreshData();
    });

    // 页面初次渲染完成
    onReady(() => {
      console.log('页面渲染完成');
    });

    // 页面隐藏
    onHide(() => {
      saveDraft();
    });

    // 页面卸载
    onUnload(() => {
      cleanup();
    });
  }
}
```

### onLoad vs onShow 的区别

| 生命周期 | 触发时机 | 触发次数 | 典型用途 |
|---------|---------|---------|---------|
| `onLoad` | 页面首次加载 | 1 次 | 初始化数据、解析路由参数 |
| `onShow` | 页面每次显示 | 多次 | 刷新数据、恢复状态 |
| `onReady` | 首次渲染完成 | 1 次 | 操作 DOM、初始化地图/Canvas |

<Warning title="核心机制: onLoad 只触发一次">

`onLoad` 在页面的生命周期中只触发一次。当用户从页面 A 跳转到页面 B，再返回页面 A 时，不会再次触发 `onLoad`——只会触发 `onShow`。所以如果需要每次进入页面都执行的逻辑（如刷新列表），应该放在 `onShow` 中，而不是 `onLoad`。

</Warning>

## Q7: uni-app 的网络请求封装

```javascript
// utils/request.js
const BASE_URL = 'https://api.example.com';

const request = (options) => {
  return new Promise((resolve, reject) => {
    // 添加 token
    const token = uni.getStorageSync('token');
    const header = {
      'Content-Type': 'application/json',
      ...options.header
    };
    if (token) {
      header['Authorization'] = `Bearer ${token}`;
    }

    uni.request({
      url: BASE_URL + options.url,
      method: options.method || 'GET',
      data: options.data,
      header,
      success(res) {
        if (res.statusCode === 200) {
          resolve(res.data);
        } else if (res.statusCode === 401) {
          // token 过期，跳转登录页
          uni.navigateTo({ url: '/pages/login/login' });
          reject(new Error('未授权'));
        } else {
          reject(new Error(res.data.message || '请求失败'));
        }
      },
      fail(err) {
        uni.showToast({ title: '网络异常', icon: 'none' });
        reject(err);
      }
    });
  });
};

// 封装常用方法
export const get = (url, data) => request({ url, method: 'GET', data });
export const post = (url, data) => request({ url, method: 'POST', data });
export const put = (url, data) => request({ url, method: 'PUT', data });
export const del = (url, data) => request({ url, method: 'DELETE', data });
```

<Note title="一句话总结">

uni-app 的网络请求封装核心是统一拦截器（token 注入、错误处理、登录过期跳转），底层通过 `uni.request` 实现跨平台兼容。

</Note>

    ```javascript
    // #endif
    ```

    ```text
    传统方案：
    JS 代码 → JS 引擎 → Bridge（序列化）→ 原生代码
    成本：Bridge 通信开销

    成本：零 Bridge 开销
    ```

    - App 端性能达到原生水平
    - 直接调用原生 API，不需要 Bridge
    - 开发者用 TypeScript 语法，不需要学 Java/Swift
    - 类型安全，编译时检查

    **适用场景：**
    uni-app x + uts 适合对 App 端性能要求极高的场景。对于小程序和 H5 端，传统的 uni-app（Vue + 条件编译）仍然是主流方案。"

面试官评价：**知道用法，但不理解本质和设计动机。**

**满分回答（90-100分）：**

> 条件编译（Conditional Compilation）是 uni-app 的核心跨端机制，本质是**编译时的代码裁剪**。在源代码中用 `#ifdef`、`#ifndef`、`#endif` 标记平台差异代码块，编译器在构建阶段根据目标平台（如 `MP-WEIXIN`、`H5`、`APP-PLUS`）决定保留或删除对应的代码。

> 与运行时的 `if (process.env.PLATFORM === 'weixin')` 不同，条件编译的未匹配代码**不会出现在编译产物中**——这意味着零运行时开销和零包体积增加。条件编译支持 JS、CSS、HTML 模板、JSON 配置，甚至整个文件级别（如 `login.weixin.ts` 只在微信编译）。

> 条件编译的设计动机是：跨端框架面临一个根本矛盾——**通用性和平台差异性的平衡**。纯运行时方案（如 Taro 3.x）把所有逻辑都放在运行时处理，开发体验好但性能有代价；纯条件编译方案让平台差异代码在编译阶段就解决了，性能接近原生，但需要开发者手动维护多平台代码。uni-app 选择了折中路线：通用逻辑走运行时（Vue 响应式 + 虚拟 DOM），平台差异走条件编译。

> 实际项目建议：平台 API 差异（登录、支付、分享）用条件编译，通用业务逻辑（列表渲染、表单处理）用 Vue 组件，两者结合是最佳实践。
