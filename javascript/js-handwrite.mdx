### call 的实现图解

call 的核心原理只有一句话：**把函数变成目标对象的临时方法，然后通过 `obj.fn()` 的方式调用**。

```text
myCall(obj, a, b) 的执行流程：

  第 1 步：创建临时属性          第 2 步：通过对象调用          第 3 步：清理
  ┌──────────────┐              ┌──────────────┐              ┌──────────────┐
  │ obj[fn] = this│  ──────→    │ obj[fn](a,b) │  ──────→    │ delete obj[fn]│
  │              │              │ this = obj ✅ │              │ 恢复原状      │
  └──────────────┘              └──────────────┘              └──────────────┘

  关键洞察：obj.fn() 调用时，this 自然指向 obj（隐式绑定规则）
  用 Symbol 避免和 obj 上已有的属性名冲突
```

```javascript
/**
 * 手写 call：改变函数的 this 指向并立即执行
 *
 * 核心原理图解：
 *   1. 把函数"伪装"成目标对象的一个临时方法  →  ctx[fn] = this
 *   2. 通过对象方法的形式调用函数            →  ctx[fn](...args)
 *      （利用"obj.fn() 时 this 指向 obj"的隐式绑定规则）
 *   3. 调用完删除临时方法，不污染目标对象     →  delete ctx[fn]
 *
 * 趣味比喻：
 *   call 就像"借壳上市"——你想用 obj 的身份（this）来执行一个函数，
 *   临时把函数挂到 obj 的门牌下（ctx[fn] = this），
 *   调用完赶紧把门牌摘掉（delete ctx[fn]），假装什么都没发生过。
 *
 * @param {*} ctx - this 指向的目标对象，null/undefined 时指向全局对象
 * @param  {...any} args - 传给函数的参数列表
 * @returns {*} 函数的返回值
 */
Function.prototype.myCall = function(ctx, ...args) {
  // Step 1: 处理边界情况
  // ctx 为 null/undefined 时指向全局对象（ES 规范行为）
  // Object(ctx) 将基本类型包装为对象（如 1 → Number(1), 'str' → String('str)')
  ctx = ctx == null ? globalThis : Object(ctx);

  // Step 2: 把当前函数挂在目标对象上
  // 用 Symbol() 作为临时属性名——Symbol 值是唯一的，绝不会和已有属性冲突
  // 如果用字符串 'tempFn'，万一 obj 上本来就有这个属性就炸了
  const fn = Symbol();
  ctx[fn] = this;                   // this 就是调用 myCall 的那个函数

  // Step 3: 通过对象方法调用，this 自然指向 ctx（隐式绑定规则生效）
  // 这一步是整个 call 的灵魂——利用 JS 的 this 绑定规则
  const result = ctx[fn](...args);

  // Step 4: 用完即弃——删除临时属性，不污染目标对象
  delete ctx[fn];

  // Step 5: 返回原函数的执行结果
  return result;
};
```

### apply 的实现图解

apply 和 call 的唯一区别：**参数是数组**。

```text
call(obj, 1, 2, 3)     →  参数平铺传入
apply(obj, [1, 2, 3])  →  参数数组传入，内部展开后效果相同

apply 的实现 = 调用 call + 展开数组
```

```javascript
/**
 * 手写 apply：和 call 一样，只是参数以数组形式传入
 *
 * 一句话理解：apply = call + 数组展开
 * call 接收参数列表 (a, b, c)，apply 接收参数数组 [a, b, c]
 * 内部只需要把数组展开成参数列表，然后交给 call 处理
 *
 * @param {*} ctx - this 指向的目标对象
 * @param {Array} args - 参数数组
 * @returns {*} 函数的返回值
 */
Function.prototype.myApply = function(ctx, args = []) {
  // 直接复用 myCall，用展开运算符 ...args 将数组转为参数列表
  // myCall(ctx, ...[1,2,3])  →  myCall(ctx, 1, 2, 3)
  return this.myCall(ctx, ...args);
};
```

### bind 的实现图解

bind 是三者中最复杂的——它返回一个新函数，而且要处理 `new` 的优先级问题。

```text
bind 的核心难点：new 的优先级高于 bind

场景 1：普通调用 → this 指向绑定的 obj
  const boundFn = fn.bind(obj)
  boundFn(a, b)     →  this = obj（绑定生效）

场景 2：new 调用 → this 指向新实例，绑定的 obj 被忽略！
  const boundFn = fn.bind(obj)
  new boundFn(a, b) →  this = 新实例 {}（new 覆盖了 bind）

为什么？ES 规范要求 [[Construct]] 内部方法会强制覆盖 this 上下文
```

```javascript
/**
 * 手写 bind：返回一个绑定了 this 和部分参数的新函数（不立即执行）
 *
 * 核心原理图解：
 *   fn.bind(obj, presetA)  返回  boundFn
 *   boundFn(callB)  →  fn.call(obj, presetA, callB)    ← 普通调用
 *   new boundFn(callB)  →  new fn(presetA, callB)       ← new 调用，忽略 obj
 *
 * 趣味比喻：
 *   bind 就像用胶带把麦克风粘在某人手上（this 被锁定）。
 *   但 new 关键字是个"拆弹专家"——它能强行把胶带撕开，
 *   把麦克风抢过来给新创建的实例。
 *
 * @param {*} ctx - this 指向的目标对象
 * @param  {...any} bindArgs - 预设的参数（柯里化，提前固定部分参数）
 * @returns {Function} 绑定后的新函数
 */
Function.prototype.myBind = function(ctx, ...bindArgs) {
  const fn = this;                    // 保存原始函数引用（闭包捕获）

  // 返回一个新函数（闭包）
  // 这个新函数"记住"了 fn 和 ctx（闭包的魔法）
  return function boundFn(...callArgs) {
    // 关键检测：是否通过 new 调用？
    // new.target 是 ES6 引入的——如果函数是 new 调用的，new.target 指向构造函数
    if (new.target) {
      // new 调用：忽略绑定的 ctx，用原函数构造新实例
      // 合并预设参数和调用参数：bindArgs 是 bind 时传的，callArgs 是 new 时传的
      return new fn(...bindArgs, ...callArgs);
    }
    // 普通调用：合并预设参数和调用参数，用 call 绑定 this
    return fn.call(ctx, ...bindArgs, ...callArgs);
  };
};
```

<Note title="💡 一句话总结">

call/apply/bind 的本质是利用"对象方法调用"的 this 规则，bind 额外需要处理 new 的优先级。call 立即执行，apply 立即执行（参数是数组），bind 返回新函数（延迟执行）。

</Note>

### call/apply/bind 三者对比速查表

| 方法 | 是否立即执行 | 参数形式 | 返回值 | 是否能被 new 覆盖 |
|------|------------|---------|--------|-----------------|
| call | 是 | 参数列表 `(a, b, c)` | 函数执行结果 | 不适用 |
| apply | 是 | 参数数组 `[a, b, c]` | 函数执行结果 | 不适用 |
| bind | 否（返回新函数） | 支持柯里化 | 新函数 | 是（new 优先级 > bind） |

<Note title="💡 一句话总结">

call/apply/bind 的本质是利用"对象方法调用"的 this 规则，bind 额外需要处理 new 的优先级。

</Note>

## Q2: 手写深拷贝

### 深拷贝的核心实现

```javascript
/**
 * 手写深拷贝：递归复制对象的所有层级，处理循环引用和特殊对象
 * @param {*} source - 要拷贝的源对象
 * @param {WeakMap} map - 用于记录已拷贝对象，解决循环引用问题
 * @returns {*} 深拷贝后的新对象
 */
function deepClone(source, map = new WeakMap()) {
  // 基本类型（null、number、string、boolean、undefined、symbol）直接返回
  if (source === null || typeof source !== 'object') return source;
  // 循环引用检测：如果已经拷贝过这个对象，直接返回之前的拷贝结果
  // 这是防止无限递归的关键——遇到 { a: obj } 这种自引用不会死循环
  if (map.has(source)) return map.get(source);

  // 处理特殊内置对象——每种都需要独立构造，不能用通用方式拷贝
  if (source instanceof Date) return new Date(source);       // Date 需要 new 构造
  if (source instanceof RegExp) return new RegExp(source);   // RegExp 需要 new 构造
  if (source instanceof Set) {
    const clone = new Set();
    map.set(source, clone);                 // 先注册再递归，防止 Set 内部值循环引用
    source.forEach(v => clone.add(deepClone(v, map))); // 递归拷贝 Set 中的每个值
    return clone;
  }
  if (source instanceof Map) {
    const clone = new Map();
    map.set(source, clone);                 // 先注册再递归
    source.forEach((v, k) => clone.set(deepClone(k, map), deepClone(v, map))); // key 和 value 都要递归
    return clone;
  }

  // 处理普通对象和数组
  // Object.create(Object.getPrototypeOf(source)) 保留原型链，避免拷贝后丢失自定义原型
  const clone = Array.isArray(source) ? [] : Object.create(Object.getPrototypeOf(source));
  map.set(source, clone);                   // 先注册再递归——这是处理循环引用的核心：先占位

  // 收集所有属性键（包括 Symbol 属性，Object.keys 不包含 Symbol）
  const keys = [...Object.keys(source), ...Object.getOwnPropertySymbols(source)];
  for (const key of keys) {
    clone[key] = deepClone(source[key], map); // 递归拷贝每个属性值
  }
  return clone;
}
```

<Warning title="⚠️ 关键细节: 为什么用 WeakMap?">

WeakMap 的 key 是弱引用，当源对象被垃圾回收时，WeakMap 中的引用也会被释放，不会造成内存泄漏。Map 是强引用，会阻止垃圾回收。


</Warning>

### 循环引用的处理策略

循环引用是深拷贝最核心的难点。用 WeakMap 记录已拷贝的对象，遇到重复引用直接返回已拷贝的版本。

```javascript
// 循环引用示例
const obj = { name: 'Tom' };
obj.self = obj; // obj.self 指向 obj 自己

// 如果不用 WeakMap，会无限递归导致栈溢出
// 用 WeakMap 后：第一次拷贝 obj 时记录，第二次遇到 obj 直接返回已拷贝版本
```

### structuredClone vs 手写深拷贝

浏览器原生提供了 `structuredClone` API，原生支持循环引用，但不支持函数、DOM 节点、Symbol 属性。

```javascript
// structuredClone 原生支持循环引用
const obj = { name: 'Tom' };
obj.self = obj;
const clone = structuredClone(obj); // ✅ 正常工作

// ❌ structuredClone 不支持函数
structuredClone({ fn: () => {} }); // DataCloneError

// ❌ structuredClone 不支持 Symbol
structuredClone({ [Symbol('id')]: 1 }); // Symbol 属性被忽略
```

## Q3: 手写防抖（debounce）和节流（throttle）

### 防抖的核心实现

```javascript
/**
 * 手写防抖（debounce）：事件触发后延迟执行，延迟期间再次触发则重新计时
 * 应用场景：搜索框输入、窗口 resize、表单验证
 * @param {Function} fn - 要防抖的函数
 * @param {number} delay - 延迟时间（毫秒）
 * @param {boolean} immediate - 是否在首次触发时立即执行（leading 模式）
 * @returns {Function} 防抖后的函数
 */
function debounce(fn, delay, immediate = false) {
  let timer = null;                       // 定时器引用，用于清除和重置
  return function(...args) {
    clearTimeout(timer);                  // 每次触发都清除上一次的定时器（重新计时）
    // immediate 模式：首次触发立即执行，之后等冷静期
    if (immediate && !timer) fn.apply(this, args);
    timer = setTimeout(() => {
      // 非 immediate 模式：冷静期结束后执行
      // immediate 模式：冷静期结束后重置 timer，允许下次立即执行
      if (!immediate) fn.apply(this, args);
      timer = null;                       // 重置 timer，让 immediate 模式下次能立即触发
    }, delay);
  };
}

/**
 * 手写节流（throttle）：在一段时间内只执行一次，控制函数执行频率
 * 应用场景：滚动事件、鼠标移动、高频点击
 * @param {Function} fn - 要节流的函数
 * @param {number} interval - 节流间隔（毫秒）
 * @param {Object} options - 配置项：leading（开头执行）、trailing（结尾执行）
 * @returns {Function} 节流后的函数
 */
function throttle(fn, interval, options = { leading: true, trailing: true }) {
  let lastTime = 0;                       // 上次执行的时间戳
  let timer = null;                       // trailing 模式的定时器引用
  return function(...args) {
    const now = Date.now();
    // 如果 leading=false 且是首次调用（lastTime=0），将 lastTime 设为当前时间
    // 这样首次调用不会立即执行，而是等到 interval 后才执行
    if (!options.leading && !lastTime) lastTime = now;
    const remaining = interval - (now - lastTime); // 距离下次可执行的剩余时间
    if (remaining <= 0) {
      // 时间到了，可以执行
      if (timer) { clearTimeout(timer); timer = null; } // 清除 trailing 定时器（避免重复执行）
      lastTime = now;                     // 更新上次执行时间
      fn.apply(this, args);               // 执行函数
    } else if (!timer && options.trailing) {
      // 时间没到，但开启了 trailing 模式：设置定时器在剩余时间后执行
      timer = setTimeout(() => {
        lastTime = options.leading ? Date.now() : 0; // leading 时更新时间戳，否则重置为 0
        timer = null;                     // 清除定时器引用
        fn.apply(this, args);             // 执行函数
      }, remaining);
    }
  };
}
```

### leading 和 trailing 的区别

防抖和节流都支持 `leading`（首次立即执行）和 `trailing`（最后一次延迟执行）选项。

```javascript
// 防抖的 immediate 参数 = leading 模式
// immediate=true: 首次触发立即执行，之后等冷静期
// immediate=false: 等冷静期结束后执行（默认）

// 节流的 leading + trailing 组合
// leading=true, trailing=true: 首尾各执行一次（默认）
// leading=true, trailing=false: 只在开头执行一次
// leading=false, trailing=true: 只在结尾执行一次
```

### 防抖和节流的实际应用场景

```javascript
// ✅ 防抖适合：搜索框输入、窗口 resize、表单验证
const search = debounce(fetchResults, 300);
searchInput.addEventListener('input', (e) => search(e.target.value));

// ✅ 节流适合：滚动事件、鼠标移动、高频点击
const onScroll = throttle(updateProgress, 100);
window.addEventListener('scroll', onScroll);

// lodash 的 throttle 实际上是 debounce 的封装
// throttle(fn, wait) ≈ debounce(fn, wait, { maxWait: wait })
```

## Q4: 手写柯里化（curry）

### 柯里化的核心实现

```javascript
/**
 * 手写柯里化（curry）：将多参数函数转为可逐个传参的链式调用
 * 核心原理：如果参数数量够了就执行，不够就返回新函数继续接收参数
 * @param {Function} fn - 要柯里化的原函数
 * @returns {Function} 柯里化后的函数
 */
function curry(fn) {
  return function curried(...args) {
    // 参数数量达到原函数的形参数数时，直接执行
    // fn.length 是原函数定义时的形参个数
    if (args.length >= fn.length) {
      return fn.apply(this, args);
    }
    // 参数不够：返回新函数，继续接收参数，并将已收集的参数合并
    return function(...moreArgs) {
      return curried.apply(this, [...args, ...moreArgs]);
    };
  };
}

const add = (a, b, c) => a + b + c;
const curriedAdd = curry(add);            // 柯里化
curriedAdd(1)(2)(3);                      // 6：每次传一个参数
curriedAdd(1, 2)(3);                      // 6：前两个一起传
curriedAdd(1)(2, 3);                      // 6：后两个一起传
```

### 柯里化的实际应用

```javascript
// 日志函数：固定 logger 和 level，只变化 message
const log = curry((logger, level, message) => {
  logger[level](`[${new Date().toISOString()}] ${message}`);
});
const appLog = log(console);    // 固定 logger
const error = appLog('error');  // 固定 level
error('服务器连接失败');         // 只传 message

// 事件绑定：固定 handler，变化 event 和 target
const bindEvent = curry((handler, event, target) => {
  target.addEventListener(event, handler);
});
```

### 柯里化与偏函数（Partial Application）的区别

柯里化是将多参数函数转为一系列单参数函数，偏函数是固定部分参数。两者常被混淆。

```javascript
/**
 * 柯里化 vs 偏函数的区别
 * 柯里化：每次只接收一个参数，返回新函数直到参数够
 * 偏函数：一次可以固定多个参数，返回接收剩余参数的函数
 */

// 柯里化：每次只接收一个参数
curry(add)(1)(2)(3);

// 偏函数：一次可以固定多个参数
function partial(fn, ...presetArgs) {
  return (...laterArgs) => fn(...presetArgs, ...laterArgs);
}
const add1 = partial(add, 1);
add1(2, 3); // 6
```

## Q5: 手写 EventEmitter

### EventEmitter 的核心实现

```javascript
/**
 * 手写 EventEmitter：发布订阅模式的核心实现
 * 核心原理：维护一个事件名 → 回调数组的映射，on 注册、off 移除、emit 触发
 */
class EventEmitter {
  constructor() {
    // 用 Map 存储事件映射：key 是事件名，value 是回调函数数组
    this.events = new Map();
  }

  /**
   * 注册事件监听器
   * @param {string} event - 事件名
   * @param {Function} callback - 回调函数
   * @returns {this} 支持链式调用
   */
  on(event, callback) {
    if (!this.events.has(event)) this.events.set(event, []); // 首次注册，初始化数组
    this.events.get(event).push(callback);                   // 将回调加入监听列表
    return this;                                              // 返回 this 支持链式调用
  }

  /**
   * 移除事件监听器
   * @param {string} event - 事件名
   * @param {Function} callback - 要移除的回调函数（可选，不传则移除该事件的所有回调）
   * @returns {this}
   */
  off(event, callback) {
    if (!this.events.has(event)) return this;                 // 事件不存在，直接返回
    if (!callback) { this.events.delete(event); }            // 不传回调，删除整个事件
    else {
      // 过滤掉匹配的回调（包括原始回调和 once 包装的 wrapper）
      // cb._original 是 once 方法中 wrapper 指向的原始回调
      const callbacks = this.events.get(event).filter(
        cb => cb !== callback && cb._original !== callback
      );
      this.events.set(event, callbacks);
    }
    return this;
  }

  /**
   * 触发事件：执行该事件名下注册的所有回调
   * @param {string} event - 事件名
   * @param  {...any} args - 传给回调的参数
   * @returns {this}
   */
  emit(event, ...args) {
    if (!this.events.has(event)) return this;                 // 事件不存在，直接返回
    this.events.get(event).forEach(cb => cb.apply(this, args)); // 依次执行所有回调
    return this;
  }

  /**
   * 注册一次性监听器：触发一次后自动移除
   * 实现原理：用 wrapper 函数包装原始回调，wrapper 执行后调用 off 移除自己
   * @param {string} event - 事件名
   * @param {Function} callback - 回调函数
   * @returns {this}
   */
  once(event, callback) {
    // 创建包装函数：执行原始回调后，立即从事件列表中移除自己
    const wrapper = (...args) => {
      callback.apply(this, args);         // 先执行原始回调
      this.off(event, wrapper);           // 再移除 wrapper 自身（只执行一次的关键）
    };
    wrapper._original = callback;         // 保存原始回调的引用，用于 off 时识别
    this.on(event, wrapper);              // 注册 wrapper 而非原始回调
    return this;
  }
}
```

### once 的实现细节

`once` 方法的关键是包装一个 wrapper 函数，wrapper 内部在执行原始回调后立即解绑自己。`_original` 属性用于 `off` 方法识别原始回调。

```javascript
// once 的执行流程
emitter.once('click', handler);
// 1. 创建 wrapper，wrapper._original = handler
// 2. wrapper 通过 on 注册
// 3. emit('click') 时执行 wrapper → wrapper 执行 handler → wrapper 调用 off 解绑自己
// 4. 第二次 emit('click') 时，wrapper 已被移除，handler 不再执行
```

### 发布订阅 vs 观察者模式

EventEmitter 是发布订阅模式的实现，和观察者模式有本质区别。

```javascript

// Subject 维护一个 Observer 列表，状态变化时直接调用 observer.update()
```

## Q6: 手写 async/await 的本质

### Generator + 自动执行器的核心实现

```javascript
/**
 * async/await 的本质：Generator + 自动执行器的语法糖
 * Generator 负责暂停和恢复，自动执行器负责用 Promise 链驱动 Generator
 */

// Generator 函数：用 yield 暂停执行，等待外部传入值后恢复
function* fetchData() {
  const user = yield fetch('/api/user');  // 暂停，等待 fetch('/api/user') 的结果
  const orders = yield fetch(`/api/orders/${user.id}`); // 暂停，等待上一步的结果
  return orders;                          // 最终返回值
}

/**
 * 自动执行器：递归驱动 Generator 执行
 * 核心逻辑：执行 gen.next() → 拿到 yield 后面的 Promise → resolve 后再执行 gen.next(value)
 * 这就是 async/await 在 V8 内部的实现原理
 * @param {GeneratorFunction} generator - Generator 构造函数
 * @returns {Promise} 最终结果
 */
function run(generator) {
  const gen = generator();                // 创建 Generator 实例
  function step(value) {
    const result = gen.next(value);       // 将上一步的值传入，恢复 Generator 执行
    if (result.done) return result.value; // Generator 执行完毕，返回最终值
    // result.value 是 yield 后面的 Promise
    // 等 Promise resolve 后，将结果传入下一步 step
    return result.value.then(val => step(val));
  }
  return step();                          // 启动执行（首次不传值）
}

run(fetchData); // 等价于 async function fetchData() { ... }
```

### Generator 的暂停与恢复机制

Generator 被 V8 编译为状态机。每次 `yield` 暂停时，V8 保存当前的执行状态（局部变量、程序计数器）到 Generator 对象的 context 中。`next()` 恢复时，从 context 中恢复状态继续执行。

```javascript
/**
 * async/await 的本质转换
 * await 右边的表达式立刻同步执行
 * await 下面的代码被打包进隐形的 Promise.then()，变成微任务
 */
async function foo() {
  console.log('1');                       // 同步执行
  await bar();                            // bar() 立刻同步执行
  console.log('2');                       // 这行被打包进微任务，不在当前同步栈中执行
}

// 等价的 Promise 写法——证明 await 下面的代码是微任务
function foo() {
  console.log('1');                       // 同步执行
  bar().then(() => {
    console.log('2');                     // 微任务：在当前同步代码全部执行完后才运行
  });
}
```

### 错误处理：try/catch 与 Generator.throw

```javascript
/**
 * Generator 的错误处理：通过 gen.throw() 将错误注入 Generator 内部
 * 这就是 async/await 中 await 的错误能被 try/catch 捕获的原因
 */
function* gen() {
  try {
    const data = yield fetch('/api');     // 如果 fetch 失败，错误会通过 throw 注入到这里
  } catch (err) {
    console.error('请求失败:', err);       // 捕获注入的错误
  }
}

/**
 * 支持错误处理的自动执行器
 * 关键：.catch(err => gen.throw(err)) 将 Promise 的 rejection 注入 Generator
 * 这样 Generator 内部的 try/catch 就能捕获异步错误
 * @param {GeneratorFunction} generator - Generator 构造函数
 * @returns {Promise}
 */
function run(generator) {
  const gen = generator();
  function step(value) {
    let result;
    try {
      result = gen.next(value);           // 恢复 Generator 执行
    } catch (err) {
      return Promise.reject(err);         // Generator 内部同步抛错，直接 reject
    }
    if (result.done) return Promise.resolve(result.value); // 执行完毕，返回结果
    // 继续驱动：Promise resolve 后传入下一步，rejection 通过 throw 注入 Generator
    return Promise.resolve(result.value)
      .then(val => step(val))             // 成功：将值传入 gen.next()
      .catch(err => gen.throw(err));      // 失败：将错误注入 Generator，触发内部的 catch
  }
  return step();
}
```

## 🔍 底层源码：V8 的 bind 实现

<div class="follow-up">
V8 的 bind 实现和手写版本有什么区别？
</div>

V8 的 `Function.prototype.bind` 实现（`src/builtins/function-bind.tq`）比手写版本复杂得多。它创建一个 `BoundFunction` 对象，内部有三个关键字段（详见 [V8 源码：src/builtins/function-bind.tq](https://github.com/v8/v8/blob/main/src/builtins/function-bind.tq)）：

- `[[BoundTargetFunction]]`：原始函数
- `[[BoundThis]]`：绑定的 this
- `[[BoundArguments]]`：预设的参数

当 BoundFunction 被 `new` 调用时，V8 的 `[[Construct]]` 内部方法会检查 `new.target`，如果有就用 `[[BoundTargetFunction]]` 的构造逻辑，`[[BoundThis]]` 被忽略。这就是为什么 `new (fn.bind(ctx))()` 中 this 指向新实例而不是 ctx。

    **答题要点：**
    用 Map 的有序性实现 O(1) 的 get/put。

    **高分回答：**
    "LRU（Least Recently Used）缓存的核心是：访问时把元素移到最前面，容量满时删除最久未使用的（最后面的）。
    ```javascript
    /**
     * 手写 LRU 缓存（Least Recently Used）
     * 核心原理：利用 Map 的插入顺序特性，访问时移到末尾，容量满时删除头部
     * get/put 时间复杂度 O(1)
     */
    class LRUCache {
      /**
       * @param {number} capacity - 缓存容量
       */
      constructor(capacity) {
        this.capacity = capacity;          // 最大容量
        this.cache = new Map();            // Map 的迭代顺序就是插入顺序
      }
      /**
       * 获取缓存值：命中后需要移到最前面（标记为最近使用）
       * @param {*} key
       * @returns {*} 缓存值，不存在返回 -1
       */
      get(key) {
        if (!this.cache.has(key)) return -1; // 未命中
        const val = this.cache.get(key);
        this.cache.delete(key);            // 先删除
        this.cache.set(key, val);          // 再重新插入——移到 Map 末尾（最近使用）
        return val;
      }
      /**
       * 写入缓存：如果已存在则更新，容量满时淘汰最久未使用的
       * @param {*} key
       * @param {*} value
       */
      put(key, value) {
        if (this.cache.has(key)) this.cache.delete(key); // 已存在则先删除（更新位置）
        this.cache.set(key, value);        // 插入到 Map 末尾
        if (this.cache.size > this.capacity) {
          // Map 迭代器的第一个就是最久未使用的（最早插入且未被访问过）
          this.cache.delete(this.cache.keys().next().value);
        }
      }
    }
    ```
    为什么用 Map？因为 Map 的迭代顺序是插入顺序，`keys().next().value` 就是最久未使用的元素。"
