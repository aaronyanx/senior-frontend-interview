## Q1: 鸿蒙的历史？为什么要有它？

```text
┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  Android     │ → │  华为 EMUI   │ → │  鸿蒙 1.0    │ → │  鸿蒙 NEXT   │
│  (2008)      │   │  (2012)      │   │  (2019)      │   │  (2024)      │
│  AOSP 基础   │   │  深度定制    │   │  分布式架构  │   │  纯血鸿蒙    │
│              │   │              │   │  IoT 优先    │   │  去 Android  │
└──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
```

<Steps>

  <Step title="2012-2018 年: 华为的技术积累">

    华为从 2012 年开始研发自有操作系统，内部代号"鸿蒙"。最初的目标是为 IoT 设备（智能家居、穿戴设备）打造一个轻量级、分布式的操作系统。2018 年美国制裁事件加速了鸿蒙的落地进程——华为需要一个不依赖 Google GMS（Google Mobile Services）的替代方案。

  </Step>
  <Step title="2019 年: 鸿蒙 1.0 发布">

    2019 年华为开发者大会上，鸿蒙 1.0 正式发布。首搭荣耀智慧屏。核心理念是**分布式架构**——多个设备可以组成一个"超级终端"，共享硬件能力（如手机的摄像头、手表的传感器、平板的屏幕）。但鸿蒙 1.0 的应用层仍然基于 AOSP（Android 开源项目），本质上是"Android + 分布式扩展"。

  </Step>
  <Step title="2020-2022 年: 鸿蒙 2.0/3.0 手机端落地">

    鸿蒙 2.0 开始在手机上大规模推送。华为引入了 HarmonyOS Connect（鸿蒙智联）生态，第三方 IoT 设备可以接入鸿蒙分布式能力。应用开发框架从 Java/JS 逐步转向 ArkTS。但底层仍然兼容 Android APK，引发"鸿蒙是否是 Android 套壳"的争议。

  </Step>
  <Step title="2023-2024 年: 鸿蒙 NEXT（纯血鸿蒙）">

    鸿蒙 NEXT（HarmonyOS NEXT）是鸿蒙的里程碑版本——**彻底去除 AOSP 代码，不再兼容 Android APK**。应用必须使用 ArkTS + ArkUI 开发，运行在自研的方舟运行时上。这意味着鸿蒙从"兼容 Android"转变为"独立生态"。2024 年第四季度，鸿蒙 NEXT 正式商用。

  </Step>
  <Step title="2025 年: 鸿蒙生态加速">

    鸿蒙 NEXT 商用后，国内主流 App（微信、支付宝、抖音、美团等）陆续推出鸿蒙原生版本。鸿蒙开发者数量快速增长，ArkTS/ArkUI 成为前端开发者需要关注的新技术栈。

  </Step>

</Steps>

**鸿蒙解决的核心问题：**

```text
问题：Android 受制于 Google，IoT 设备生态碎片化

Android → Google 控制 GMS → 华为手机无法使用 Google 服务
IoT 设备 → 各厂商协议不互通 → 智能家居是"设备孤岛"
多设备协同 → 手机、平板、手表各自独立 → 数据和能力不互通

解决：分布式架构 + 自研应用框架

鸿蒙 → 分布式软总线 → 多设备组成"超级终端"
     → ArkTS + ArkUI → 统一的应用开发框架
     → 方舟编译器 → 高性能 AOT 编译
```

<Note title="一句话总结">

鸿蒙的核心创新是**分布式架构**——通过分布式软总线将多个设备组成"超级终端"，共享硬件能力和数据，同时用 ArkTS + ArkUI 提供统一的应用开发体验。

</Note>

### ArkTS：鸿蒙的应用开发语言

ArkTS 是基于 TypeScript 的扩展语言，专门为鸿蒙应用开发设计。它在 TypeScript 的基础上增加了**声明式 UI 描述能力**和**状态管理装饰器**。

```typescript
// ArkTS 的声明式 UI 语法
@Entry
@Component
struct HelloWorld {
  @State message: string = 'Hello HarmonyOS';

  build() {
    Column() {
      Text(this.message)
        .fontSize(30)
        .fontWeight(FontWeight.Bold)

      Button('Click Me')
        .onClick(() => {
          this.message = 'Hello ArkUI!';
        })
    }
    .width('100%')
    .height('100%')
    .justifyContent(FlexAlign.Center)
  }
}
```

**ArkTS vs TypeScript 的关键区别：**

| 维度 | TypeScript | ArkTS |
|------|-----------|-------|
| **UI 描述** | 无（需要框架如 React/Vue）| 内置声明式 UI（build 函数）|
| **状态管理** | 无（需要状态管理库）| 内置装饰器（@State、@Prop、@Link）|
| **编译方式** | 编译为 JS，运行在 JS 引擎 | 方舟编译器 AOT 编译为原生代码 |
| **运行时** | V8/JavaScriptCore | 方舟运行时（自研）|
| **性能** | JIT + 解释执行 | AOT 编译，接近原生 |

### ArkUI：声明式 UI 框架

ArkUI 是鸿蒙的声明式 UI 开发框架，设计理念类似 Flutter 的 Widget 系统和 SwiftUI。

```text
ArkUI 的核心概念：

@Component（组件）
├── @Entry：页面入口组件（类似 Flutter 的 MaterialApp）
├── @Component：可复用的 UI 组件（类似 Flutter 的 StatelessWidget）
└── @CustomDialog：自定义弹窗

装饰器（状态管理）
├── @State：组件内部状态（类似 Flutter 的 setState）
├── @Prop：父组件单向传值（类似 React 的 props）
├── @Link：父子组件双向绑定（类似 Vue 的 v-model）
├── @Provide/@Consume：跨组件层级共享状态（类似 React 的 Context）
└── @Observed/@ObjectLink：嵌套对象的深度观察
```

```typescript
// ArkUI 的组件通信示例
@Component
struct ParentComponent {
  @State count: number = 0;

  build() {
    Column() {
      Text(`Parent count: ${this.count}`)
      ChildComponent({ count: $count })  // @Link 双向绑定
    }
  }
}

@Component
struct ChildComponent {
  @Link count: number;  // 接收父组件的双向绑定

  build() {
    Button(`Child count: ${this.count}`)
      .onClick(() => {
        this.count++;  // 修改会同步到父组件
      })
  }
}
```

### 方舟编译器与运行时

```text
方舟编译器的编译流程：

ArkTS 源码 → 静态类型检查 → 方舟字节码（ABC） → AOT 编译 → 原生机器码
                                        ↓
                              方舟运行时解释执行（开发阶段）

与 JavaScript 引擎的对比：
JavaScript → V8 解析 → JIT 编译 → 原生代码（运行时编译，有延迟）
ArkTS → 方舟编译器 → AOT 编译 → 原生代码（安装时编译，零运行时开销）
```

方舟编译器的核心优势：
- **AOT 编译**：在安装阶段就将 ArkTS 编译为原生机器码，运行时零编译开销
- **类型推导**：利用 ArkTS 的类型信息进行激进优化，生成更高效的机器码
- **并发 GC**：垃圾回收不阻塞主线程，减少卡顿

<Warning title="核心机制: ArkTS 的类型限制">

ArkTS 相比 TypeScript 做了**严格化限制**——移除了 TypeScript 中的动态特性（如 `any` 类型、`eval`、`with` 语句），目的是让方舟编译器能进行更激进的 AOT 优化。这意味着：

- 不支持 `any` 类型（必须明确类型）
- 不支持动态属性访问（如 `obj[dynamicKey]`）
- 不支持 `eval()` 和 `with` 语句
- 不支持运行时修改对象结构（如给对象添加新属性）

这些限制让习惯了 JavaScript 灵活性的开发者感到不适应，但它们是 AOT 编译高性能的前提。

</Warning>

<Tip title='通俗比喻: 鸿蒙的"交通系统"'>

鸿蒙的架构就像一个城市的交通系统：ArkTS 是**驾驶规则**（告诉司机怎么开车），ArkUI 是**道路和交通标志**（定义车辆怎么行驶和停靠），方舟编译器是**交通管理局**（把驾驶规则翻译成每个路口的具体信号灯指令）。分布式软总线是**高速公路网**——连接城市中的所有区域（设备），让车辆（数据和服务）可以在不同区域之间自由流动。

</Tip>

## Q3: 鸿蒙 vs Android vs iOS

### 基础梳理

| 对比维度 | 鸿蒙（HarmonyOS NEXT）| Android | iOS |
|---------|----------------------|---------|-----|
| **开发语言** | ArkTS（基于 TypeScript）| Kotlin/Java | Swift/ObjC |
| **UI 框架** | ArkUI（声明式）| Jetpack Compose / XML | SwiftUI / UIKit |
| **编译方式** | 方舟编译器 AOT | ART AOT/JIT | LLVM AOT |
| **内核** | 鸿蒙微内核 | Linux 宏内核 | XNU 混合内核 |
| **分布式能力** | 原生支持（分布式软总线）| 无原生支持 | Continuity（有限）|
| **应用分发** | 华为应用市场 | Google Play | App Store |
| **跨设备** | 手机/平板/手表/车机/IoT | 主要手机/平板 | 主要手机/平板/电脑 |
| **生态成熟度** | 快速增长中 | 最成熟 | 最成熟 |
| **包格式** | HAP（HarmonyOS Ability Package）| APK/AAB | IPA |

### 专家视角深度剖析

很多候选人只知道"鸿蒙是华为做的操作系统"，但大厂面试官真正在意的是：**你是否理解鸿蒙在分布式能力和开发范式上的独特性，以及它与 Android/iOS 的本质差异？**

**鸿蒙的三个核心技术差异：**

**1. 分布式软总线**

```text
传统模式（Android/iOS）：
手机拍照 → 通过网络上传 → 电脑下载 → 编辑
每一步都需要手动操作，依赖网络

鸿蒙分布式模式：
手机拍照 → 分布式软总线 → 电脑直接访问手机文件系统
文件就像在本地一样，无需手动传输
```

**2. Stage 模型 vs FA 模型**

```text
FA 模型（旧版，鸿蒙 3.0 及以前）：
├── 基于 Android 的 Activity 概念
├── PageAbility + ServiceAbility
├── 兼容 Android 生态

Stage 模型（新版，鸿蒙 NEXT）：
├── 自研的应用模型
├── UIAbility + ExtensionAbility
├── 更好的分布式能力和生命周期管理
└── 鸿蒙 NEXT 唯一支持的模型
```

**3. 一次开发，多端部署**

```text
Flutter 的跨端：一套代码 → 编译到不同平台 → 各平台独立运行
鸿蒙的跨端：一套代码 → 自适应布局 → 多设备协同运行

关键区别：
Flutter = 多个独立的应用（各跑各的）
鸿蒙 = 一个应用的多个分身（协同工作）
```

### 鸿蒙与前端开发的关系

对于前端开发者来说，鸿蒙是一个**新的就业方向**。ArkTS 基于 TypeScript，ArkUI 的声明式语法类似 React/Vue，前端开发者的学习成本相对较低。

```text
前端开发者转鸿蒙的路径：
├── TypeScript 基础 → 直接上手 ArkTS
├── React/Vue 经验 → 理解 ArkUI 的声明式范式
├── CSS 布局经验 → 适应 ArkUI 的 Flex/Grid 布局
└── 需要学习的新概念：分布式能力、Stage 模型、方舟编译器
```

struct Counter {

### 网络请求封装

```typescript
// HTTP 请求封装
import { http } from '@kit.NetworkKit';

interface RequestConfig {
  url: string;
  method: 'GET' | 'POST' | 'PUT' | 'DELETE';
  data?: object;
  header?: object;
}

class HttpRequest {
  private baseUrl: string = 'https://api.example.com';

  async request<T>(config: RequestConfig): Promise<T> {
    const httpRequest = http.createHttp();

    try {
      const response = await httpRequest.request(
        this.baseUrl + config.url,
        {
          method: http.RequestMethod.GET,
          header: {
            'Content-Type': 'application/json',
            ...config.header,
          },
          extraData: config.data ? JSON.stringify(config.data) : undefined,
        }
      );

      if (response.responseCode === 200) {
        return JSON.parse(response.result as string) as T;
      } else if (response.responseCode === 401) {
        // token 过期，跳转登录页
        router.pushUrl({ url: 'pages/LoginPage' });
        throw new Error('未授权');
      } else {
        throw new Error(`请求失败: ${response.responseCode}`);
      }
    } finally {
      httpRequest.destroy();
    }
  }
}

export const httpRequest = new HttpRequest();
```

### 分布式能力调用

```typescript
// 分布式数据管理：跨设备同步数据
import { distributedDataObject } from '@kit.ArkData';

@Entry
@Component
struct DistributedNote {
  @State noteContent: string = '';

  async aboutToAppear() {
    // 创建分布式数据对象
    const dataObject = distributedDataObject.create(context, {
      content: this.noteContent,
    });

    // 监听跨设备数据变更
    dataObject.on('change', (sessionId: string, field: string) => {
      if (field === 'content') {
        this.noteContent = dataObject['content'];
      }
    });
  }

  build() {
    Column() {
      TextEditor({ text: this.noteContent })
        .onChange((value: string) => {
          this.noteContent = value;
        })
    }
  }
}
```

<Note title="一句话总结">

鸿蒙开发的核心是 ArkTS + ArkUI 的声明式开发范式，加上分布式软总线的跨设备协同能力。前端开发者可以快速上手，但需要理解分布式架构和方舟编译器的特殊性。

</Note>

  <Accordion title="ArkTS 和 TypeScript 有什么区别？为什么要单独做一个语言？" icon="layer-group">

    **答题要点：**
    ArkTS 基于 TypeScript 但做了严格化限制，目的是支持 AOT 编译。

    **高分回答：**
    "ArkTS 是基于 TypeScript 的扩展语言，但做了关键的**严格化限制**：

    **移除的动态特性：**
    - 不支持 `any` 类型（所有类型必须明确）
    - 不支持 `eval()` 和 `with` 语句
    - 不支持运行时给对象添加新属性
    - 不支持动态属性访问（如 `obj[dynamicKey]`）

    **新增的能力：**
    - 声明式 UI 描述（`build()` 函数）
    - 状态管理装饰器（`@State`、`@Prop`、`@Link`）
    - 组件生命周期（`aboutToAppear`、`aboutToDisappear`）

    **为什么要单独做语言？**

    TypeScript 编译为 JavaScript 后运行在 V8 等 JS 引擎中，这些引擎使用 JIT 编译——首次执行时才编译为机器码，有性能延迟。方舟编译器需要**在安装阶段就将代码编译为原生机器码（AOT）**，这要求源代码的类型信息在编译时完全确定。TypeScript 的 `any`、动态属性访问等特性让 AOT 编译器无法进行激进优化，所以 ArkTS 移除了这些动态特性。

    ```text
    TypeScript → V8 JIT → 运行时编译（有延迟）
    ArkTS → 方舟编译器 AOT → 安装时编译（零运行时开销）
    ```

    对前端开发者的建议：如果你习惯了 TypeScript 的灵活性，ArkTS 的限制会让你不适应。但这些限制是 AOT 高性能的前提——和 Dart/Flutter 的设计思路类似。"

  <Accordion title="方舟编译器的 AOT 编译和 V8 的 JIT 编译有什么区别？" icon="stopwatch">

    **答题要点：**
    AOT 在安装时编译，JIT 在运行时编译，各有优劣。

    **高分回答：**
    "AOT（Ahead-of-Time）和 JIT（Just-in-Time）是两种不同的编译策略：

    **V8 的 JIT 编译：**
    ```text
    JavaScript 源码 → 解析器 → 字节码 → 解释执行（首次）
                                      ↓
                              监测热点函数
                                      ↓
                              JIT 编译为机器码（后续）
    ```
    - 优点：启动快（解释执行），可根据运行时信息优化（如类型推测）
    - 缺点：首次执行性能差，编译有延迟，需要存储编译缓存

    **方舟编译器的 AOT 编译：**
    ```text
    ArkTS 源码 → 方舟编译器 → 原生机器码（安装时）
    运行时直接执行机器码，零编译开销
    ```
    - 优点：运行时零编译开销，启动即满性能，内存占用更可控
    - 缺点：安装时间稍长，无法利用运行时类型信息优化

    **为什么方舟选择 AOT？**
    手机 App 对**启动速度和运行流畅度**要求极高。JIT 编译在首次执行时会有明显的性能延迟（用户首次打开某个功能时卡顿）。AOT 在安装时就完成了编译，用户打开 App 就是满性能。这对鸿蒙 NEXT 的用户体验至关重要。

    **实际影响：**
    - App 首次启动：AOT 更快（无解释执行阶段）
    - App 安装时间：AOT 更慢（需要编译）
    - 运行时内存：AOT 更可控（无 JIT 编译缓存）"

  </Accordion>

  <Accordion title="鸿蒙的 @State 和 React 的 useState 有什么异同？" icon="node">

    **答题要点：**
    两者都是声明式 UI 的状态管理机制，但底层实现和触发机制不同。

    **高分回答：**
    "鸿蒙的 `@State` 和 React 的 `useState` 都是声明式 UI 的状态管理装饰器，但有几个关键区别：

    **相同点：**
    - 都是组件级别的状态管理
    - 状态变化时触发 UI 重新渲染
    - 都是声明式 UI 范式的核心

    **不同点：**

    | 维度 | @State（ArkUI）| useState（React）|
    |------|---------------|-----------------|
    | **语法** | 装饰器 `@State` | Hook `useState()` |
    | **状态类型** | 可以是任意类型 | 可以是任意类型 |
    | **更新方式** | 直接赋值 `this.count++` | 调用 setter `setCount(c + 1)` |
    | **批量更新** | 同步更新 | 自动批量（React 18+）|
    | **触发范围** | 触发整个组件 rebuild | 触发整个组件 re-render |
    | **深度观察** | 支持（对象属性变化可检测）| 不支持（需要展开运算符）|

    ```typescript
    // ArkUI @State
    @Component
    struct Counter {
      @State count: number = 0;

      build() {
        Button(`${this.count}`)
          .onClick(() => this.count++)  // 直接赋值
      }
    }

    // React useState
    function Counter() {
      const [count, setCount] = useState(0);

      return (
        <button onClick={() => setCount(count + 1)}>
          {count}
        </button>
      );
    }
    ```

    **鸿蒙 @State 的深度观察：**
    ```typescript
    @State user: User = { name: 'Alice', age: 25 };

    // 直接修改属性也能触发更新
    this.user.age = 26;  // UI 会自动更新
    ```

    React 的 `useState` 不支持这种深度观察——修改对象属性不会触发更新，需要创建新对象。"

  </Accordion>

## 面试官视角：满分回答 vs 及格回答

### 问题："鸿蒙和 Android 的本质区别是什么？"

**及格回答（60-70分）：**
> "鸿蒙是华为自己做的操作系统，不依赖 Google。开发语言是 ArkTS，UI 框架是 ArkUI。"

面试官评价：**知道表面区别，但不理解技术架构和设计取舍。**

**满分回答（90-100分）：

> 鸿蒙和 Android 的本质区别在三个层面：**内核架构、应用框架、分布式能力**。

> 内核层面，Android 基于 Linux 宏内核，鸿蒙使用自研的微内核。微内核将系统服务（如文件系统、驱动）从内核空间移到用户空间，提高了安全性和可维护性——单个服务崩溃不会导致整个系统崩溃。鸿蒙的微内核还通过了形式化验证（数学证明），安全性高于 Linux 内核。

> 应用框架层面，Android 使用 Kotlin/Java + Jetpack Compose，鸿蒙使用 ArkTS + ArkUI。ArkTS 基于 TypeScript 但移除了动态特性，支持方舟编译器的 AOT 编译——在安装阶段就将代码编译为原生机器码，运行时零编译开销<sup>[2]</sup>。这和 Flutter 的 Dart AOT 编译思路类似，但方舟编译器针对 ArkTS 的类型系统做了更深度的优化。

> 分布式能力是鸿蒙最大的差异化。通过分布式软总线，多个鸿蒙设备可以组成"超级终端"，共享硬件能力和数据<sup>[1]</sup>。这是 Android 和 iOS 都没有的原生能力——Android 的 Nearby Share 只能传文件，iOS 的 Continuity 只能在 Apple 设备间接力，而鸿蒙的分布式能力可以让手机直接调用平板的摄像头、让手表的传感器数据实时同步到车机。
