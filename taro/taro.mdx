## Q1: Taro 的历史？为什么要有它？

```text
┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  微信小程序   │ → │  多端爆发    │ → │  重复开发    │ → │  Taro 诞生   │
│  2017 年上线  │   │  支付宝/百度 │   │  一套业务    │   │  2018 年发布 │
│              │   │  字节/京东    │   │  写 N 遍     │   │  一套代码    │
└──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
```

<Steps>

  <Step title="2017 年: 小程序爆发，多端割裂">

    2017 年微信小程序上线后，支付宝小程序、百度智能小程序、字节小程序、京东小程序相继推出。每个平台有自己的组件库、API 和生命周期，但业务逻辑高度相似。企业面临一个致命问题：**同一个电商活动页，要写 5 个版本。**

  </Step>
  <Step title="2018 年: 京东凹凸实验室发布 Taro">

    2018 年，京东凹凸实验室开源了 Taro（取名自意大利语"乌龟"，寓意稳健跨端）。Taro 的核心理念是：**用 React 的语法写一套代码，编译输出到微信/支付宝/百度/字节/京东小程序，以及 H5 和 React Native。**

  </Step>
  <Step title="Taro 1.x/2.x: 编译时方案">

    Taro 1.x 和 2.x 采用"编译时"方案——在构建阶段把 JSX 编译成各小程序平台的模板代码。优点是运行时零开销，缺点是 JSX 的动态能力受限（无法完全映射小程序模板的表达能力）。

  </Step>
  <Step title="2020 年: Taro 3.x 重构为运行时方案">

    Taro 3.x 进行了架构重构，从"编译时"转向"运行时"。核心思路是：**在小程序的逻辑层运行一个完整的 React/Vue 运行时，通过虚拟 DOM diff 后调用小程序的 setData 来更新视图。** 这让 Taro 获得了接近 Web 开发的完整能力，但也引入了运行时性能开销。

  </Step>

</Steps>

**Taro 解决的核心问题：**

```text
问题：多端小程序的重复开发

微信小程序 ─┐
支付宝小程序 ─┤
百度小程序 ─┼→ 同一套业务逻辑，写了 N 遍
字节小程序 ─┤
H5       ─┘

解决：React 语法 + 跨端编译

  React 代码 ──→ Taro 编译器/运行时 ──→ 微信小程序
                                    ──→ 支付宝小程序
                                    ──→ H5
                                    ──→ React Native
```

<Note title="一句话总结">

Taro 让你用 React 语法写一套代码，编译或运行时适配到多个小程序平台和 H5/RN，核心是解决多端重复开发的效率问题。

</Note>

<CardGroup cols={2}>

<Card title="没有 Taro 的时代">

同一个电商首页，微信版用 WXML + WXSS + 微信 API，支付宝版用 AXML + ACSS + 支付宝 API。改一个按钮颜色，要改 5 个仓库。就像一个翻译公司，每来一个新语种就从头翻译一遍整本书，不复用任何已有译文。

</Card>
<Card title="有了 Taro 之后">

用 React 写一份代码，Taro 帮你翻译成各平台的"方言"。改按钮颜色只需要改一处。就像请了一个万能同声传译——你说中文，台下英语、日语、法语听众同时听懂。

</Card>
</CardGroup>

<Tip title="面试技巧">

回答 Taro 历史时，重点说清"为什么"——多端重复开发的痛点。面试官想听的是你理解问题的本质，而不是背版本号。

</Tip>

## Q2: Taro 核心架构

Taro 的架构经历了从"编译时"到"运行时"的重大转变。理解这个转变的动因，是回答 Taro 面试题的关键。

### Taro 1.x/2.x: 编译时架构

```text
┌──────────────────┐
│   React JSX 代码  │
└────────┬─────────┘
         │  Taro 编译器（Babel 插件）
         ▼
┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐
│  微信小程序产物   │   │  支付宝小程序产物  │   │  H5 产物         │
│  WXML + WXSS     │   │  AXML + ACSS     │   │  HTML + CSS      │
│  + JS 逻辑       │   │  + JS 逻辑       │   │  + JS 逻辑       │
└──────────────────┘   └──────────────────┘   └──────────────────┘
```

编译时方案的核心思路：**在构建阶段，把 JSX 语法树直接转换成各平台的模板代码。** 运行时几乎零开销，因为编译产物就是原生代码。

**编译时方案的致命限制：**

| 能力 | Web/React | 小程序模板 | 编译时能否映射？ |
|------|-----------|-----------|----------------|
| 条件渲染 `{flag && &lt;Comp /&gt;}` | 支持 | `wx:if` | 能 |
| 列表渲染 `.map()` | 支持 | `wx:for` | 能 |
| 动态组件 `&lt;Comps[type] /&gt;` | 支持 | 不支持 | 不能 |
| 高阶组件 HOC | 支持 | 无对应概念 | 不能 |
| render props | 支持 | 无对应概念 | 不能 |
| `ref` 动态操作 DOM | 支持 | 不支持 | 不能 |

### Taro 3.x: 运行时架构

```text
┌────────────────────────────────────────────────────────┐
│                    小程序逻辑层（JSCore）                 │
│  ┌────────────┐   ┌────────────┐   ┌────────────────┐  │
│  │  React /   │ → │  Taro 虚拟 │ → │  setData()     │  │
│  │  Vue 运行时 │   │  DOM diff  │   │  更新视图层     │  │
│  └────────────┘   └────────────┘   └────────────────┘  │
└────────────────────────────────────────────────────────┘
                          │ setData
                          ▼
┌────────────────────────────────────────────────────────┐
│                    小程序视图层（WebView）                │
│  ┌──────────────────────────────────────────────────┐  │
│  │  Taro 内置组件（View, Text, Image...）           │  │
│  │  通过 template 渲染虚拟 DOM 树                    │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘
```

<Warning title="核心机制: Taro 3.x 的双线程通信">

Taro 3.x 的运行时方案意味着：React 的虚拟 DOM diff 在小程序的**逻辑层（JSCore）**中执行，diff 结果通过 `setData()` 发送到**视图层（WebView）**进行渲染。

- 每次 `setData()` 都是一次跨线程通信，数据量越大，通信耗时越长。
- 这是 Taro 3.x 性能瓶颈的根本原因——原生小程序开发可以精确控制 `setData` 的粒度，而 Taro 的虚拟 DOM diff 是自动批量的。
- Taro 通过"批量合并 setData"和"增量更新"来缓解这个问题。

</Warning>

### 双线程模型

小程序采用**双线程模型**（Dual-thread Model）——逻辑层和视图层分离，运行在不同的线程（甚至不同的运行环境）中。这是微信小程序官方团队在 2017 年设计的核心架构<sup>[1]</sup>，目的是隔离 JS 逻辑和 UI 渲染，防止恶意脚本直接操作 DOM：

```text
┌──────────────────┐     setData      ┌──────────────────┐
│    逻辑层         │ ──────────────→ │    视图层         │
│    (JSCore)       │                  │    (WebView)     │
│                   │     事件回调      │                   │
│    JS 业务逻辑    │ ←────────────── │    UI 渲染        │
│    网络请求       │                  │    用户交互       │
│    数据处理       │                  │                   │
└──────────────────┘                  └──────────────────┘

        两个线程通过 setData / 事件系统 通信
```

<Tip title="通俗比喻: 餐厅前厅和后厨">

小程序的双线程就像餐厅的前厅和后厨。后厨（逻辑层）负责做菜（执行 JS 逻辑），前厅（视图层）负责上菜（渲染 UI）。后厨做好一道菜，通过传菜窗口（setData）递给前厅。如果后厨一次性递了太多菜（大量 setData），传菜窗口就会堵塞，前厅的客人（用户）就得等——这就是小程序掉帧的原因。

</Tip>

### Taro 4.x: 半编译半运行时

Taro 4.x 引入了"半编译半运行时"的思路——在编译阶段做更多静态分析，减少运行时的 setData 次数<sup>[4]</sup>。核心策略包括：

- **静态模板提取**：编译阶段分析 JSX 中的静态部分，直接生成原生模板
- **按需 diff**：只有动态部分才走虚拟 DOM diff
- **组件级别拆分**：每个组件独立 setData，减少单次通信数据量

神光曾在掘金详细剖析过 Taro 的编译时逻辑（AST 语法树转换）和运行时渲染底层的差异<sup>[6]</sup>，指出半编译半运行时方案的本质是把编译时和运行时的优势结合起来：静态部分享受编译时的零开销，动态部分享受运行时的完整能力。

## Q3: Taro vs uni-app 对比

### 基础梳理

| 对比维度 | Taro | uni-app |
|---------|------|---------|
| **出品方** | 京东凹凸实验室 | DCloud |
| **核心框架** | React（主推）/ Vue | Vue（主推）/ uts |
| **架构方案** | 运行时（3.x）| 条件编译 + 运行时 |
| **跨端范围** | 微信/支付宝/百度/字节/H5/RN | 微信/支付宝/百度/字节/H5/App(weex) |
| **TypeScript** | 原生支持 | 支持（后期加强） |
| **社区规模** | GitHub 34k+ star | 生态更广（插件市场丰富） |
| **企业背书** | 京东、美团、B 站 | 600 万+ 开发者 |
| **适用人群** | React 技术栈团队 | Vue 技术栈团队 |

### 架构差异深度剖析

Taro 和 uni-app 在架构设计上有根本性的差异：

**Taro 的思路：运行时模拟**

```text
React JSX → Taro 运行时（在 JSCore 中执行 React）→ setData → 视图层
```

Taro 选择在小程序的逻辑层中运行一个完整的 React 运行时。这意味着你可以在小程序中使用 React 的全部能力（Hooks、Context、Suspense 等），但代价是运行时体积和 setData 的性能开销。

**uni-app 的思路：条件编译 + 运行时**

```text
Vue SFC → 条件编译（#ifdef MP-WEIXIN）→ 各平台原生代码
         → 通用逻辑 → uni-app 运行时适配层
```

uni-app 更强调"条件编译"——用 `#ifdef` 指令在编译阶段为不同平台生成不同的代码。通用逻辑走运行时适配，平台差异部分走条件编译。这让 uni-app 在小程序端的性能更接近原生。

### 性能对比

| 场景 | Taro 3.x | uni-app |
|------|----------|---------|
| 首屏渲染 | 较慢（需要初始化 React 运行时） | 较快（条件编译产物更轻量） |
| 列表滚动 | 需要虚拟列表优化 | 长列表性能较好 |
| setData 效率 | 批量 diff，可能产生冗余数据 | 更精确的 setData 控制 |
| 包体积 | 较大（包含 React 运行时） | 较小（按需引入） |
| H5 性能 | 接近原生 React | 接近原生 Vue |

### 适用场景

```text
选 Taro 的场景：
├── 团队技术栈是 React
├── 需要同时输出 React Native（App 端）
├── 项目需要 React 生态的丰富组件（如 Ant Design）
└── 对 TypeScript 支持要求高

选 uni-app 的场景：
├── 团队技术栈是 Vue
├── 需要覆盖更多小程序平台（尤其是快应用）
├── 需要 App 端原生能力（uni-app 的 App 端基于 weex/uts）
├── 追求小程序端的极致性能
└── 预算有限，需要快速出原型（uni-app 生态的模板和插件更丰富）
```

<Note title="一句话总结">

Taro 适合 React 团队追求开发体验，uni-app 适合 Vue 团队追求多端覆盖和小程序性能。没有绝对的好坏，只有适不适合。

</Note>

## Q4: Taro 的编译时和运行时有什么区别？

这是 Taro 面试中出场率最高的问题。回答时必须结合 Taro 1.x 到 3.x 的架构演进。

### 编译时（Taro 1.x/2.x）

```text
开发者写 JSX
    │
    ▼
Babel 插件分析 JSX AST
    │
    ├─ 条件渲染 → wx:if / wx:elif / wx:else
    ├─ 列表渲染 → wx:for
    ├─ 事件绑定 → bindtap / catchtap
    └─ 组件引用 → usingComponents
    │
    ▼
输出各平台原生模板 + JS 逻辑
```

编译时的核心工作发生在**构建阶段**。Babel 插件会分析 JSX 的抽象语法树（AST），把 JSX 节点映射成小程序的模板语法。

**编译时的优点：**
- 运行时零开销——编译产物就是原生代码
- 包体积小——不需要携带框架运行时
- 渲染性能好——直接调用原生渲染

**编译时的致命缺点：**
- JSX 的动态能力无法完全映射到小程序模板
- 不支持高阶组件、render props、动态组件等 React 高级特性
- 每次 API 变更都需要更新编译器

### 运行时（Taro 3.x）

```text
开发者写 React/Vue 代码
    │
    ▼
编译阶段：只做语法转换（JSX → React.createElement）
    │
    ▼
运行时：在小程序 JSCore 中执行 React 运行时
    │
    ├─ React Reconciler 做虚拟 DOM diff
    ├─ Taro Renderer 把 diff 结果转为 setData 调用
    └─ 视图层通过 template 渲染
```

运行时的核心工作发生在**小程序运行阶段**。Taro 3.x 在小程序的逻辑层中维护了一个完整的 React/Vue 运行时，通过自定义的 Renderer 把虚拟 DOM 的变更转换成 `setData()` 调用。

```javascript
// Taro 3.x 的核心原理（简化版）
import { Reconciler } from 'react-reconciler';

// 自定义 Renderer：把 React 的变更映射到小程序 setData
const TaroRenderer = Reconciler({
  createInstance(type, props) {
    return { type, props, children: [] };
  },
  appendChild(parent, child) {
    parent.children.push(child);
  },
  // ...其他方法
  commitUpdate(instance, updatePayload, type, oldProps, newProps) {
    // 核心：把虚拟 DOM 变更转为 setData
    instance._page.setData({
      [instance._path]: diff(oldProps, newProps)
    });
  }
});
```

**运行时的优点：**
- 支持 React/Vue 的全部能力（Hooks、Context、HOC 等）
- 开发体验接近 Web 开发
- 框架升级不需要更新编译器

**运行时的代价：**
- 需要携带 React/Vue 运行时，包体积增大
- setData 是批量的，可能产生冗余通信
- 首屏渲染较慢（需要初始化运行时）

### 关键区别总结

| 维度 | 编译时（1.x/2.x） | 运行时（3.x） |
|------|------------------|-------------|
| 核心工作阶段 | 构建时 | 运行时 |
| 运行时开销 | 零 | 需要 React/Vue 运行时 |
| React 能力 | 受限（只能用 JSX 的子集） | 完整 |
| 渲染性能 | 接近原生 | 有 setData 通信开销 |
| 包体积 | 小 | 大 |
| 维护成本 | 高（每个平台需要独立编译器） | 低（统一运行时适配层） |

<Warning title="核心机制: 为什么 Taro 3.x 选择运行时？">

编译时方案的根本问题是：**JSX 的表达能力是图灵完备的，但小程序模板的表达能力是受限的。** 你无法用编译的方式把任意 JSX 逻辑映射到 `wx:if` 和 `wx:for`。随着 React 生态的发展（Hooks、Suspense、Concurrent Mode），编译时方案的局限性越来越大，维护成本也越来越高。Taro 3.x 选择运行时方案，本质上是用性能换取开发体验和可维护性。

</Warning>

## Q5: Taro 3.x 的架构详解

Taro 3.x 的架构可以分为三层：**编译层、运行时层、适配层。**

```text
┌─────────────────────────────────────────────────────────┐
│                       编译层                              │
│  ┌──────────┐  ┌──────────┐  ┌───────────────────────┐  │
│  │ Babel    │  │ Webpack  │  │ 各平台插件             │  │
│  │ 插件     │  │ / Vite   │  │ (taro-plugin-* )      │  │
│  └──────────┘  └──────────┘  └───────────────────────┘  │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                     运行时层                              │
│  ┌──────────┐  ┌──────────┐  ┌───────────────────────┐  │
│  │ React /  │  │ Taro     │  │ Runtime API           │  │
│  │ Vue 运行时│  │ Reconciler│  │ (usePageScroll 等)   │  │
│  └──────────┘  └──────────┘  └───────────────────────┘  │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                     适配层                                │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────┐  │
│  │ 微信适配 │  │ 支付宝   │  │ H5 适配  │  │ RN 适配│  │
│  │          │  │ 适配     │  │          │  │        │  │
│  └──────────┘  └──────────┘  └──────────┘  └────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 编译层

编译层的核心职责是：把开发者写的 React/Vue 代码转换成各平台可运行的产物。

```javascript
// 编译层做的事情（简化）
// 输入：React JSX
function App() {
  return <View className="app"><Text>Hello</Text></View>;
}

// 输出：各平台的页面配置 + JS Bundle
// 微信小程序：
Page({
  data: { root: { type: 'view', ... } },
  onLoad() { /* Taro 运行时初始化 */ }
})
```

编译层不做 JSX 到模板的转换（那是 1.x 的做法），只做：
- 语法降级（ES6+ → ES5）
- 样式处理（CSS Modules、Sass → 各平台样式）
- 平台差异代码注入

### 运行时层

运行时层是 Taro 3.x 的核心。它包含：

1. **Taro Reconciler**：基于 `react-reconciler` 实现的自定义协调器，把 React 的虚拟 DOM 变更转换为小程序的 `setData` 调用
2. **Taro DOM**：在 JSCore 中模拟了一套 DOM API（`document.createElement`、`element.appendChild` 等），让 React 以为自己运行在浏览器中
3. **事件系统**：把小程序的事件（`tap`、`scroll` 等）映射为 React 的合成事件

```javascript
// Taro 在 JSCore 中模拟 DOM
const document = {
  createElement(tagName) {
    return {
      tagName,
      attributes: {},
      children: [],
      setAttribute(key, value) {
        this.attributes[key] = value;
      },
      appendChild(child) {
        this.children.push(child);
      }
    };
  }
};

// React 以为自己在操作真实 DOM，实际上在操作虚拟对象
// Taro Reconciler 监听变更，批量发 setData
```

### Taro Reconciler 的设计

Taro 3.x 的核心是基于 `react-reconciler` 实现的自定义协调器。崔健（卡颂）在讲解 React 跨端渲染引擎时，深入对比过 Taro 基于 `React Reconciler` 连接小程序双线程的极致实现<sup>[7]</sup>。Taro Reconciler 监听 React 组件树的变更，将 diff 结果转换为 `setData()` 调用——这本质上是把 React 的虚拟 DOM 变更"翻译"成小程序的通信协议。霍春阳也结合 Vue 3 渲染器架构，探讨过多种跨端 Renderer（针对 Taro 的 Vue 适配机制）的实现原理<sup>[8]</sup>。

### 适配层

适配层负责处理各平台的差异：

| 差异类型 | 示例 | 适配方式 |
|---------|------|---------|
| 组件差异 | 微信用 `&lt;view&gt;`，H5 用 `&lt;div&gt;` | Taro 组件库统一抽象 |
| API 差异 | 微信用 `wx.request`，支付宝用 `my.request` | Taro.request 统一封装 |
| 生命周期差异 | 小程序有 `onShow`/`onHide`，Web 没有 | Taro Hooks 统一暴露 |
| 导航差异 | 小程序用 `navigateTo`，H5 用 History API | Taro.navigateTo 统一调用 |

## Q6: Taro 的性能优化方案有哪些？

Taro 3.x 的性能瓶颈主要来自**setData 通信**。优化思路围绕"减少 setData 的频率和数据量"展开。

### setData 的性能问题

```text
┌──────────────┐         ┌──────────────┐
│   逻辑层      │ setData │   视图层      │
│   (JSCore)   │ ──────→ │   (WebView)  │
│              │  序列化   │              │
│  React diff  │  → JSON  │  解析 JSON   │
│              │  → 传输   │  → 更新 DOM  │
└──────────────┘         └──────────────┘

每次 setData 的成本 = JSON 序列化 + 跨线程传输 + JSON 解析 + DOM 更新
```

### 优化策略一：减少 setData 的数据量

```javascript
// ❌ 错误：每次 setData 传整个列表
this.setData({
  list: this.state.list // 1000 条数据全部传输
});

// ✅ 正确：只传变化的部分（路径更新）
this.setData({
  'list[0].name': 'new name' // 只传输一条数据
});
```

### 优化策略二：减少 setData 的频率

```javascript
// ❌ 错误：循环中多次 setData
for (let i = 0; i < 100; i++) {
  this.setData({ [`list[${i}].checked`]: true });
  // 100 次 setData，100 次跨线程通信
}

// ✅ 正确：批量合并
const updates = {};
for (let i = 0; i < 100; i++) {
  updates[`list[${i}].checked`] = true;
}
this.setData(updates); // 1 次 setData
```

### 优化策略三：虚拟列表

```text
┌─────────────────────────┐
│     可视区域（10 项）     │  ← 只渲染可见部分
├─────────────────────────┤
│     缓冲区（5 项）       │  ← 上下各预留 5 项
├─────────────────────────┤
│                         │
│     不渲染的区域          │  ← 1000 项中只渲染 20 项
│                         │
└─────────────────────────┘
```

```javascript
// 使用 Taro 虚拟列表组件
import { VirtualList } from '@tarojs/components';

function LongList({ data }) {
  return (
    <VirtualList
      height={500}       // 容器高度
      itemSize={80}       // 每项高度
      itemCount={data.length}
      renderItem={({ index, style }) => (
        <View style={style}>{data[index].name}</View>
      )}
    />
  );
}
```

### 优化策略四：WXS 处理交互动画

对于需要频繁更新的交互动画（如列表滑动时的吸顶效果），可以用 WXS 在视图层直接处理，避免跨线程通信：

```javascript
// WXS 在视图层直接执行，不需要 setData
// 适用于：滚动动画、手势跟随等高频交互
// 微信小程序的 WXS 语法
function onPageScroll(event) {
  var scrollTop = event.scrollTop;
  // 在视图层直接计算，不需要传回逻辑层
  return scrollTop > 100 ? 'fixed' : 'static';
}
```

### 企业级 setData 优化实践

网易云音乐大前端团队在百万级流量项目中使用 Taro 架构，积累了大量 setData 优化经验<sup>[9]</sup>。他们的核心策略是：将页面拆分为多个独立组件，每个组件单独管理自己的 setData，避免父组件变更导致整个页面重渲染。政采云前端团队（ZooTeam）在 Taro 企业级复杂业务落地中，也输出了跨端微侖前端基建的深度技术博文<sup>[10]</sup>。

### 优化策略五：分包加载

```text
┌─────────────────────────┐
│        主包（2MB）        │  ← 首页必须的代码
│  首页 + 公共组件 + 公共逻辑│
├─────────────────────────┤
│  分包 A（1MB）            │  ← 商品详情页
│  分包 B（1MB）            │  ← 订单页
│  分包 C（1MB）            │  ← 个人中心
└─────────────────────────┘
总计：5MB，但首屏只加载主包 2MB
```

<Warning title="核心机制: 小程序的包体积限制">

微信小程序规定：**单个包不超过 2MB，总体积不超过 20MB（包含分包）。** 超过 2MB 的包无法上传。Taro 通过分包配置，把非首屏的代码拆到分包中，按需加载。

</Warning>

## Q7: Taro 的高频面试题精选

### Taro 的跨端原理是什么？

Taro 的跨端原理可以总结为"**一层抽象，多端适配**"：

```text
┌─────────────────────────┐
│    开发者代码（React）     │  ← 统一的开发语法
├─────────────────────────┤
│    Taro 抽象层            │  ← 统一组件 + 统一 API
│  ┌───────┐ ┌──────────┐ │
│  │组件层  │ │ API 层   │ │
│  │View   │ │ request  │ │
│  │Text   │ │ navigate │ │
│  │Image  │ │ storage  │ │
│  └───────┘ └──────────┘ │
├─────────────────────────┤
│    平台适配层             │  ← 各平台的实现
│  微信    支付宝   H5  RN  │
└─────────────────────────┘
```

Taro 在上层提供统一的组件（`View`、`Text`、`Image`）和统一的 API（`Taro.request`、`Taro.navigateTo`），在底层通过适配层对接各平台的原生实现。

    **答题要点：**
    核心原因是 JSX 的表达能力是图灵完备的，而小程序模板的表达能力是受限的，编译时方案无法覆盖 React 的全部能力。

    **高分回答：**
    "Taro 1.x/2.x 的编译时方案有一个根本性矛盾：JSX 的表达能力是图灵完备的（支持任意 JavaScript 表达式），但小程序模板的表达能力是受限的（只有 `wx:if`、`wx:for` 等基础指令）。这意味着你无法用编译的方式把任意 JSX 逻辑映射到小程序模板。

    Taro 3.x 选择运行时方案，本质上是**用性能换取开发体验和可维护性**。在小程序的逻辑层中运行一个完整的 React 运行时，通过虚拟 DOM diff 后调用 `setData` 更新视图。这样开发者可以使用 React 的全部能力，代价是运行时体积和 setData 的通信开销。"

    **高分回答：**
    "Taro 3.x 的性能瓶颈主要来自 `setData` 通信。每次 `setData` 的成本 = JSON 序列化 + 跨线程传输 + JSON 解析 + DOM 更新。

    优化策略有五个层次：
    1. **减少数据量**：使用路径更新只传变化的部分，如 `setData({ 'list[0].name': 'new' })`

    3. **虚拟列表**：只渲染可视区域的列表项，1000 条数据只渲染 20 条
    4. **WXS 处理动画**：在视图层直接执行交互逻辑，避免跨线程通信
    5. **分包加载**：首屏只加载主包，非首屏代码按需加载

  <Accordion title='Taro 4.x 的半编译半运行时方案是怎么实现的？' icon="skull">

    **答题要点：**
    结合编译时的静态分析和运行时的动态能力，取两者之长。

    **高分回答：**
    "Taro 4.x 的核心改进是引入了'半编译半运行时'（也叫'静态模板 + 运行时'）方案。

    **原理：**
    在编译阶段，Taro 会分析 JSX 的抽象语法树，识别出**静态部分**（如纯文本、固定结构的布局）和**动态部分**（如条件渲染、列表渲染、变量绑定）。静态部分直接生成小程序的原生模板，动态部分保留在运行时通过虚拟 DOM diff 处理。

    ```text
    编译阶段：
    JSX → AST 分析 → 静态部分 → 原生模板（零运行时开销）
                     → 动态部分 → 运行时 diff（React 能力完整保留）
    ```

    **效果：**
    - 静态内容的渲染不需要 `setData`，直接用原生模板渲染
    - 只有动态变化的部分才触发 `setData`
    - 首屏渲染速度提升 30%-50%（静态模板不需要初始化 React）
    - 包体积减小（静态模板不携带 React 运行时代码）

    这个方案的本质是**把编译时和运行时的优势结合起来**：静态部分享受编译时的零开销，动态部分享受运行时的完整能力。"

  </Accordion>

  <Accordion title='Taro 项目如何做单元测试？' icon="skull">

    **答题要点：**
    Taro 项目需要特殊的测试配置，因为 Taro 组件依赖小程序环境。

    **高分回答：**
    "Taro 项目的单元测试面临一个核心问题：Taro 组件依赖小程序的运行时环境（如 `Taro.navigateTo`、`usePageScroll` 等），这些 API 在 Node.js 测试环境中不存在。

    **解决方案：**

    1. **Jest + 模拟 Taro API**
    ```javascript
    // jest.setup.js
    jest.mock('@tarojs/taro', () => ({
      navigateTo: jest.fn(),
      usePageScroll: jest.fn((cb) => {
        useEffect(() => { cb({ scrollTop: 0 }); }, []);
      }),
      getCurrentInstance: jest.fn(() => ({
        router: { params: {} }
      }))
    }));
    ```

    2. **React Testing Library 测试组件逻辑**
    ```javascript
    import { render, screen, fireEvent } from '@testing-library/react';
    import Counter from './Counter';

    test('点击按钮增加计数', () => {
      render(<Counter />);
      fireEvent.click(screen.getByText('+1'));
      expect(screen.getByText('1')).toBeTruthy();
    });
    ```

    3. **E2E 测试**：使用 miniprogram-automator（微信官方）或 Taro 的 E2E 测试插件，在真实的小程序环境中测试。

    **最佳实践：** 业务逻辑（纯函数、Hooks）用 Jest 单测，组件用 React Testing Library，页面级交互用 E2E 测试。"

  </Accordion>

## 面试官视角：满分回答 vs 及格回答

### 问题："说说 Taro 的架构演进"

**及格回答（60-70分）：**
> "Taro 1.x 是编译时方案，把 JSX 编译成小程序模板。Taro 3.x 改成了运行时方案，在小程序里跑 React。运行时性能差一些但开发体验好。"

面试官评价：**知道版本变化，但不理解为什么变。**

**满分回答（90-100分）：**

> Taro 的架构经历了从"编译时"到"运行时"的重大转变。1.x/2.x 采用编译时方案——Babel 插件分析 JSX 的抽象语法树，把 JSX 节点映射成小程序的原生模板（`wx:if`、`wx:for`）。运行时零开销，但致命问题是：JSX 的表达能力是图灵完备的，小程序模板的表达能力是受限的，无法映射 React 的高阶组件、render props、Hooks 等高级特性。

> Taro 3.x 重构为运行时方案——在小程序的逻辑层（JSCore）中运行一个完整的 React 运行时，通过自定义的 Reconciler 把虚拟 DOM 变更转换为 `setData()` 调用。这让 Taro 获得了 React 的全部能力，代价是运行时体积和 `setData` 的跨线程通信开销。核心性能瓶颈在于每次 `setData` 都需要 JSON 序列化 + 跨线程传输 + JSON 解析。

> Taro 4.x 引入了"半编译半运行时"方案，在编译阶段做静态分析，静态部分直接生成原生模板（零运行时开销），动态部分保留虚拟 DOM diff。首屏渲染速度提升 30%-50%。

> 实际项目建议：用虚拟列表优化长列表、批量合并 `setData` 减少通信频率、分包加载控制首屏包体积。对于极致性能场景，可以用 WXS 处理交互动画，避免跨线程通信。
