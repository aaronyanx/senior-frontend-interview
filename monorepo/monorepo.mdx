## Q1: Monorepo 的历史？为什么要有它？

```text
┌──────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  Polyrepo    │ → │  依赖地狱    │ → │  Monorepo    │ → │  工具链成熟  │
│  各自为政    │   │  版本碎片    │   │  统一管理    │   │  pnpm/Turbo  │
└──────────────┘   └──────────────┘   └──────────────┘   └──────────────┘
```

<Steps>

  <Step title="2006-2013: Polyrepo 时代">

    在 Monorepo 流行之前，业界普遍采用 **Polyrepo**（多仓库）模式——每个 npm 包一个 Git 仓库。Google、Microsoft 等大厂虽然内部早已使用 Monorepo，但前端社区还没有形成统一的工程化方案。

  </Step>
  <Step title="2014: Google 发表 Monorepo 论文">

    Google 发表了经典的工程实践论文《Why Google Stores Billions of Lines of Code in a Single Repository》，系统性地阐述了 Monorepo 的核心价值：**代码共享、原子提交、统一版本**。这篇论文让 Monorepo 从大厂内部实践变成了业界共识。

  </Step>
  <Step title="2016-2018: 工具链涌现">

    Lerna（2015）首次为前端 Monorepo 提供了成熟的工具支持。随后 pnpm（2016）、Yarn Workspaces（2017）相继发布，解决了依赖管理的核心问题。

  </Step>
  <Step title="2021-至今: Monorepo 成为标配">

    Turborepo（2021 被 Vercel 收购）、Nx（Nrwl）等构建编排工具成熟，pnpm workspace 成为事实标准。现代前端项目（Next.js、Vue、Vite）全部采用 Monorepo 架构。

  </Step>

</Steps>

**Polyrepo 的核心痛点：**

```text
问题：多个仓库管理多个包

repo-a/  →  依赖 lodash@4.17.20
repo-b/  →  依赖 lodash@4.17.21    ← 版本不一致，行为可能不同
repo-c/  →  依赖 lodash@4.17.19

修改一个公共函数 → 需要分别修改 3 个仓库 → 分别发版 → 分别更新依赖
一个跨包 Bug → 需要同时修改多个仓库 → 原子性无法保证

解决：所有包放在一个仓库

monorepo/
  packages/
    a/  →  统一 lodash@4.17.21
    b/  →  统一 lodash@4.17.21
    c/  →  统一 lodash@4.17.21

修改一个公共函数 → 一次 commit 搞定 → 原子提交 → 统一版本
```

<Note title="一句话总结">

Monorepo 用"一个仓库管理多个包"解决了 Polyrepo 的代码碎片化和版本不一致问题。

</Note>

<CardGroup cols={2}>

<Card title="Polyrepo 的困境">

每个包一个仓库，改一个公共 API 需要跨多个仓库修改、分别发版、逐个更新依赖。就像一家连锁餐厅，每家店的菜单、配方、供应商都不统一，总部想改一道菜的配方，需要逐店通知。

</Card>
<Card title="Monorepo 的优势">

所有包在一个仓库，改一个公共 API 只需一次 commit，所有包同时生效。就像连锁餐厅的中央厨房——配方统一管理，改一次所有门店同时更新。

</Card>
</CardGroup>

<Tip title="面试技巧">

回答时先说 Polyrepo 的痛点（代码碎片化、版本不一致、跨包修改困难），再说 Monorepo 如何解决。面试官想听的是你理解问题的本质，而不是背定义。

</Tip>

## Q2: npm vs pnpm vs yarn 对比

**三大包管理器的历史演进：**

```text
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  npm@1       │ → │  yarn@1      │ → │  pnpm        │
│  2010        │   │  2016        │   │  2016        │
│  Isaac       │   │  Facebook    │   │  Zoltan      │
│  最早的      │   │  解决 npm    │   │  解决 node_  │
│  包管理器    │   │  的性能问题  │   │  modules 痛点│
└──────────────┘   └──────────────┘   └──────────────┘
```

### npm 的历史

npm（Node Package Manager）由 Isaac Z. Schlueter 于 2010 年创建，是 Node.js 的默认包管理器。早期的 npm@3 引入了扁平化的 `node_modules` 结构，解决了依赖嵌套过深的问题，但也带来了**幽灵依赖**和**依赖分身**等新问题。

### pnpm 的历史

pnpm（Performant npm）由 Zoltan Kochan 于 2016 年创建。它的核心理念是：**用硬链接和软链接来共享依赖，彻底解决 node_modules 的磁盘浪费和幽灵依赖问题**。

### yarn 的历史

Yarn 由 Facebook 于 2016 年发布，最初是为了解决 npm@3 的性能和确定性问题（lockfile、并行安装）。Yarn@1 引入了 `yarn.lock`，Yarn@2+（Berry）引入了 Plug'n'Play（PnP）模式，彻底抛弃 `node_modules`。

### 三者的核心区别

| 对比维度 | npm | yarn | pnpm |
|---------|-----|------|------|
| **诞生时间** | 2010 | 2016 | 2016 |
| **node_modules 结构** | 扁平化（Flat） | 扁平化（Flat） | 非扁平化（Content-addressable） |
| **幽灵依赖** | 有 | 有 | 无 |
| **依赖分身** | 有 | 有 | 几乎无 |
| **磁盘占用** | 高 | 高 | 低（硬链接共享） |
| **安装速度** | 慢 | 中 | 快 |
| **Lockfile** | package-lock.json | yarn.lock | pnpm-lock.yaml |
| **Workspace 支持** | npm workspaces | yarn workspaces | pnpm workspace |

<Warning title="核心机制: 扁平化 node_modules 的问题">

npm 和 yarn 采用扁平化的 `node_modules` 结构——把所有依赖提升到顶层。这导致两个严重问题：

- **幽灵依赖（Phantom Dependencies）**：你可以 import 没有在 `package.json` 中声明的包（因为它们被提升到了顶层）
- **依赖分身（Dependency Doppelgangers）**：同一个包的不同版本只能提升一个，其他版本嵌套安装，浪费磁盘

</Warning>

### node_modules 结构对比

```text
=== npm / yarn 的扁平化结构 ===

node_modules/
  lodash/              ← 提升到顶层（可能是 A 依赖的版本）
  A/
    node_modules/
      lodash/          ← 嵌套安装（版本冲突时）
  B/
    node_modules/
      lodash/          ← 嵌套安装（版本冲突时）

问题：
1. 项目代码可以直接 import lodash（幽灵依赖）
2. 同一个 lodash 可能被安装多次（依赖分身）

=== pnpm 的非扁平化结构 ===

node_modules/
  .pnpm/
    lodash@4.17.21/    ← 全局内容寻址存储（硬链接）
    A@1.0.0/
      node_modules/
        lodash -> ../../lodash@4.17.21  ← 软链接
  A -> .pnpm/A@1.0.0/node_modules/A    ← 软链接
  B -> .pnpm/B@1.0.0/node_modules/B    ← 软链接

优势：
1. 只能 import package.json 中声明的依赖（无幽灵依赖）
2. 同一版本的包只存储一次（硬链接，无依赖分身）
```

<Tip title="通俗比喻: 仓库管理">

npm 的 `node_modules` 就像一个杂乱的仓库——所有东西都堆在门口（扁平化），你随手就能拿到没登记过的工具（幽灵依赖）。pnpm 的 `node_modules` 像一个有序的仓库——每件工具都有编号（内容寻址），只有登记过的人才能借（无幽灵依赖），相同的工具只存一份（硬链接共享）。

</Tip>

## Q3: pnpm 核心原理

pnpm 的核心设计目标是解决 npm/yarn 扁平化 `node_modules` 的两大痛点：**幽灵依赖**和**依赖分身**。

### 幽灵依赖（Phantom Dependencies）

幽灵依赖是指：你可以 import 一个**没有在 `package.json` 中声明**的包。这是因为 npm/yarn 的扁平化机制把子依赖提升到了 `node_modules` 顶层。

```javascript
// package.json 只声明了 express
{
  "dependencies": {
    "express": "^4.18.0"
  }
}

// 但你可以直接 import lodash（express 的子依赖）
// ❌ 这就是幽灵依赖——你没有声明 lodash，但它"碰巧"被提升到了顶层
const _ = require('lodash');

// 危险：express 升级后可能不再依赖 lodash，你的代码就崩了
```

<Warning title="幽灵依赖的危害">

幽灵依赖的最大风险是**不确定性**：

- 你依赖的包没有在 `package.json` 中声明，所以版本更新时可能突然消失
- 不同环境下安装结果可能不同（因为提升顺序受依赖树结构影响）
- 新同事接手项目时，看到 `import lodash` 但 `package.json` 里没有，完全无法理解依赖关系

</Warning>

### 依赖分身（Dependency Doppelgangers）

依赖分身是指：同一个包的**不同版本**被重复安装，浪费磁盘空间。

```text
场景：A 依赖 lodash@4.17.20，B 依赖 lodash@4.17.21

=== npm 的处理 ===
node_modules/
  lodash@4.17.21/          ← 提升一个版本到顶层
  A/
    node_modules/
      lodash@4.17.20/      ← 另一个版本嵌套安装（依赖分身）
  B/
  // lodash 被安装了两次，浪费磁盘

=== pnpm 的处理 ===
store/
  lodash@4.17.20/          ← 全局存储，硬链接
  lodash@4.17.21/          ← 全局存储，硬链接

node_modules/.pnpm/
  lodash@4.17.20/          ← 硬链接到 store（不额外占用磁盘）
  lodash@4.17.21/          ← 硬链接到 store（不额外占用磁盘）

// 同一版本的包只存储一次，不同项目共享
```

### 软链接与硬链接（Symlink / Hardlink）

pnpm 的核心机制是**硬链接 + 软链接**的组合：

```text
=== 硬链接（Hardlink）===
磁盘上的文件 ←→ 硬链接
同一个文件，多个入口，不额外占用磁盘
删除一个硬链接不影响其他硬链接

=== 软链接（Symlink）===
软链接 → 指向目标文件的路径
类似快捷方式，指向另一个位置
```

pnpm 的存储结构：

```text
全局 store（~/.pnpm-store/）
  └── lodash@4.17.21/
      ├── index.js    ← 真实文件（只存一份）

项目 node_modules/
  └── .pnpm/
      └── lodash@4.17.21/
          └── index.js  ← 硬链接到 store（不额外占磁盘）

  └── A/
      └── node_modules/
          └── lodash -> .pnpm/lodash@4.17.21  ← 软链接
```

### pnpm 的 node_modules 结构

```text
node_modules/
  .pnpm/                           ← 所有依赖的真实安装位置
    express@4.18.2/
      node_modules/
        express/                   ← 硬链接到 store
        body-parser/               ← 软链接
        lodash/                    ← 软链接
    lodash@4.17.21/
      node_modules/
        lodash/                    ← 硬链接到 store

  express -> .pnpm/express@4.18.2/node_modules/express   ← 软链接（顶层只能看到直接依赖）
  A -> .pnpm/A@1.0.0/node_modules/A                      ← 软链接
```

**关键设计：** 顶层 `node_modules` 只暴露 `package.json` 中声明的直接依赖，子依赖通过 `.pnpm` 目录内的软链接访问。这彻底消除了幽灵依赖。

<Tip title="通俗比喻: 图书馆">

pnpm 的全局 store 就像图书馆的书库——每本书只存一份（硬链接）。每个项目就像一个阅览室——通过借书卡（软链接）引用书库里的书。你想看《JavaScript 高级程序设计》（lodash），必须在借书卡上登记（`package.json` 声明），不能偷偷从书架上拿别的阅览室的书（无幽灵依赖）。

</Tip>

## Q4: Monorepo 工具对比

### Lerna（已废弃）

Lerna 是最早的前端 Monorepo 管理工具（2015），由 Sebastian McKenzie（Webpack 作者）创建。它提供了包发布（publish）、版本管理（version）、依赖提升（hoisting）等功能。**2022 年 Lerna 被 Nx 团队接管，已不再活跃维护。**

### Turborepo

Turborepo 由 Jared Palmer 于 2021 年创建，后被 Vercel 收购。它的核心价值是**任务编排 + 远端缓存**——通过拓扑排序确定构建顺序，通过缓存避免重复构建。

### Nx

Nx 由 Nrwl（前 Angular 核心团队）创建，是一个全功能的 Monorepo 管理工具。它提供了任务编排、依赖图分析、增量构建、代码生成等功能，适合大型企业级项目。

### pnpm workspace

pnpm workspace 是 pnpm 内置的 Monorepo 支持。它通过 `pnpm-workspace.yaml` 定义工作区，自动处理包之间的依赖链接。**pnpm workspace 只解决依赖管理，不解决构建编排。**

### Bazel

Bazel 由 Google 开源，是 Google 内部 Monorepo 的构建工具。它支持多语言、细粒度增量构建、分布式缓存，但学习曲线极陡，前端项目很少使用。

| 对比维度 | Lerna | Turborepo | Nx | pnpm workspace | Bazel |
|---------|-------|-----------|-----|----------------|-------|
| **维护状态** | 已废弃 | 活跃 | 活跃 | 活跃 | 活跃 |
| **核心能力** | 包发布 | 任务编排+缓存 | 全功能 | 依赖管理 | 构建系统 |
| **学习成本** | 低 | 低 | 中 | 低 | 高 |
| **远端缓存** | 无 | 内置 | 内置 | 无 | 内置 |
| **任务编排** | 基础 | 强 | 强 | 无 | 强 |
| **适用规模** | 小型 | 中大型 | 大型 | 任意 | 超大型 |
| **前端友好度** | 高 | 高 | 高 | 高 | 低 |

<Note title="一句话总结">

pnpm workspace 管依赖，Turborepo/Nx 管构建编排。现代 Monorepo 的最佳组合是 **pnpm workspace + Turborepo**。

</Note>

## Q5: Turborepo 核心原理

Turborepo 的核心价值是**用缓存和并行化来加速 Monorepo 的构建**。

### 远端缓存（Remote Caching）

Turborepo 的核心创新是**远端缓存**——把构建产物上传到远端服务器，团队成员共享缓存，避免重复构建。

```text
=== 没有远端缓存 ===

开发者 A: 构建 packages/ui → 产物 hash(abc123) → 花费 60s
开发者 B: 构建 packages/ui → 产物 hash(abc123) → 花费 60s（重复构建！）

=== 有远端缓存 ===

开发者 A: 构建 packages/ui → 产物 hash(abc123) → 花费 60s → 上传到远端
开发者 B: 构建 packages/ui → 产物 hash(abc123) → 命中远端缓存 → 花费 2s（下载缓存）
```

**缓存键（Cache Key）的计算：**

```text
缓存键 = Hash(源文件内容 + 依赖版本 + 环境变量 + 构建命令)

源文件变化 → Hash 变化 → 缓存失效 → 重新构建
源文件不变 → Hash 不变 → 命中缓存 → 跳过构建
```

### 拓扑排序（Topological Sort）

Monorepo 中的包之间存在依赖关系（如 `app` 依赖 `ui`，`ui` 依赖 `utils`）。Turborepo 用**拓扑排序**确定构建顺序，确保被依赖的包先构建。

```text
依赖关系：
  app → ui → utils

拓扑排序结果：
  1. utils    （无依赖，先构建）
  2. ui       （依赖 utils，utils 构建完后才能构建）
  3. app      （依赖 ui，ui 构建完后才能构建）

并行化：
  第一批：utils
  第二批：ui（等 utils 完成）
  第三批：app（等 ui 完成）
```

### 任务编排

Turborepo 通过 `turbo.json` 配置任务依赖关系：

```json
{
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**"]
    },
    "test": {
      "dependsOn": ["build"]
    },
    "lint": {}
  }
}
```

`^build` 表示：先构建所有依赖的包，再构建当前包。`lint` 没有依赖，可以并行执行。

```text
turbo run build test lint 的执行顺序：

  第一批（并行）：utils:lint, ui:lint, app:lint
  第二批：utils:build（lint 完成 + 无依赖）
  第三批：ui:build（依赖 utils:build 完成）
  第四批：app:build（依赖 ui:build 完成）
  第五批：utils:test, ui:test, app:test（依赖各自 build 完成）
```

<Tip title="通俗比喻: 工厂流水线">

Turborepo 就像一个智能工厂调度系统。它知道哪些零件（包）可以同时生产（并行），哪些必须等前置零件完成（拓扑排序）。更重要的是，如果一个零件上次已经生产过且原材料没变（缓存命中），就直接从仓库取货，不需要重新生产（远端缓存）。

</Tip>

## Q6: Monorepo 最佳实践

### 目录结构

```text
monorepo/
  apps/                          ← 应用（可部署的项目）
    web/                         ← Web 应用
    admin/                       ← 管理后台
    mobile/                      ← 移动端应用

  packages/                      ← 公共包（被多个应用共享）
    ui/                          ← UI 组件库
    utils/                       ← 工具函数
    config/                      ← 共享配置（ESLint、TSConfig）
    types/                       ← 共享类型定义

  package.json                   ← 根 package.json（只放 devDependencies）
  pnpm-workspace.yaml            ← pnpm 工作区配置
  turbo.json                     ← Turborepo 任务配置
  tsconfig.base.json             ← 基础 TypeScript 配置
```

```yaml
# pnpm-workspace.yaml
packages:
  - "apps/*"
  - "packages/*"
```

### 依赖管理

```json
// 根 package.json — 只放开发工具
{
  "devDependencies": {
    "turbo": "^1.10.0",
    "typescript": "^5.0.0",
    "eslint": "^8.0.0"
  }
}
```

```json
// apps/web/package.json — 声明自己的依赖
{
  "dependencies": {
    "@monorepo/ui": "workspace:*",      // 引用工作区内的包
    "@monorepo/utils": "workspace:*",
    "react": "^18.0.0"
  }
}
```

<Warning title="依赖管理原则">

- 根 `package.json` 只放全局开发工具（turbo、typescript、eslint）
- 每个包的 `package.json` 声明自己的依赖，不要依赖隐式提升
- 用 `workspace:*` 引用工作区内的包，不要用版本号
- 用 `catalog:` 统一管理共享依赖的版本（pnpm 9+ 特性）

</Warning>

### 构建优化

```json
// turbo.json
{
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**", ".next/**"]
    },
    "test": {
      "dependsOn": ["build"],
      "outputs": []
    },
    "lint": {
      "outputs": []
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```

**关键优化策略：**

| 策略 | 说明 | 效果 |
|------|------|------|
| **增量构建** | 只构建变化的包 | 构建时间减少 50-80% |
| **并行执行** | 无依赖的任务并行 | 多核 CPU 利用率提升 |
| **远端缓存** | CI/CD 和开发者共享缓存 | 首次构建后几乎零成本 |
| **产物缓存** | 缓存 `dist/`、`.next/` 等产物 | 避免重复编译 |

### CI/CD 集成

```yaml
# .github/workflows/ci.yml
name: CI
```

## Q7: 高频面试题

### pnpm 为什么比 npm 快？

pnpm 快的核心原因是**硬链接 + 内容寻址存储**：

1. **全局 store**：所有包的真实文件只存储一次在 `~/.pnpm-store/` 中
2. **硬链接**：项目中的 `node_modules` 通过硬链接引用 store 中的文件，不额外复制
3. **增量安装**：已存在于 store 中的包不需要重新下载

```text
npm 安装 lodash:
  下载 → 解压 → 复制到 node_modules/  （完整流程）

pnpm 安装 lodash:
  下载 → 解压到 store/ → 硬链接到 node_modules/  （跳过复制）
  如果 store 中已有 → 直接硬链接（跳过下载和解压）
```

### 幽灵依赖是什么？怎么解决？

幽灵依赖是 npm/yarn 扁平化 `node_modules` 的副作用——你可以 import 没有在 `package.json` 中声明的包。

```javascript
// package.json 只声明了 express
{ "dependencies": { "express": "^4.18.0" } }

// 但你可以直接使用 lodash（express 的子依赖）
const _ = require('lodash');  // ❌ 幽灵依赖

// 解决方案：使用 pnpm
// pnpm 的 node_modules 结构只暴露直接依赖
// import lodash 会直接报错：Cannot find module 'lodash'
```

### Monorepo 和 Polyrepo 的区别？

| 对比维度 | Monorepo | Polyrepo |
|---------|----------|----------|
| **仓库数量** | 一个仓库，多个包 | 每个包一个仓库 |
| **代码共享** | 直接引用，路径引用 | 通过 npm 发包 |
| **原子提交** | 支持（一次 commit 修改多个包） | 不支持（需要多个 PR） |
| **版本管理** | 统一版本或独立版本 | 各自独立版本 |
| **CI/CD** | 需要工具支持（Turborepo） | 每个仓库独立 CI |
| **适用场景** | 组件库、中台系统、全栈项目 | 独立微服务、开源库 |

### Turborepo 的缓存是怎么实现的？

Turborepo 的缓存分为**本地缓存**和**远端缓存**：

```text
缓存键 = Hash(源文件 + 依赖 + 环境变量 + 命令)

本地缓存：
  node_modules/.cache/turbo/
    hash(abc123)/
      stdout     ← 标准输出
      stderr     ← 标准错误
      dist/      ← 构建产物

远端缓存：
  api.vercel.com/cache/
    hash(abc123)/
      stdout + stderr + dist/  ← 压缩上传

缓存命中条件：
  1. 缓存键（Hash）完全一致
  2. 缓存未过期
  3. 本地或远端有对应的缓存记录
```

## 十、面试官视角：满分回答 vs 及格回答

### 问题："说说 pnpm 和 npm 的区别，以及 Monorepo 的优势"

**及格回答（60-70分）：**
> "pnpm 比 npm 快，因为它用硬链接和软链接来共享依赖。Monorepo 就是把所有代码放在一个仓库里，方便管理。"

面试官评价：**知道结论但不理解原理**。

**满分回答（90-100分）：**

> pnpm 和 npm 的核心区别在于 `node_modules` 的结构。npm 采用扁平化结构，把所有依赖提升到顶层，这导致两个问题：**幽灵依赖**（可以 import 未声明的包）和**依赖分身**（同一包的不同版本重复安装）。pnpm 用**内容寻址存储 + 硬链接 + 软链接**的方案彻底解决了这两个问题——所有包的真实文件只存储一次在全局 store 中，项目通过硬链接引用，顶层只暴露 `package.json` 中声明的直接依赖。<sup>[1]</sup>

> Monorepo 的核心价值是**代码共享和原子提交**。在 Polyrepo 模式下，修改一个公共 API 需要跨多个仓库修改、分别发版、逐个更新依赖。Monorepo 只需要一次 commit，所有包同时生效。配合 Turborepo 的**拓扑排序和远端缓存**，可以实现增量构建——只构建变化的包，构建时间从 10 分钟降到 1 分钟。<sup>[2]</sup>

> 实际项目中的建议：使用 **pnpm workspace + Turborepo** 的组合。pnpm 管依赖（解决幽灵依赖和依赖分身），Turborepo 管构建编排（拓扑排序 + 远端缓存）。目录结构上，`apps/` 放可部署的应用，`packages/` 放共享的库。根 `package.json` 只放全局开发工具，每个包的 `package.json` 声明自己的依赖。<sup>[3]</sup>

  <Accordion title="workspace:* 是什么意思？和版本号有什么区别？" icon="layer-group">

    **答题要点：**
    `workspace:*` 是 pnpm/yarn 的工作区协议，表示引用工作区内的包。

    **高分回答：**
    "`workspace:*` 是 pnpm 的工作区协议（workspace protocol），表示引用工作区内的包，而不是从 npm registry 下载。
    - `workspace:*`：匹配工作区内的任何版本
    - `workspace:^1.0.0`：匹配工作区内 `^1.0.0` 的版本
    - `workspace:~1.0.0`：匹配工作区内 `~1.0.0` 的版本

    发布到 npm 时，pnpm 会自动把 `workspace:*` 转换为实际的版本号。这保证了开发时引用本地包，发布时引用正确的版本。"

    **高分回答：**
    "Turborepo 和 Nx 都是优秀的 Monorepo 构建编排工具，核心区别：
    1. **Turborepo**：轻量、零配置、与 Vercel 生态深度集成。适合中小型项目和快速上手。

    - 小团队、Next.js 项目 → Turborepo
    - 大团队、多技术栈、需要代码生成 → Nx
    - 只需要依赖管理 → pnpm workspace（不需要额外工具）"

    **答题要点：**
    pnpm 提供了自动清理机制。

    **高分回答：**
    "pnpm 的全局 store 确实会随着使用逐渐增大，因为每个版本的包都会被存储。但 pnpm 提供了清理机制：
    1. `pnpm store prune`：删除不再被任何项目引用的 store 条目
    2. `pnpm store status`：查看 store 中哪些条目不再被引用
    3. CI/CD 环境中可以配置定期清理

    实际上，即使 store 增大，总的磁盘占用仍然比 npm/yarn 小得多——因为同一版本的包只存一份，所有项目通过硬链接共享。"

  </Accordion>

  <Accordion title="Monorepo 的包之间如何互相引用？" icon="node">

    **答题要点：**
    用 `workspace:*` 协议引用，发布时自动转换。

    **高分回答：**
    "Monorepo 中包之间互相引用有两种方式：
    1. **workspace 协议**（推荐）：在 `package.json` 中用 `workspace:*` 引用
    ```json
    { "dependencies": { "@monorepo/utils": "workspace:*" } }
    ```
    2. **路径引用**：直接用相对路径 import（不推荐，破坏包的独立性）
    ```javascript
    import { helper } from '../../packages/utils/src/index'; // ❌ 不推荐
    ```

    `workspace:*` 的好处是：开发时引用本地包（实时生效），发布到 npm 时自动替换为实际版本号（保证生产环境正确）。"

  </Accordion>

  <Accordion title="如何处理 Monorepo 中的版本发布？" icon="node">

    **答题要点：**
    有两种策略：固定版本（Fixed）和独立版本（Independent）。

    **高分回答：**
    "Monorepo 的版本发布有两种策略：
    1. **固定版本（Fixed）**：所有包统一版本号，一次发布所有变更的包。适合包之间强耦合的项目（如 Vue 生态）。
    2. **独立版本（Independent）**：每个包独立版本号，只发布有变更的包。适合包之间松耦合的项目（如组件库）。

    工具推荐：
    - `changeset`：最流行的 Monorepo 版本管理工具，支持独立版本和自动 changelog
    - `pnpm version`：pnpm 内置的版本管理
    - `semantic-release`：基于 commit message 自动确定版本号"

  </Accordion>

  <Accordion title="幽灵依赖在生产环境会出问题吗？" icon="skull">

    **答题要点：**
    幽灵依赖在生产环境会导致不可预测的崩溃。

    **高分回答：**
    "幽灵依赖在生产环境**一定会出问题**，只是时间早晚：
    1. **依赖升级导致包消失**：express 升级后不再依赖 lodash，你的 `require('lodash')` 直接报错
    2. **不同环境行为不一致**：本地开发能跑（因为 npm 扁平化），CI 环境崩了（因为安装顺序不同）
    3. **Docker 构建失败**：`npm ci` 的安装结果和 `npm install` 不同，幽灵依赖可能消失

    解决方案只有一个：**使用 pnpm**。pnpm 的 node_modules 结构从根源上消除了幽灵依赖——顶层只暴露 `package.json` 中声明的直接依赖。"

  </Accordion>

    **高分回答：**
    "从 npm 迁移到 pnpm 的步骤：
    1. **删除 node_modules 和 lockfile**：`rm -rf node_modules package-lock.json`
    2. **安装 pnpm**：`npm install -g pnpm`
    3. **运行 pnpm install**：生成 `pnpm-lock.yaml`
    4. **修复幽灵依赖**：pnpm 会报错找不到未声明的依赖，逐个添加到 `package.json`
    5. **运行测试**：确保所有功能正常

    迁移的最大挑战是**修复幽灵依赖**。你可能需要把几十个隐式依赖显式声明到 `package.json` 中。可以用 `pnpm install --reporter=ndjson` 查看哪些包解析失败。"
