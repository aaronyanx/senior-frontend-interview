> PRD 到代码生成（PRD-to-Code）是 AI 辅助开发的端到端工作流。面试官问这个，本质上是在考察你：**是否理解 AI 如何重塑前端开发流程，以及能不能把 Prompt Engineering、Agent、工具调用串联成一个完整的工程化闭环。** 如果你答不清楚，说明你对 AI 的使用停留在"单点工具"而不是"系统化工作流"。

## Q1: PRD 到代码生成的演进历史

```text
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 手动开发  │ → │ 低代码   │ → │ Copilot  │ → │ Agent    │ → │ PRD→Code │
│ (2010)   │   │ (2018)   │   │ (2021)   │   │ (2024)   │   │ (2025+)  │
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
  全手写         拖拽+配置      行级补全       对话式生成      端到端自动化
```

<Steps>

  <Step title="手动开发时代 (2010-2017)">

  前端工程师拿到 PRD 后，手动编写每一行代码。从需求理解、架构设计、UI 实现到测试，全部依赖人工。一个中等复杂度的页面需要 3-5 天开发周期，沟通成本占项目时间的 30% 以上。

  </Step>
  <Step title="低代码与可视化搭建 (2018-2020)">

  低代码平台（如宜搭、微搭）通过拖拽组件和配置属性生成代码。解决了简单表单和中后台页面的快速搭建，但灵活性不足——复杂交互和定制需求仍然需要手写代码。

  </Step>
  <Step title="AI 代码补全 (2021-2023)">

  GitHub Copilot 的出现改变了开发模式。AI 从"工具"变成了"副驾驶"——根据上下文自动补全代码行和函数。开发效率提升 30-55%，但仍然需要开发者主导架构和流程。

  </Step>
  <Step title="对话式代码生成 (2024)">

  Cursor、Claude Code、Windsurf 等 AI-native IDE 兴起。开发者通过自然语言描述需求，AI 生成完整的文件和模块。从"补全"进化到"生成"，但仍然需要人工串联各个步骤。

  </Step>
  <Step title="端到端 PRD→Code (2025+)">

  V0、Bolt.new、Lovable、GitHub Copilot Workspace 等工具实现了从 PRD 到可部署代码的自动化管线。AI Agent 解析 PRD、生成架构、编写代码、生成测试、部署上线，开发者从"写代码"转变为"审代码"。

  </Step>

</Steps>

**PRD→Code 解决的核心问题：**

```text
问题：传统开发流程中，PRD → 设计稿 → 前端代码 → 测试 → 部署

每个环节都需要人工介入，沟通成本高、开发周期长

PRD 理解偏差 → 返工 → 延期 → 超预算

解决：AI 端到端自动化

PRD → AI 解析 → 自动生成架构 + 代码 + 测试 → 人工审查 → 部署

开发周期从周级缩短到天级，沟通成本降低 60%+
```

<Note title="💡 一句话总结">

PRD→Code 的核心不是"让 AI 替代程序员"，而是"把开发流程从手工作坊升级为自动化管线"——开发者从写代码转变为审代码和定义规范。

</Note>

<CardGroup cols={2}>

<Card title="🚫 没有 PRD→Code 工作流">

产品经理写 PRD → 设计师出设计稿 → 前端看 PRD 理解需求 → 手写代码 → 手写测试 → 部署。每个环节都有信息损耗，PRD 理解偏差导致返工率高达 25%。一个中等页面从 PRD 到上线需要 1-2 周。

</Card>
<Card title="✅ 引入 PRD→Code 工作流">

产品经理写 PRD → AI 解析 PRD 生成技术方案 → AI 生成代码和测试 → 开发者审查和调整 → 自动部署。信息在管线中无损流转，开发者聚焦在架构决策和质量把控上。同样的页面从 PRD 到上线缩短到 1-2 天。

</Card>
</CardGroup>

<Tip title="🎢 通俗比喻: 从手工作坊到自动化工厂">

传统开发就像手工作坊——师傅（开发者）拿到图纸（PRD），一锤一锤地敲出产品。PRD→Code 就像自动化工厂——图纸（PRD）送进流水线（AI 管线），机器自动切割（代码生成）、打磨（测试生成）、组装（部署），师傅只需要在最后检查质量（人工审查）。师傅不需要消失，但角色从"体力劳动者"变成了"质量管理者"。

</Tip>

## Q2: PRD 到代码生成的完整流程

PRD→Code 的本质是一条**自动化管线**，核心逻辑只有 5 步：

```text
┌───────────────────────────────────────────────────────────────┐
│                    PRD → Code 完整管线                        │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐      │
│  │ PRD 解析 │ → │ 架构设计 │ → │ 代码生成 │ → │ 测试生成 │      │
│  └─────────┘   └─────────┘   └─────────┘   └─────────┘      │
│       │                                              │        │
│       │            ┌─────────┐                       │        │
│       └──────────→ │ 人工审查 │ ←─────────────────────┘        │
│                    └────┬────┘                                │
│                         ▼                                     │
│                    ┌─────────┐                                │
│                    │ CI/CD   │                                │
│                    │ 部署上线 │                                │
│                    └─────────┘                                │
└───────────────────────────────────────────────────────────────┘
```

<Warning title="⚠️ 核心机制: 人工审查是不可省略的环节">

AI 生成的代码质量已经很高，但仍然存在三个风险：

- **幻觉风险**：AI 可能编造不存在的 API 或使用已废弃的方法
- **架构偏差**：AI 对复杂业务逻辑的理解可能与实际需求有偏差
- **安全漏洞**：AI 可能生成存在 XSS、CSRF 等安全隐患的代码

人工审查不是"走形式"，而是管线中的质量关口。审查重点：架构合理性、业务逻辑正确性、安全合规性。

</Warning>

<Tip title="🎢 通俗比喻: 自动驾驶的 L2 与 L5">

PRD→Code 的自动化程度就像自动驾驶的分级。L2（Copilot）：AI 做辅助（补全代码），人主导。L3（Cursor）：AI 做大部分工作（生成模块），人监督。L4（V0/Bolt）：AI 做全流程（PRD→代码→测试），人在关键节点审查。L5（完全自主）：AI 做全流程且不需要人——目前还不存在。大多数团队处在 L3-L4 阶段，核心瓶颈不是 AI 能力不够，而是信任不够——开发者还没有建立"审查 AI 代码"的工作习惯。

</Tip>

### 各阶段的工具链

| 阶段 | 工具 | 作用 |
|------|------|------|
| **PRD 解析** | Claude / GPT-4 | 从 PRD 提取功能需求、非功能需求、技术方案 |
| **设计稿生成** | Figma AI / V0 | 从需求描述生成 UI 设计稿 |
| **代码生成** | V0 / Cursor / Claude Code | 从设计稿或需求生成前端代码 |
| **测试生成** | Qodo / Copilot | 从代码自动生成单元测试和 E2E 测试 |
| **部署** | Vercel / Netlify / GitHub Actions | 自动化 CI/CD 部署 |

### 管线中的数据流

| 输入 | 中间产物 | 输出 |
|------|---------|------|
| PRD 文档 | 结构化需求 JSON | 技术方案文档 |
| 技术方案 | 组件树 + API 接口定义 | 前端代码 |
| 前端代码 | 测试用例列表 | 测试代码 |
| 测试代码 | 测试报告 | 部署就绪的产物 |

## Q3: PRD 解析与需求提取

PRD 解析的核心挑战是**把非结构化的自然语言转化为结构化的需求描述**。PRD 文档通常用自然语言写成，包含功能需求、非功能需求、业务规则、边界条件等信息，AI 需要从中提取出可执行的技术规格。

```text
┌──────────────────────────────────────────────────┐
│                  PRD 文档（非结构化）              │
│  "用户可以在购物车中修改商品数量，                  │
│   数量不能超过库存，修改后实时更新总价..."          │
└────────────────────┬─────────────────────────────┘
                     │ AI 解析
                     ▼
┌──────────────────────────────────────────────────┐
│              结构化需求 JSON                       │
│  {                                               │
│    "feature": "购物车数量修改",                    │
│    "functional": [...],                          │
│    "nonFunctional": [...],                       │
│    "businessRules": [...]                        │
│  }                                               │
└──────────────────────────────────────────────────┘
```

<Tip title="🎢 通俗比喻: 同声传译">

PRD 解析就像同声传译——把产品经理的"方言"（非结构化 PRD）翻译成开发者的"标准语"（结构化需求 JSON）。好的翻译不是逐字翻译，而是理解语义后用目标语言重新表达。AI 解析 PRD 也是一样——不是提取原文中的关键词，而是理解业务意图后生成结构化描述。如果 PRD 写"用户体验要好"，AI 需要翻译成"首屏加载 &lt; 2s、交互响应 &lt; 100ms"。

</Tip>

### PRD 质量对输出的影响

PRD 的质量直接决定 AI 生成代码的质量——这就是"Garbage In, Garbage Out"原则。

| PRD 质量 | AI 输出质量 | 典型问题 |
|---------|-----------|---------|
| 高质量（有验收条件、有边界说明） | 高 | 几乎不需要修改 |
| 中质量（有功能描述，缺边界条件） | 中 | 需要补充边界处理逻辑 |
| 低质量（模糊的一句话需求） | 低 | 生成的代码与预期偏差大 |

```javascript
// ❌ 低质量 PRD → AI 无法生成准确代码
const badPRD = "做一个购物车页面";

// ✅ 高质量 PRD → AI 能生成高质量代码
const goodPRD = `
购物车页面需求：
1. 展示商品列表（商品图片、名称、单价、数量）
2. 数量可通过 +/- 按钮修改，支持手动输入
3. 数量不能超过库存，不能小于 1
4. 修改数量后实时更新小计和总价
5. 支持删除商品（二次确认弹窗）
6. 空购物车展示引导页面
7. 性能要求：100 个商品时渲染流畅
`;
```

## Q4: 设计稿到代码

设计稿到代码（Design-to-Code）是 PRD→Code 管线中最直观的环节。核心原理是**AI 读取设计稿的结构化数据（图层、样式、布局），映射为对应的前端代码**。

```text
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  Figma 设计  │  →   │  结构化数据  │  →   │  前端代码    │
│  (图层/样式) │      │  (JSON/AST) │      │  (React/Vue) │
└─────────────┘      └─────────────┘      └─────────────┘
       │                                        │
       │           Figma MCP / V0               │
       └────────────────────────────────────────┘
```

<Warning title="⚠️ 核心机制: Figma MCP 是设计稿到代码的桥梁">

Figma MCP（Model Context Protocol）是 Anthropic 提出的标准化协议，让 AI Agent 能直接读取 Figma 文件的结构化数据——图层树、样式、布局约束、设计 Token。传统方式是截图让 AI 识别（信息损耗大），MCP 方式是直接读取设计数据（信息无损）。

- **截图方式**：AI 只能看到像素，无法精确获取间距、字号、颜色值
- **MCP 方式**：AI 读取完整的图层树和样式数据，生成的代码与设计稿像素级一致

</Warning>

```javascript
// Figma MCP 的工作原理（简化）
// 1. AI Agent 通过 MCP 协议调用 Figma API
const figmaData = await mcp.call("figma.getFile", {
  fileKey: "abc123",
  nodeIds: ["1:2"], // 指定页面或组件的节点 ID
});

// 2. 返回的结构化数据包含完整的图层树
// {
//   "name": "CartPage",
//   "type": "FRAME",
//   "layoutMode": "VERTICAL",
//   "children": [
//     { "name": "Header", "type": "FRAME", "children": [...] },
//     { "name": "CartList", "type": "FRAME", "children": [...] },
//     { "name": "Footer", "type": "FRAME", "children": [...] }
//   ],
//   "styles": { "padding": 16, "gap": 12 }
// }

// 3. AI 根据结构化数据生成代码
function generateComponent(figmaNode) {
  const layout = figmaNode.layoutMode === "VERTICAL" ? "flex-col" : "flex-row";
  const children = figmaNode.children.map(generateComponent);
  return `
    <div className="${layout} p-${figmaNode.styles.padding / 4}">
      ${children.join("\n")}
    </div>
  `;
}
```

<Tip title="🎢 通俗比喻: 从蓝图到工厂图纸">

Figma 设计稿就像建筑蓝图——好看但不能直接用来施工。传统方式是施工队（开发者）看着蓝图手动砌墙（写代码），容易出现偏差。Figma MCP 就像把蓝图数字化——AI 读取每个房间的尺寸、门窗位置、材料规格（图层树和样式数据），自动生成施工图纸（前端代码）。从"看着蓝图猜尺寸"变成"读取精确数据自动生成"。

</Tip>

### V0 的设计稿转代码原理

V0 是 Vercel 推出的 AI UI 生成工具，支持从文本描述或截图生成 React + Tailwind CSS 代码。它的核心能力是**理解 UI 语义并映射为组件**。

```javascript
// V0 的工作流程：
// 1. 用户输入（文本描述或截图）
const userInput = "一个电商购物车页面，左侧商品列表，右侧订单摘要";

// 2. AI 生成组件树
// CartPage
// ├── CartHeader (标题 + 商品数量)
// ├── CartItemList (商品卡片列表)
// │   └── CartItem × N (图片 + 名称 + 价格 + 数量控件)
// └── OrderSummary (小计 + 运费 + 优惠 + 总价 + 结算按钮)

// 3. 生成的代码（React + Tailwind CSS + shadcn/ui）
import { Button } from "@/components/ui/button";
import { Card } from "@/components/ui/card";

function CartPage({ items }) {
  const totalPrice = items.reduce((sum, item) => sum + item.price * item.quantity, 0);

  return (
    <div className="grid grid-cols-1 lg:grid-cols-3 gap-8">
      <div className="lg:col-span-2 space-y-4">
        <h2 className="text-2xl font-bold">购物车 ({items.length} 件商品)</h2>
        {items.map(item => (
          <Card key={item.id} className="flex gap-4 p-4">
            <img src={item.image} alt={item.name} className="w-24 h-24 object-cover rounded" />
            <div className="flex-1">
              <h3 className="font-medium">{item.name}</h3>
              <p className="text-lg font-bold">¥{item.price}</p>
            </div>
          </Card>
        ))}
      </div>
      <div>
        <Card className="p-6 sticky top-4">
          <h3 className="text-lg font-bold mb-4">订单摘要</h3>
          <div className="flex justify-between mb-2">
            <span>商品总价</span><span>¥{totalPrice}</span>
          </div>
          <Button className="w-full mt-4">去结算</Button>
        </Card>
      </div>
    </div>
  );
}
```

### Locofy 与 Anima 的对比

| 维度 | Locofy | Anima |
|------|--------|-------|
| **核心定位** | 设计稿转生产级代码 | 设计稿转原型 + 代码 |
| **支持框架** | React、Vue、Next.js、HTML/CSS | React、Vue、HTML/CSS |
| **AI 能力** | AI 自动检测组件边界和布局 | AI 辅助但更依赖手动调整 |
| **代码质量** | 接近生产级，可直接使用 | 需要较多手动调整 |
| **最佳场景** | 开发者主导的快速实现 | 设计师主导的原型验证 |

## Q5: 代码到测试

AI 生成测试的核心原理是**分析代码的结构和行为，自动生成覆盖正常路径、边界条件和异常场景的测试用例**。

```text
┌──────────────────────────────────────────────────────┐
│                   源代码                               │
│  function add(a, b) { return a + b; }                │
└────────────────────┬─────────────────────────────────┘
                     │ AI 分析
          ┌──────────┼──────────┐
          ▼          ▼          ▼
    正常路径测试   边界条件测试   异常场景测试
    add(1,2)=3    add(0,0)=0   add(null,1)=?
    add(5,3)=8    add(-1,1)=0  add("1",2)=?
```

```javascript
// AI 生成单元测试（Jest）的 Prompt
const testGenerationPrompt = `

## 任务

- 使用 describe/it 结构组织测试
- Mock 外部依赖（API 调用、localStorage 等）

## 源代码
${sourceCode}
`;

// 端到端工作流的自动化脚本（简化版）
// 使用 Claude Code + V0 + Playwright 的工具链

// Step 1: 解析 PRD
async function parsePRD(prdPath) {
  const prdContent = await readFile(prdPath);
  const requirements = await claude.chat({
    system: "你是前端架构师，从 PRD 中提取结构化需求",
    message: `解析以下 PRD：\n${prdContent}`,
    responseFormat: { type: "json_object" },
  });
  return JSON.parse(requirements);
}

// Step 2: 生成设计稿（通过 V0 API 或 Figma MCP）
async function generateDesign(requirements) {
  const design = await v0.generate({
    prompt: `根据以下需求生成 UI 设计：${JSON.stringify(requirements.features)}`,
    framework: "react",
    style: "tailwind",
  });
  return design; // 返回设计稿 URL 或组件代码
}

// Step 3: 生成代码（通过 Cursor / Claude Code）
async function generateCode(design, requirements) {
  const code = await claude.chat({
    system: `你是资深前端工程师。根据设计稿和需求生成生产级代码。
    技术栈：${requirements.technicalConstraints.join(", ")}`,
    message: `设计稿：${design}\n需求：${JSON.stringify(requirements)}`,
  });
  return code;
}

// Step 4: 生成测试
async function generateTests(sourceCode) {
  const tests = await claude.chat({
    system: "你是测试工程师。为以下代码生成 Jest 单元测试和 Playwright E2E 测试。",
    message: sourceCode,
  });
  return tests;
}

// Step 5: 运行测试 + 部署
async function runTestsAndDeploy() {
  await exec("npm run test");
  await exec("npm run test:e2e");
  await exec("git push origin main"); // 触发 CI/CD
}

// 完整管线
async function prdToCode(prdPath) {
  const requirements = await parsePRD(prdPath);
  const design = await generateDesign(requirements);
  const code = await generateCode(design, requirements);
  const tests = await generateTests(code);

  await writeFiles(code, tests);
  await runTestsAndDeploy();
}
```

<Tip title="最佳实践: 工具链选择">

工具链选择的核心原则：**每个环节用最擅长的工具**。

- **PRD 解析**：Claude（长上下文理解能力强，200K tokens 窗口）
- **设计稿生成**：V0（React + Tailwind 生态最成熟）
- **代码生成**：Cursor（IDE 集成度最高，支持 @codebase 上下文）
- **测试生成**：Qodo / GitHub Copilot（测试生成专用优化）
- **部署**：Vercel（与 V0 生态无缝集成）

不要试图用一个工具完成所有环节——每个工具都有自己的优势领域。

</Tip>

### 工具链选择矩阵

| 场景 | 推荐工具链 | 原因 |
|------|-----------|------|

## 🔍 底层源码：AI Agent 如何编排 PRD→Code 管线

<div class="follow-up">
AI Agent 是怎么编排 PRD→Code 管线的？
</div>

AI Agent 使用 LangGraph 的图结构来编排多步骤管线。每个节点是一个独立的任务（PRD 解析、代码生成、测试生成），边定义了数据流转关系（详见 [LangGraph 文档](https://langchain-ai.github.io/langgraph/)）：

```python
# LangGraph 编排 PRD→Code 管线（简化版）
from langgraph.graph import StateGraph

# 定义状态
class PipelineState(TypedDict):
    prd_content: str
    requirements: dict
    design: str
    code: str
    tests: str
    review_feedback: str

# 定义节点
async def parse_prd(state: PipelineState) -> dict:
    requirements = await llm.invoke(PARSE_PROMPT.format(prd=state["prd_content"]))
    return {"requirements": requirements}

async def generate_code(state: PipelineState) -> dict:
    code = await llm.invoke(CODE_PROMPT.format(req=state["requirements"]))
    return {"code": code}

async def generate_tests(state: PipelineState) -> dict:
    tests = await llm.invoke(TEST_PROMPT.format(code=state["code"]))
    return {"tests": tests}

async def human_review(state: PipelineState) -> dict:
    # 人工审查节点——管线暂停，等待人工反馈
    return {"review_feedback": await get_human_feedback(state)}

# 构建图
graph = StateGraph(PipelineState)
graph.add_node("parse_prd", parse_prd)
graph.add_node("generate_code", generate_code)
graph.add_node("generate_tests", generate_tests)
graph.add_node("human_review", human_review)

# 定义边（数据流转）
graph.add_edge("parse_prd", "generate_code")
graph.add_edge("generate_code", "generate_tests")
graph.add_edge("generate_tests", "human_review")
graph.add_conditional_edges("human_review", route_review)

# 运行管线
pipeline = graph.compile()
result = await pipeline.invoke({"prd_content": prd_content})
```

这段代码做了 3 件事：1) 用 StateGraph 定义管线的状态和节点；2) 每个节点调用 LLM 完成特定任务；3) human_review 节点暂停管线等待人工反馈，实现 Human-in-the-Loop。

"工具链选择取决于三个因素：项目类型、团队规模、质量要求。

**快速原型验证**（个人/小团队）：V0 + Vercel。从描述到部署最快（分钟级），适合验证想法、做 Demo。缺点是代码质量中等，不适合生产环境。

**企业级项目**（中大团队）：Cursor + Claude Code + Playwright。代码质量和测试覆盖率最高，支持 @codebase 上下文理解整个项目。缺点是学习成本高，需要建立 CLAUDE.md 规范。

**设计驱动开发**（设计团队主导）：Figma MCP + Claude Code。设计稿到代码精度最高，像素级还原。缺点是依赖 Figma，设计师需要学习 MCP 配置。

**全栈项目**（前后端一体）：Bolt.new / Lovable。前后端一体化生成，适合全栈开发者。缺点是灵活性不足，复杂定制需要手动调整。

核心原则：每个环节用最擅长的工具，不要试图用一个工具完成所有环节。"

  </Accordion>

  <Accordion title="PRD→Code 和传统的低代码平台有什么区别?" icon="skull">

**答题要点：**
从灵活性、代码质量、可维护性三个维度对比。

**高分回答：**
"表面上看，PRD→Code 和低代码都是'自动生成代码'，但本质区别很大：

**灵活性**：低代码平台只能生成平台支持的组件和逻辑，超出平台能力的需求无法实现。PRD→Code 生成的是标准代码（React/Vue/TypeScript），没有平台锁定，任何需求都能实现。

**代码质量**：低代码生成的代码通常是平台私有格式（JSON Schema），不可读、不可维护。PRD→Code 生成的是标准开源代码，可读、可维护、可版本控制。

**可维护性**：低代码平台的代码和平台绑定——平台升级可能导致不兼容，平台停服则代码报废。PRD→Code 生成的代码完全独立，不依赖任何平台。

**适用场景**：低代码适合简单的表单和中后台页面（80% 的标准场景）。PRD→Code 适合所有场景，特别是复杂交互和定制需求。

反面证据：低代码平台的'低代码'是相对的——复杂需求仍然需要写大量配置代码，配置代码的学习成本不比写代码低。PRD→Code 的优势是'生成标准代码'，开发者不需要学习任何新工具。"

  </Accordion>

  <Accordion title="PRD→Code 的未来趋势是什么?" icon="skull">

**答题要点：**
从多 Agent 协作、自主调试、持续学习三个方向回答。

**高分回答：**
"PRD→Code 的未来趋势有三个方向：

1. **多 Agent 协作**：不再是单个 AI 完成所有工作，而是多个专业 Agent 分工协作——架构 Agent 设计方案、前端 Agent 写 UI 代码、测试 Agent 生成测试、DevOps Agent 配置部署。每个 Agent 专注自己的领域，通过 LangGraph 编排协作。

2. **自主调试**：当前的 PRD→Code 管线是'生成 → 人审查 → 修复'的循环。未来的趋势是 AI 自主运行测试、分析失败原因、自动修复代码、重新测试——直到所有测试通过。Devin 和 Claude Code 已经在探索这个方向。

3. **持续学习**：AI 从项目的代码审查反馈中学习——哪些模式被接受、哪些被拒绝、团队的编码风格偏好。随着项目积累，AI 生成的代码越来越符合团队标准。CLAUDE.md 和 AGENT.md 就是这个方向的早期形态。

终极愿景：PRD→Code 从'L4（AI 做全流程，人审查）'进化到'L5（AI 做全流程，不需要人）'。但这需要 AI 在架构决策、安全审查、业务理解上达到人类专家水平——短期内不太可能。"

  </Accordion>
